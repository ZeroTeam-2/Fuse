# queue-execution

## ADDED Requirements

### Requirement: Очередь выполнения сценариев
Система SHALL исполнять сценарии асинхронно через очередь: основной процесс (NestJS API) при запросе запуска создаёт документ `Run` и помещает задачу выполнения в очередь Yandex Message Queue (совместимую с Amazon SQS); выделенные процессы-воркеры забирают задачи из очереди и исполняют сценарий (резолв маппингов, HTTP-вызовы endpoints, шаги delay/periodic/вложенный сценарий), не держа HTTP-запрос клиента. Воркеры и основной процесс взаимодействуют через очередь и общее хранилище состояния (`Run` в MongoDB), а не через разделяемую память.

#### Scenario: Запуск ставится в очередь
- **WHEN** аутентифицированный пользователь запускает опубликованный сценарий
- **THEN** система создаёт `Run` со статусом `queued`, помещает задачу в очередь и немедленно отвечает клиенту идентификатором запуска

#### Scenario: Воркер забирает и исполняет
- **WHEN** свободный воркер получает задачу из очереди
- **THEN** он загружает `Run` и сценарий, исполняет шаги и обновляет состояние запуска по мере выполнения

### Requirement: Жизненный цикл и статусы запуска
Система SHALL отслеживать статус каждого запуска и обеспечивать переходы: `queued` → `running` → (при необходимости `waiting_input` → `running`) → `done` | `error` | `cancelled`. Статус `running` означает активное исполнение воркером; `waiting_input` — пауза на шаге со страницей ввода/загрузки; `done` — успешное завершение; `error` — остановка из-за ошибки шага; `cancelled` — отмена пользователем. Клиент SHALL наблюдать переходы статуса в реальном времени.

#### Scenario: Переход в ожидание ввода
- **WHEN** исполнение доходит до шага со страницей ввода
- **THEN** статус запуска становится `waiting_input`, а после отправки пользователем данных — снова `running`

#### Scenario: Успешное завершение
- **WHEN** воркер завершает последний шаг без ошибок
- **THEN** статус запуска становится `done` с итоговым результатом

#### Scenario: Отмена запуска
- **WHEN** пользователь запрашивает отмену активного запуска
- **THEN** статус становится `cancelled` и исполнение прекращается перед очередным шагом

### Requirement: Сохранение промежуточных состояний и резюме
Воркер SHALL сохранять промежуточное состояние в документе `Run` после каждого шага (статус шага, входы, выходы, тайминги, журнал событий). При падении воркера (или истечении visibility timeout очереди) задача возвращается в очередь, другой воркер загружает `Run` и продолжает исполнение с последнего незавершённого шага без потери уже выполненных шагов и собранных данных.

#### Scenario: Резюме после сбоя воркера
- **WHEN** воркер падает после завершения 3 из 5 шагов
- **THEN** задача возвращается в очередь, новый воркер продолжает с шага 4, используя сохранённые выходы шагов 1–3

#### Scenario: Возобновление после ввода пользователя
- **WHEN** запуск находится в `waiting_input` и пользователь отправляет данные
- **THEN** основной процесс помещает resume-задачу в очередь, воркер продолжает исполнение с этим вводом

### Requirement: Ретраи и обработка сбоев очереди
Система SHALL обрабатывать сбои доставки по принципу at-least-once: задача остаётся видимой для повторного забора после истечения visibility timeout, если воркер не подтвердил завершение; после превышения максимального числа попыток получения (`maxReceiveCount`) задача перемещается в очередь недоставленных сообщений (dead-letter queue), а статус запуска становится `error` с указанием причины. Для длительных периодических шагов воркер SHALL продлевать visibility timeout (Yandex Message Queue `ChangeMessageVisibility`), удерживая задачу до завершения.

#### Scenario: Исчерпание попыток
- **WHEN** задача превышает `maxReceiveCount` неудачных попыток
- **THEN** она попадает в dead-letter queue, а запуск помечается `error`

#### Scenario: Удержание длительного шага
- **WHEN** воркер выполняет периодический шаг дольше текущего visibility timeout
- **THEN** он продлевает видимость сообщения, чтобы задача не была подхвачена повторно до завершения шага
