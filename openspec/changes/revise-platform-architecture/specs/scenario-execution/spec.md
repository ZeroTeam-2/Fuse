# scenario-execution

## MODIFIED Requirements

### Requirement: Загрузка файлов с докачкой
Для файловых шагов система SHALL применять порог разовой загрузки: файлы размером до 25 МБ (конфигурируемая переменная `UPLOAD_SINGLE_MAX_MB`, по умолчанию 25) загружаются одним запросом (multipart/form-data), файлы размером более 25 МБ — **обязательно** чанками (multipart/init-part/complete/abort, состояние частей в Mongo). Чанковая загрузка SHALL отображать прогресс (проценты и байты), поддерживать паузу/продолжение и отмену. При обрыве соединения загруженные чанки MUST сохраняться, и пользователь MAY возобновить загрузку с места обрыва. После загрузки система SHALL показывать стадию обработки (опрос статуса) и результат (извлечённые поля/текст) с возможностью загрузить другой файл. Файлы MUST проверяться на допустимый формат и лимит размера, заданные страницей шага.

#### Scenario: Небольшой файл — разовая загрузка
- **WHEN** пользователь загружает файл размером ≤25 МБ
- **THEN** система загружает его одним запросом без чанков

#### Scenario: Большой файл — чанковая загрузка
- **WHEN** пользователь загружает файл размером более 25 МБ
- **THEN** система принудительно использует чанковую загрузку с индикатором прогресса

#### Scenario: Возобновление после обрыва
- **WHEN** соединение обрывается на середине чанковой загрузки
- **THEN** система показывает «Соединение прервалось на X% — загруженные чанки сохранены» и после «Возобновить загрузку» продолжает с того же места

## ADDED Requirements

### Requirement: Поток выполнения в реальном времени через WebSocket
Система SHALL доставлять ход выполнения сценария пользователю через WebSocket (а не Server-Sent Events). Клиент подписывается на события конкретного запуска по `runId` и получает: переходы статуса запуска, события шагов (`step:start`, `step:done`, `page:required`, `run:done`, `run:error`) и прогресс. События журналируются в документе запуска с монотонным идентификатором, поэтому реконнект безопасен: клиент передаёт последний полученный идентификатор и получает только пропущенные события без потерь и дублей.

#### Scenario: Подписка и получение событий
- **WHEN** клиент открывает WS-соединение и подписывается на `runId` активного запуска
- **THEN** система начинает доставлять события шагов и изменения статуса в реальном времени

#### Scenario: Безопасный реконнект
- **WHEN** WS-соединение обрывается и клиент переподключается с последним получённым идентификатором события
- **THEN** система досылает только пропущенные события, не дублируя уже полученные
