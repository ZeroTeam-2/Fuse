# auth-profile

## MODIFIED Requirements

### Requirement: Регистрация и вход
Система SHALL аутентифицировать пользователя исключительно через Yandex ID (OAuth 2.0, Authorization Code): при первом входе аккаунт создаётся автоматически из профиля Yandex (имя, email, аватар), при последующих — данные обновляются (апсерт). Никакой регистрации или входа по email и паролю не существует. После успешного входа система SHALL выдавать сессию (JWT access+refresh) в httpOnly cookie. Все разделы, кроме публичного маркетплейса, MUST быть доступны только аутентифицированному пользователю.

#### Scenario: Первый вход через Yandex
- **WHEN** пользователь нажимает «Войти через Yandex» и разрешает доступ в Yandex ID
- **THEN** система создаёт аккаунт из профиля Yandex, выдаёт сессию и открывает маркетплейс с данными пользователя (инициалы/аватар в шапке)

#### Scenario: Повторный вход
- **WHEN** уже существующий пользователь входит через Yandex
- **THEN** система обновляет профиль из Yandex при необходимости, выдаёт новую сессию и открывает маркетплейс

#### Scenario: Отказ/ошибка авторизации Yandex
- **WHEN** пользователь отменяет доступ или токен Yandex невалиден
- **THEN** система отклоняет вход с сообщением об ошибке и не создаёт сессию

### Requirement: Профиль пользователя
Система SHALL хранить и позволять редактировать личные данные: ФИО, email, фото профиля. При первом входе личные данные SHALL заполняться из профиля Yandex ID; аватар по умолчанию берётся из Yandex, пользователь MAY загрузить свой или удалить его (тогда показываются инициалы). Личные данные MUST быть видны только самому пользователю и владельцам API, чьи сценарии он запускает.

#### Scenario: Изменение личных данных
- **WHEN** пользователь меняет ФИО и нажимает «Сохранить изменения»
- **THEN** система сохраняет данные и подтверждает сохранение

#### Scenario: Загрузка аватара
- **WHEN** пользователь загружает файл изображения
- **THEN** система сохраняет аватар и отображает его вместо инициалов; не-изображения отклоняются

## REMOVED Requirements

### Requirement: Смена пароля
**Reason**: Аутентификация переведена на Yandex ID OAuth 2.0 — паролей в системе больше нет.
**Migration**: Вход и управление доступом осуществляются через Yandex ID; смена пароля не требуется и не предусмотрена.
