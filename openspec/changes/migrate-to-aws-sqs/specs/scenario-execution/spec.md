## MODIFIED Requirements

### Requirement: Очередь исполнения и архитектура main/worker
Система SHALL исполнять сценарии через очередь задач (AWS SQS) по архитектуре main process + workers. Main process (API-сервер) создаёт документ `Run` со статусом `pending` и отправляет SQS-сообщение с payload `{ runId }` в очередь `scenario-execution`; worker process получает сообщение, исполняет шаги и сохраняет промежуточное состояние (результаты шагов, тайминги) в документ `Run` после каждого шага. При падении worker'а сообщение становится видимым в SQS после истечения visibility timeout и перезапускается, worker загружает промежуточное состояние из `Run` и продолжает с последнего сохранённого шага. Система SHALL поддерживать статусы `Run`: `pending`, `running`, `waiting_input`, `completed`, `failed`, `cancelled`. Параллельное масштабирование воркеров — через concurrency-настройку SQS consumer'а. Сообщения, не обработанные после 3 доставок, SHALL перемещаться в Dead Letter Queue для ручного разбора.

#### Scenario: Создание задачи на исполнение
- **WHEN** пользователь запускает сценарий
- **THEN** main process создаёт `Run` со статусом `pending`, отправляет SQS-сообщение `{ runId }` в очередь `scenario-execution` и возвращает `runId` для подключения к WebSocket

#### Scenario: Восстановление после падения worker'а
- **WHEN** worker падает на шаге K из N
- **THEN** SQS возвращает сообщение в очередь после истечения visibility timeout, worker перезапускает обработку, загружает промежуточное состояние из `Run` и продолжает с шага K без потери выполненных шагов

#### Scenario: Успешный запуск потока
- **WHEN** worker исполняет N шагов с валидными входами
- **THEN** система выполняет все шаги по порядку, передавая данные по маппингу, сохраняет результаты каждого шага в `Run` и возвращает итоговый результат с суммарным временем

#### Scenario: Ошибка стороннего API
- **WHEN** endpoint шага возвращает ошибку
- **THEN** `Run` переходит в статус `failed`, пользователь видит на каком шаге и какая ошибка произошла

#### Scenario: Отмена запуска до начала обработки
- **WHEN** пользователь отменяет запуск, а SQS-сообщение ещё не доставлено worker'у
- **THEN** `Run` переходит в статус `cancelled`, а worker при получении сообщения проверяет статус и завершает обработку без исполнения шагов

#### Scenario: Poison message после повторных отказов
- **WHEN** сообщение не обработано успешно после 3 доставок (worker падал каждый раз)
- **THEN** SQS перемещает сообщение в Dead Letter Queue `scenario-execution-dlq` для ручного разбора

### Requirement: Real-time прогресс через WebSocket
Система SHALL предоставлять real-time обновления прогресса выполнения сценария через WebSocket-соединение по `runId`. Worker публикует события (`step:start`, `step:done`, `page:required`, `run:done`, `run:error`, `run:status`) в комнату запуска; клиент, подписанный на `runId`, получает их немедленно. Worker и gateway живут в одном процессе, поэтому внешний брокер для ретрансляции не требуется.

При подключении клиента к `runId` сервер MUST немедленно отправить этому клиенту снапшот текущего состояния `Run` (статус, `currentStep`, накопленные `stepResults`, текст ошибки при наличии) — до и независимо от последующих инкрементальных событий. Клиент, подключившийся после начала исполнения (в том числе после завершения запуска, после реконнекта или перезагрузки страницы), MUST получить актуальное состояние и MUST NOT остаться без обратной связи из-за событий, отправленных до его входа в комнату.

#### Scenario: Подключение после старта исполнения
- **WHEN** worker начал (и, возможно, уже завершил) исполнение до того, как клиент подключился к `runId`
- **THEN** клиент при подключении получает снапшот с текущим статусом и результатами шагов и отображает актуальное состояние вместо бесконечного «Выполняем сценарий…»

#### Scenario: Быстрый запуск завершился до подключения
- **WHEN** запуск падает или завершается за миллисекунды — раньше, чем браузер успел открыть WebSocket
- **THEN** клиент при подключении получает снапшот со статусом `failed` (с текстом ошибки) либо `completed` (с результатами) и показывает соответствующий экран

#### Scenario: Reconnect после обрыва WebSocket
- **WHEN** WebSocket-соединение обрывается во время выполнения
- **THEN** клиент переподключается по `runId`, получает снапшот текущего статуса и результаты выполненных шагов без потери прогресса
