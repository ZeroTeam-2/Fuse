## ADDED Requirements

### Requirement: Воспроизводимый сид локальной среды

Проект SHALL предоставлять команду, которая наполняет локальную БД данными, достаточными для работы в UI: пользователь, приложение и сценарии. Команда MUST быть идемпотентной — повторный запуск не создаёт дубликатов и не падает. Сид MUST работать только против локального профиля окружения.

#### Scenario: Сид на пустой БД

- **WHEN** разработчик выполняет `pnpm seed` после `pnpm infra:reset`
- **THEN** в БД появляются сид-пользователь, демо-приложение и сценарии
- **AND** в UI список сценариев непустой, и сценарий можно открыть в редакторе

#### Scenario: Повторный сид не плодит дубликаты

- **WHEN** `pnpm seed` выполняется повторно на уже засеянной БД
- **THEN** команда завершается успешно
- **AND** количество сид-пользователей, приложений и сценариев не увеличивается

#### Scenario: Сид не запускается против удалённой БД

- **WHEN** локальный профиль (`.env.local`) не активен, то есть `MONGODB_URL` указывает не на локальный контейнер
- **THEN** сид отказывается выполняться с внятной ошибкой
- **AND** ни одна запись в удалённую БД не создаётся

### Requirement: Исполнимый сид-сценарий

Сид SHALL создавать сценарий, запуск которого доходит до терминального статуса без обращения к внешним сервисам. Для шагов, которым нужен HTTP-эндпоинт, локальная инфраструктура SHALL предоставлять мок-сервис, и `baseUrl` сид-приложения MUST указывать на него.

#### Scenario: Герметичный сценарий доходит до конца

- **WHEN** сид-сценарий без внешних вызовов запускается на локальной среде
- **THEN** запуск проходит через локальную очередь и локальный воркер
- **AND** статус доходит до терминального (`completed`), а не зависает в «Выполняем сценарий…»

#### Scenario: API-шаг обращается к мок-сервису

- **WHEN** запускается сид-сценарий с шагом типа `api`
- **THEN** воркер обращается к мок-сервису из локальной инфраструктуры, а не к внешнему API
- **AND** запуск завершается терминальным статусом

### Requirement: Вход тестов без OAuth

Тесты и инструменты SHALL иметь возможность действовать от имени сид-пользователя без прохождения Yandex OAuth: JWT подписывается тем же `JWT_SECRET` и передаётся в куке `access_token`. Приложение MUST NOT содержать режима, отключающего проверку авторизации: guard, стратегия и проверка подписи остаются в силе на всех окружениях.

#### Scenario: Аутентификация подписанной кукой

- **WHEN** запрос к защищённому эндпоинту приходит с валидной кукой `access_token` для сид-пользователя
- **THEN** запрос обрабатывается как обычный авторизованный запрос
- **AND** OAuth-флоу при этом не задействуется

#### Scenario: Обхода авторизации не появляется

- **WHEN** приложение запущено в любом окружении, включая локальное
- **THEN** запрос без валидного токена к защищённому эндпоинту отклоняется с 401
- **AND** в коде нет флага или ветки, отключающей проверку токена
