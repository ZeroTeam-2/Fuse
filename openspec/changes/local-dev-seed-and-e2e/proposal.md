## Why

После `pnpm infra:reset` локальная БД пустая: нет пользователя, приложения и сценария, поэтому в UI нечего открыть и нечего запустить. Войти тоже нельзя — единственный вход через Yandex OAuth. Из-за этого проверка изменения `local-docker-infra-dev` упёрлась в задачу 4.4: сквозной запуск сценария из UI подтвердить не удалось (round-trip через очередь проверен напрямую, но не через интерфейс), а UI-тестов в проекте нет вовсе — `apps/frontend/e2e/smoke.spec.ts` импортирует `@playwright/test`, которого нет в зависимостях, то есть не запускается.

Итог: локальная среда воспроизводима, но непригодна к работе без ручного наполнения, а регрессии в UI (например, недавние дубли сценариев при ошибке навигации) ловятся только руками.

## What Changes

- **Сид локальной БД** (`pnpm seed`): пользователь, демо-приложение и два сценария — герметичный `delay`-сценарий и `api`-сценарий на мок-сервис. Идемпотентен: повторный запуск не плодит дубли.
- **Мок-API в инфраструктуре**: контейнер с HTTP-моком в `docker-compose.dev.yml`, на который смотрит `baseUrl` сид-приложения. Даёт живой эндпоинт для `api`-шагов без обращения к внешним сервисам.
- **Вход без OAuth для тестов**: JWT для сид-пользователя подписывается тем же `JWT_SECRET` и кладётся в куку `access_token` — ту же, что проверяет `JwtStrategy`. **Обхода авторизации в коде приложения не появляется**: guard, стратегия и проверка подписи остаются нетронутыми.
- **E2E на Playwright** (`pnpm test:e2e`): подключается зависимость `@playwright/test`, добавляется конфиг, оживает существующий `e2e/smoke.spec.ts`. Тесты: создание сценария создаёт ровно один (регрессия на дубли) и запуск сценария доходит до терминального статуса — это закрывает задачу 4.4 репитабельно.
- **Smoke-тест инфраструктуры** (`pnpm test:infra`): очередь и DLQ существуют, бакет существует, round-trip в SQS и S3 проходит.

## Capabilities

### New Capabilities
- `local-dev-seed`: воспроизводимое наполнение локальной среды — сид-данные, идемпотентность, мок-API для шагов, вход тестов под сид-пользователя без OAuth.
- `local-dev-testing`: автоматические проверки локальной среды — smoke-тест инфраструктуры и e2e-тесты UI поверх сид-данных.

## Impact

- **Новое**: `scripts/seed.ts` (или `apps/backend/scripts/`), `apps/frontend/playwright.config.ts`, e2e-тесты, smoke-тест инфраструктуры, мок-сервис в `docker-compose.dev.yml`.
- **Зависимости**: `@playwright/test` (dev) + загрузка браузеров.
- **Конфигурация**: скрипты `seed`, `test:e2e`, `test:infra` в корневом `package.json`.
- **Код приложения не меняется.** Ни guard, ни `JwtStrategy`, ни OAuth-флоу не трогаются — тесты входят валидной подписанной кукой, как обычный пользователь.
- **Закрывает** задачу 4.4 изменения `local-docker-infra-dev` и его Open Question про сид.
- **Риск**: сид-данные должны переживать эволюцию схем (Step/App/Scenario). Смягчение: сид пишется поверх тех же типов из `@fuse/shared`, поэтому расхождение ловится компилятором.
