## MODIFIED Requirements

### Requirement: Real-time прогресс через WebSocket
Система SHALL предоставлять real-time обновления прогресса выполнения сценария через WebSocket-соединение по `runId`. Worker публикует события (`step:start`, `step:done`, `page:required`, `input:required`, `run:done`, `run:error`, `run:status`) в комнату запуска; клиент, подписанный на `runId`, получает их немедленно.

При подключении клиента к `runId` сервер MUST немедленно отправить этому клиенту снапшот текущего состояния `Run` (статус, `currentStep`, накопленные `stepResults`, текст ошибки при наличии) — до и независимо от последующих инкрементальных событий. Клиент, подключившийся после начала исполнения (в том числе после завершения запуска, после реконнекта или перезагрузки страницы), MUST получить актуальное состояние и MUST NOT остаться без обратной связи из-за событий, отправленных до его входа в комнату.

**Поток событий MUST NOT быть единственным источником прогресса.** Живое соединение, по которому ничего не приходит, неотличимо для клиента от «сценарий ещё выполняется», поэтому потерянный поток событий (запуск исполнил другой процесс, worker умер, gateway моргнул) оставляет пользователя в бесконечном «Выполняем сценарий…». Клиент SHALL вести сторож тишины: таймер сбрасывается каждым входящим событием, и если событий не было дольше порога, клиент MUST один раз перечитать состояние запуска (`GET /api/runs/:id`) и применить его теми же правилами, что и снапшот. При продолжающейся тишине интервал MUST расти (до потолка), чтобы длинные шаги (задержка, периодический опрос) не превращались в череду запросов. В запуске с нормальным потоком событий сторож MUST NOT сделать ни одного запроса; в фазе ожидания пользовательского ввода тишина ожидаема, и перечитывание MUST NOT затирать уже введённые в форму значения.

**Масштабирование воркеров требует общей шины.** События доставляются клиенту тем процессом, в котором живёт его сокет; сегодня worker и gateway — один процесс, поэтому `emit` в комнату долетает. Как только воркеры разносятся на несколько процессов (что предусмотрено требованием об очереди), событие, опубликованное одним процессом, MUST доходить до клиента, подключённого к другому, — через общий backplane (socket.io Redis adapter либо эквивалент). Без него шаги исполняются, а UI об этом не узнаёт.

#### Scenario: Подключение после старта исполнения
- **WHEN** worker начал (и, возможно, уже завершил) исполнение до того, как клиент подключился к `runId`
- **THEN** клиент при подключении получает снапшот с текущим статусом и результатами шагов и отображает актуальное состояние вместо бесконечного «Выполняем сценарий…»

#### Scenario: Быстрый запуск завершился до подключения
- **WHEN** запуск падает или завершается за миллисекунды — раньше, чем браузер успел открыть WebSocket
- **THEN** клиент при подключении получает снапшот со статусом `failed` (с текстом ошибки) либо `completed` (с результатами) и показывает соответствующий экран

#### Scenario: Reconnect после обрыва WebSocket
- **WHEN** WebSocket-соединение обрывается во время выполнения
- **THEN** клиент переподключается по `runId`, получает снапшот текущего статуса и результаты выполненных шагов без потери прогресса

#### Scenario: Поток событий потерян при живом соединении
- **WHEN** соединение установлено, но события по запуску не приходят (например, запуск исполнил другой процесс, чьи события ушли в его собственный gateway), а запуск при этом уже завершился
- **THEN** после порога тишины клиент перечитывает состояние запуска, показывает результат и MUST NOT остаться в бесконечном «Выполняем сценарий…»

#### Scenario: Нормальный запуск не порождает лишних запросов
- **WHEN** события `step:start` / `step:done` / `run:done` приходят потоком
- **THEN** сторож тишины сбрасывается каждым событием и не делает ни одного запроса за состоянием

### Requirement: Движок исполнения шагов
Worker SHALL исполнять шаги сценария последовательно: для каждого шага разрешать входы по маппингу (выходы предыдущих шагов, ввод пользователя, константы, включая подстановку `{{token}}`-шаблонов), выполнять HTTP-вызов endpoint стороннего API и сохранять ответ как выходы шага в документ `Run`.

Пользовательский ввод MUST быть доступен **каждому** шагу запуска, а не только тому, на котором он был введён: worker MUST засеивать ввод значениями, собранными перед запуском (`Run.inputs`), и переносить накопленный ввод от шага к шагу. Данные страницы ввода шага MUST дополнять накопленный ввод, а не заменять его. Шаг «Другой сценарий» MUST получать срез входов, адресованных его вложенным шагам.

Для шагов, привязанных к приложению (`api`, `periodic`, `file`), worker MUST разрешать **абсолютный URL** вызова: загружать `App` по `appId` шага и склеивать `App.baseUrl` с путём шага (`path` / `pollPath` / `uploadPath`). Относительный путь шага MUST NOT передаваться в HTTP-клиент напрямую. Если приложение не найдено либо у него не задан `baseUrl`, шаг MUST завершаться доменной ошибкой с диагностикой, указывающей на приложение и на необходимость переимпорта спецификации. Резолвнутый URL MUST проходить проверку SSRF перед вызовом.

Шаг «Задержка» MUST приостанавливать поток на заданное время; «Периодический запрос» MUST опрашивать endpoint с интервалом до признака завершения, публикуя прогресс; «Другой сценарий» MUST исполнять вложенный сценарий как один шаг. Ошибка шага MUST переводить `Run` в статус `failed` с диагностикой (шаг, endpoint, статус ответа).

Запись результатов шагов MUST быть идемпотентной: повторная обработка одного и того же `runId` (повторная доставка сообщения очереди) MUST NOT создавать дублирующиеся записи `stepResults` с одинаковым `stepIndex` и MUST NOT переводить `Run` из терминального статуса (`completed`, `failed`, `cancelled`) обратно в незавершённый.

#### Scenario: Резолв базового URL приложения
- **WHEN** worker исполняет `api`-шаг с `path: "/collections"` у приложения с `baseUrl: "https://dadata-fake.cloud.astral-dev.ru"`
- **THEN** worker вызывает `https://dadata-fake.cloud.astral-dev.ru/collections` и сохраняет тело ответа как результат шага

#### Scenario: Base path приложения сохраняется
- **WHEN** у приложения `baseUrl: "https://api.example.com/v1"`, а путь шага — `/collections`
- **THEN** worker вызывает `https://api.example.com/v1/collections`

#### Scenario: У приложения не задан базовый URL
- **WHEN** worker исполняет шаг приложения, у которого `baseUrl` пуст
- **THEN** шаг завершается ошибкой с сообщением, называющим приложение и предлагающим переимпортировать спецификацию, а `Run` переходит в `failed` — без низкоуровневой ошибки разбора URL

#### Scenario: Подстановка шаблона в константе
- **WHEN** константа шага содержит `Bearer {{access_token}}` и предыдущий шаг вернул `access_token`
- **THEN** worker подставляет значение токена в заголовок Authorization перед HTTP-вызовом

#### Scenario: Пользовательский ввод доступен на всех шагах
- **WHEN** пользователь ввёл значения перед запуском либо на странице ввода шага 1, а ручной ввод помечен также у шага 3
- **THEN** шаг 3 видит накопленный пользовательский ввод и разрешает свой `user`-источник из него, а не получает пустое значение

#### Scenario: Периодический опрос до завершения
- **WHEN** шаг «Периодический запрос» опрашивает endpoint каждые 5 сек до признака завершения в ответе
- **THEN** worker публикует прогресс (значение поля прогресса) на каждой итерации и завершает шаг при получении признака завершения

#### Scenario: Повторная доставка сообщения не дублирует результаты
- **WHEN** сообщение с одним и тем же `runId` доставляется worker'у повторно
- **THEN** результаты шагов перезаписываются по индексу, дубликаты `stepResults` не появляются, а уже терминальный `Run` не возвращается в `running` или `waiting_input`

#### Scenario: Доменная ошибка шага не уходит на повторную доставку
- **WHEN** шаг падает детерминированно (недоступный endpoint, ошибка стороннего API)
- **THEN** worker фиксирует `failed` в `Run`, публикует ошибку и подтверждает сообщение очереди — повторные доставки того же детерминированного сбоя не выполняются

### Requirement: Простой режим запуска
Вкладка «Запуск» SHALL по умолчанию предлагать простой режим: форма ручных входов сценария, кнопка «Получить результат», индикация выполняемых шагов (название, сервис, статус), и итоговая сводка результата с кнопкой «Запустить снова».

Форма запуска MUST собирать **все** значения сценария, помеченные ручным вводом, по всем шагам — и параметры, и значения условий фильтрации — а не только входы первого шага. Поля MUST быть сгруппированы по шагам и подписаны шагом и параметром; обязательные поля MUST блокировать запуск. Значения, которые запросит страница ввода своего шага, MUST NOT дублироваться в этой форме. Если ручных значений у сценария нет, форма MUST NOT показываться.

Итоговая сводка MUST отображать **данные результата** — полезную нагрузку последнего успешного шага (объект — как пары «поле → значение»; коллекция — как количество записей и предпросмотр; скаляр — как есть), а не только тайминги шагов. Суммарное время выполнения MUST вычисляться по фактическим длительностям шагов. Счётчик запусков карточки MUST инкрементироваться при каждом завершённом запуске.

#### Scenario: Запуск в один клик
- **WHEN** пользователь вводит значение (например, ИНН) и нажимает «Получить результат»
- **THEN** система показывает прогресс по шагам и итоговую сводку без отображения технических деталей

#### Scenario: Форма собирает ручные значения всех шагов
- **WHEN** ручной ввод помечен у параметра шага 1 и у значения условия фильтрации шага 3
- **THEN** форма запуска показывает оба поля с указанием их шагов, и оба значения доезжают до соответствующих шагов при исполнении

#### Scenario: Результат содержит данные ответа
- **WHEN** сценарий из одного шага «Получить список организаций» завершается успешно
- **THEN** пользователь видит данные организаций, возвращённые API, а не только длительность шага

#### Scenario: Ошибка запуска видна пользователю
- **WHEN** запуск завершается статусом `failed`
- **THEN** пользователь видит экран ошибки с текстом причины и шагом, на котором она произошла

### Requirement: Режим playground
Система SHALL предоставлять пошаговый режим: каждый шаг показывает метод, путь, сервис, описание и поле ввода; пользователь запускает шаги по одному и видит JSON-ответ каждого шага с временем выполнения; MUST быть кнопки «Сбросить» и запуск всех шагов подряд с финальным статусом «Сценарий выполнен» и суммарным временем.

Сбор ручных входов в playground MUST вести себя так же, как в простом режиме: перед запуском показывается та же форма со всеми ручными значениями сценария, обязательные поля блокируют запуск, а запрос недостающего значения по ходу исполнения отображается так же.

#### Scenario: Пошаговое выполнение
- **WHEN** пользователь выполняет шаги по одному в playground
- **THEN** после каждого шага отображается его ответ (JSON, тайминг, сервис), а после последнего — итоговый статус

#### Scenario: Паритет сбора входов с простым режимом
- **WHEN** пользователь запускает в playground сценарий с ручными значениями на разных шагах
- **THEN** он заполняет ту же форму ручных входов, что и в простом режиме, и обязательные поля так же блокируют запуск

### Requirement: Резолв массивных выходов шага

При разрешении входа, замапленного на выход предыдущего шага, worker SHALL учитывать, что результат шага-источника может быть **массивом** (сам результат — коллекция либо массивом является указанное поле результата). Значение поля MUST браться из **одного элемента** массива, а не из массива целиком.

Если для входа задано условие фильтрации, worker MUST отобрать элементы массива по этому условию и взять поле из отобранного элемента. Значение сравнения MUST разрешаться теми же правилами, что и маппинг: константа (с подстановкой `{{s0:key}}`-шаблонов), выход другого шага либо пользовательский ввод. Перед сравнением значение MUST приводиться к типу поля элемента; сравнения порядка (`>`, `<`, `≥`, `≤`) для чисел MUST быть числовыми, а не строковыми.

Поведение при неоднозначной выборке:
- **0 совпадений** — шаг MUST завершаться доменной ошибкой, называющей шаг, поле и условие; `Run` переходит в `failed`. Тихая подстановка пустого значения MUST NOT происходить.
- **более одного совпадения** — worker MUST брать первый элемент и записывать предупреждение в результат шага; исполнение продолжается.
- **массив без заданного условия** — worker MUST брать первый элемент (обратная совместимость со сценариями, настроенными до появления фильтров), а не подставлять пустое значение.

Пользовательский ввод MUST доходить до резолвера маппингов — как для источника «ручной ввод» у самого параметра, так и для значения условия фильтрации — независимо от того, есть ли у шага страница ввода. Значения ручного ввода у шагов **без** страницы собираются формой запуска и добираются по ходу исполнения (см. capability `manual-input-collection`). Значение условия фильтрации, помеченное ручным вводом и не заданное к моменту исполнения шага, MUST запрашиваться у пользователя, а не приводить к ошибке «условие не отобрало ни одного элемента».

#### Scenario: Отбор элемента по условию

- **WHEN** шаг 1 вернул массив организаций, а вход шага 2 замаплен на `s0:id` с условием «`inn` = 7707083893»
- **THEN** worker находит в массиве организацию с этим ИНН и подставляет во вход её `id`

#### Scenario: Условие не дало совпадений

- **WHEN** фильтр по массиву результата шага-источника не отобрал ни одного элемента
- **THEN** шаг завершается ошибкой с диагностикой (шаг, поле, условие), `Run` переходит в `failed`, и следующие шаги не исполняются с пустым значением

#### Scenario: Условию соответствует несколько элементов

- **WHEN** условию удовлетворяют три элемента массива
- **THEN** worker берёт первый из них, записывает в результат шага предупреждение о неоднозначности и продолжает исполнение

#### Scenario: Массив без условия

- **WHEN** вход замаплен на выход шага, который вернул массив, а условие фильтрации не задано (сценарий настроен до появления фильтров)
- **THEN** worker берёт поле из первого элемента массива вместо подстановки пустого значения

#### Scenario: Числовое сравнение

- **WHEN** условие задано как «`amount` > 1000», а элементы содержат `amount: 900` и `amount: 1200`
- **THEN** worker сравнивает значения как числа и отбирает элемент с `amount: 1200` (а не сравнивает строки посимвольно)

#### Scenario: Значение условия приходит со страницы шага

- **WHEN** значение условия настроено как «ручной ввод», а у шага есть страница ввода, на которой пользователь его указал
- **THEN** worker получает это значение из данных страницы и применяет его в условии фильтрации

#### Scenario: Значение условия у шага без страницы ввода

- **WHEN** значение условия настроено как «ручной ввод», а страницы ввода у шага нет
- **THEN** значение берётся из входов запуска, собранных формой перед стартом, и условие отбирает элемент — вместо падения шага с «условие не отобрало ни одного элемента»
