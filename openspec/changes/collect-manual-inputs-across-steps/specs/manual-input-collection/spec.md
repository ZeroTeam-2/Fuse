## ADDED Requirements

### Requirement: Перечисление ручных значений сценария

Система SHALL уметь перечислить все значения сценария, помеченные как ручной ввод, единым расчётом на сервере. В перечисление MUST входить два вида значений:

1. **параметр шага** — входное поле, чей источник в маппинге задан как ручной ввод (`mappings[key] = "user"`);
2. **значение условия фильтрации** — операнд сравнения, чей режим задан как ручной ввод (`filter.value.mode = "user"`), адресуемый ключом `filter:{paramKey}`.

Перечисление MUST обходить **все** шаги сценария по порядку, а для шага «Другой сценарий» — рекурсивно спускаться во вложенный сценарий, сохраняя путь до шага. Каждый пункт перечисления MUST нести: скоуп-ключ (уникальный в пределах запуска, включающий путь до шага), индекс и заголовок шага, ключ параметра, вид значения (параметр или условие фильтрации), человекочитаемую подпись, тип значения, признак обязательности и **источник** — спросит ли значение общая форма перед запуском или страница ввода этого шага.

Подпись и тип параметра MUST браться из схемы endpoint'а шага, а для значения условия фильтрации — из схемы поля элемента массива, по которому идёт сравнение. Значение условия фильтрации MUST всегда считаться обязательным: без него условие не отбирает ни одного элемента и шаг падает.

Одно и то же перечисление MUST использоваться и клиентом (для построения формы), и воркером (для проверки полноты входов) — расхождение между тем, что спросили, и тем, что требуется при исполнении, MUST NOT возникать.

#### Scenario: Ручные значения собраны со всех шагов

- **WHEN** в сценарии из четырёх шагов ручной ввод помечен у параметра шага 1, у значения условия фильтрации шага 2 и у параметра шага 4
- **THEN** перечисление возвращает три пункта — по одному на каждое значение — каждый с указанием своего шага, ключа параметра и типа

#### Scenario: Ручные значения вложенного сценария

- **WHEN** шаг сценария — «Другой сценарий», внутри которого есть параметр с ручным вводом
- **THEN** перечисление включает это значение с ключом, содержащим путь до шага вложенного сценария, и оно не конфликтует с одноимённым параметром внешнего сценария

#### Scenario: Сценарий без ручных значений

- **WHEN** ни один параметр и ни одно условие фильтрации сценария не помечены ручным вводом
- **THEN** перечисление пусто, и запуск сценария не требует от пользователя никакого ввода

### Requirement: Форма ручных входов перед запуском

Перед запуском сценария система SHALL показывать пользователю форму со всеми перечисленными ручными значениями сценария. Форма MUST группировать поля по шагам и для каждого поля показывать, к какому шагу и параметру оно относится, а для условия фильтрации — что вводится значение сравнения.

Поле ввода MUST соответствовать типу значения (число вводится как число, логическое — как переключатель, строка — как текст), а введённое значение MUST приводиться к типу параметра перед отправкой. Обязательные поля MUST блокировать запуск: пока хотя бы одно не заполнено, кнопка запуска неактивна, а незаполненные поля подсвечены.

Если у сценария нет ручных значений, форма MUST NOT показываться — запуск остаётся в один клик.

#### Scenario: Форма показывает принадлежность полей

- **WHEN** пользователь открывает запуск сценария, где ручной ввод помечен у параметров разных шагов
- **THEN** форма показывает поля, сгруппированные по шагам, с заголовком шага и подписью параметра, а не безымянный список

#### Scenario: Обязательное поле блокирует запуск

- **WHEN** в форме есть незаполненное обязательное значение (обязательный параметр endpoint'а либо значение условия фильтрации)
- **THEN** кнопка запуска неактивна, и сценарий не стартует, пока значение не введено

#### Scenario: Запуск без ручных значений

- **WHEN** у сценария нет значений, помеченных ручным вводом
- **THEN** пользователь запускает сценарий одной кнопкой, без формы

### Requirement: Доставка ручных входов в исполнение

Ручные значения, собранные формой, SHALL передаваться в запуск и сохраняться в документе `Run`. Воркер MUST засеивать ими пользовательский ввод шага и MUST сохранять этот ввод **на протяжении всего запуска**: значения, собранные перед стартом, MUST быть доступны каждому шагу, а не только первому. Данные страницы ввода шага MUST дополнять этот ввод, а не заменять его.

Шаг «Другой сценарий» MUST получать срез входов, адресованных его вложенным шагам.

Хранение входов в `Run` MUST переживать перезапуск воркера: после падения и повторной обработки того же запуска ранее введённые значения MUST NOT запрашиваться повторно.

При разрешении маппинга отсутствующее ручное значение MUST NOT подменяться суррогатом: подстановка всего объекта пользовательского ввода вместо конкретного поля MUST NOT происходить. Отсутствие **необязательного** значения MUST давать пустое значение с предупреждением в результате шага; отсутствие **обязательного** — запрос значения у пользователя (см. требование о доборе).

#### Scenario: Значение доезжает до шага без страницы ввода

- **WHEN** у шага 3 параметр помечен ручным вводом, страницы ввода у шага нет, а пользователь ввёл значение в форме перед запуском
- **THEN** воркер подставляет это значение во вход шага 3 и вызывает endpoint с ним, а не с пустым значением

#### Scenario: Значение условия фильтрации доезжает до фильтра

- **WHEN** значение условия фильтрации помечено ручным вводом и введено в форме перед запуском
- **THEN** воркер применяет его в условии отбора элемента массива, и шаг не падает с «условие не отобрало ни одного элемента»

#### Scenario: Ввод не теряется между шагами

- **WHEN** пользователь ввёл значения перед запуском, а исполнение дошло до последнего шага сценария
- **THEN** последний шаг видит те же введённые значения, что и первый

#### Scenario: Отсутствующий необязательный вход не подменяется объектом входов

- **WHEN** необязательный параметр помечен ручным вводом, но значение для него не задано
- **THEN** параметр разрешается в пустое значение с предупреждением в результате шага, и в него MUST NOT попадать объект пользовательского ввода целиком

### Requirement: Добор недостающих значений по ходу исполнения

Если перед исполнением шага обязательного ручного значения нет во входах запуска, воркер SHALL запросить его у пользователя, а не падать и не отправлять пустоту. Запуск MUST переходить в статус `waiting_input`, а система MUST публиковать событие запроса ввода с указанием шага и списка недостающих значений (тех же дескрипторов, что и в форме запуска). Клиент MUST показать те же поля, что и форма запуска, и отправить введённые значения; воркер MUST продолжить исполнение с ними.

Добранные значения MUST дописываться во входы `Run`, чтобы пережить перезапуск воркера и не запрашиваться повторно. Ожидание ввода MUST быть ограничено тем же таймаутом, что и ожидание страницы ввода; по истечении таймаута шаг MUST завершаться ошибкой с диагностикой. Отмена запуска во время ожидания MUST прерывать ожидание.

Необязательные значения MUST NOT вызывать добор: их отсутствие — осознанный выбор пользователя.

#### Scenario: Запуск через API без входов

- **WHEN** запуск создан без ручных значений (например, программно), а шагу требуется обязательное значение ручного ввода
- **THEN** запуск переходит в `waiting_input`, пользователь получает запрос с полем этого значения, и после отправки исполнение продолжается

#### Scenario: Добор не повторяется после перезапуска воркера

- **WHEN** пользователь ввёл добранное значение, после чего воркер упал и запуск обрабатывается повторно
- **THEN** воркер берёт значение из входов `Run` и не запрашивает его снова

#### Scenario: Таймаут ожидания добора

- **WHEN** запрошенное значение не введено в течение таймаута ожидания ввода
- **THEN** шаг завершается ошибкой с диагностикой (шаг, параметр), а `Run` переходит в `failed`

#### Scenario: Необязательное значение не запрашивается

- **WHEN** необязательный параметр с ручным вводом остался незаполненным
- **THEN** воркер не останавливает исполнение и не запрашивает значение, а продолжает шаг без него
