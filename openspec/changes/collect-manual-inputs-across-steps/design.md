## Context

Ручной ввод в системе держится на одном механизме — странице ввода шага. `worker.service.ts:executeStep` дожидается `waitForPageInput`, кладёт результат в `ctx.userInput`, и только оттуда резолвер (`mapping-resolver.ts`) берёт значения для источника `"user"` и для операнда фильтра (`filterInputKey` → `filter:{paramKey}`). Всё остальное — дыры:

- `POST /api/runs` принимает `{ scenarioId }`; у `Run` нет поля входов; форма запуска (`RunPanel.vue:11-37`) не собирает вообще ничего, хотя спека обещает «форму с входами первого шага».
- `StepContext` пересоздаётся на каждой итерации цикла шагов **без** `userInput` (`worker.service.ts:249-255`) — введённое на шаге 1 не видно шагу 2. Наследует ввод только вложенный сценарий (`:657`).
- Резолвер при отсутствии ключа подставляет в параметр весь объект ввода: `userInput?.[field] ?? userInput` (`:211`), и на это есть юнит-тест.
- **Поле страницы нечем связать с параметром.** Ключ поля не редактируется: он выводится слагификацией подписи (`PageEditor.syncFieldKey`, `:49-56`, кириллица допускается) и показывается read-only (`:124`). Подпись «ИНН организации» даёт ключ `инн_организации`, а резолвер ищет `inn`. Связать поле с параметром можно, лишь назвав подпись английским ключом параметра — тогда пользователь при запуске видит поле с подписью `orgId`. Ключ `filter:{paramKey}` недостижим в принципе (двоеточие вырезается), то есть операнд фильтра в режиме «ручной ввод» сегодня не может прийти ниоткуда.
- `StepConfig.vue:738` обещает «общую форму», которой не существует.

Ограничения площадки: NestJS + Mongoose (не Prisma), очередь — SQS, real-time — socket.io (namespace `runs`, комната `run:{runId}`), клиент→сервер команды идут по HTTP, а не по сокету. Шаги сценария хранятся как `Mixed` и **не валидируются** на сервере. Уже существуют ровно те примитивы, которые нужны для запроса значения посреди запуска: статус `waiting_input`, событие `page:required`, эндпоинт `POST /api/runs/:id/page-submit` и почтовый ящик `Run.pageData` (пишется, вычитывается, `$unset`-ится).

Продуктовые решения приняты до дизайна: **гибрид** (общая форма перед запуском + добор недостающего по ходу) и **страница закрывает свой шаг** (её поля не спрашиваются повторно).

## Goals / Non-Goals

**Goals:**
- Один серверный расчёт списка ручных значений сценария, которым пользуются и форма, и воркер, — чтобы «что спросили» и «что нужно при исполнении» не могли разойтись.
- Ручное значение доезжает до резолвера у **любого** шага: со страницей ввода и без, для параметра и для операнда фильтра, во внешнем сценарии и во вложенном.
- Обязательное значение либо собрано формой до старта, либо запрошено по ходу; молча пустым в HTTP-вызов не уезжает никогда.
- Паритет простого режима и playground.

**Non-Goals:**
- Редактор страниц вообще: типы полей на странице (там до сих пор только текст), реальная валидация `required` при сабмите страницы, превью, файловый шаг. В охвате — только привязка поля к значению шага (решение 10), без которой «страница закрывает свой шаг» не выполняется.
- Общая шина событий для нескольких процессов (socket.io Redis adapter) и разведение окружений (своя очередь и MinIO для дева) — обе задачи вскрылись при проверке (решение 11), обе выносятся отдельно. Здесь — только клиентский сторож тишины, чтобы потерянный поток событий не оборачивался бесконечным «Выполняем сценарий…».
- Пресеты/запоминание введённых значений между запусками, шаблоны запуска.
- Валидация шагов сценария на сервере как таковая (DTO шагов по-прежнему `Mixed`).
- Защита исполнителя от циклических ссылок между сценариями — её нет и сейчас; здесь только гарантируем, что **перечисление** не зациклится.

## Decisions

### 1. Перечисление ручных значений живёт на сервере, в одном модуле

Новый модуль `apps/backend/src/execution/manual-inputs.ts`: чистая функция обхода шагов + сервис, дорезающий подписи и типы из схем endpoint'ов (через `ScenariosService.getStepSchema`, который уже умеет и `api`, и `scenario`-шаги). Наружу — `GET /api/scenarios/:id/manual-inputs`, рядом с существующим `step-schema/:index`. Воркер импортирует тот же модуль.

*Почему не считать список на фронте.* Фронт умеет тянуть `step-schema/:index` и мог бы обойти шаги сам, но тогда правило «что считается ручным значением» существовало бы в двух местах (фронт для формы, воркер для проверки полноты) и разъехалось бы на первом же изменении — например, когда у фильтра появится второй операнд. Воркеру список нужен в любом случае: без него он не знает, чего ждать при доборе.

Дескриптор (в `packages/shared`):

```ts
export interface ManualInputDescriptor {
  key: string;              // скоуп-ключ запуска, см. решение 2
  stepPath: number[];       // путь до шага: [3] или [2, 0] для вложенного
  stepIndex: number;        // индекс шага на своём уровне
  stepTitle: string;
  paramKey: string;         // ключ входа шага
  kind: "param" | "filter"; // сам параметр или операнд его условия фильтрации
  label: string;
  type: SchemaField["type"];
  required: boolean;
  source: "form" | "page";  // кто спрашивает: общая форма или страница шага
}
```

Тип и подпись `param` берутся из `inputs` схемы шага. Для `kind: "filter"` — из схемы **элемента** массива шага-источника: источник известен из `mappings[paramKey] = "s{idx}:{outKey}"`, поле сравнения — из `filter.field`, тип — из `items` соответствующего `SchemaField` выхода шага-источника. Если схема бедная и типа нет — деградируем в `string`.

`required`: для `param` — `SchemaField.required`; для `filter` — **всегда true** (без операнда условие не отбирает ничего и шаг падает).

Сломанный шаг (нет приложения/endpoint'а) MUST NOT ронять экран запуска: такой шаг пропускается в перечислении с пометкой, экран запуска остаётся рабочим.

### 2. Скоуп-ключи: путь до шага + локальный ключ, который уже понимает резолвер

Ключ входа запуска: `s{i}` через точку по пути вложенности, двоеточие, локальный ключ.

```
s0:orgId            параметр orgId шага 0
s1:filter:status    операнд условия фильтрации параметра status шага 1
s2.s0:orgId         параметр orgId шага 0 вложенного сценария, лежащего на шаге 2
```

Локальная часть — ровно то, что резолвер ищет сегодня (`field` и `filter:{field}`). Поэтому **контракт `mapping-resolver` не меняется**: воркер, входя в шаг с путём `P`, отбирает из `Run.inputs` записи с префиксом `P`, срезает префикс и получает привычный плоский `Record<string, unknown>`. Резолвер не знает ни про скоупы, ни про вложенность.

*Почему не плоский ключ.* Два шага одного сценария сплошь и рядом имеют вход с одинаковым ключом (`id`, `guid`, `inn`) и разным смыслом. Плоский ящик склеил бы их в одно поле формы — и молча подставил бы значение одного шага в другой.

### 3. `Run` хранит входы; почтовый ящик ввода честно переименовывается

- `Run.inputs?: Record<string, unknown>` (`Mixed`) — значения по скоуп-ключам, собранные формой; сюда же **дописываются** добранные по ходу значения, чтобы пережить перезапуск воркера и не спрашиваться повторно.
- `Run.pageData` → `Run.pendingInput?: { stepIndex: number; data: Record<string, unknown> }` — тот же одноместный ящик, но теперь он обслуживает и страницу, и добор. Воркер на переходный период читает `run.pendingInput ?? run.pageData`, чтобы запуски в полёте на момент деплоя не сломались; ветку с `pageData` убираем следующим change'ом.

`CreateRunDto` расширяется до `{ scenarioId: string; inputs?: Record<string, unknown> }`.

*Почему не второй ящик рядом с `pageData`.* Два почти одинаковых поля и два почти одинаковых цикла ожидания разъедутся; ожидание ввода — это одна механика, у которой меняется только вопрос.

### 4. Ожидание ввода — один примитив на два случая

`waitForPageInput` обобщается в `waitForUserInput(runId, stepIndex, request)`, где `request` — либо страница шага, либо список дескрипторов добора. Механика та же, что сейчас: статус → `waiting_input`, публикация события, опрос `Run` раз в секунду до `PAGE_INPUT_TIMEOUT_MS` (30 мин), отмена запуска прерывает ожидание, таймаут → доменная ошибка.

Событие добора — **новое**, `input:required` с payload `{ stepIndex, stepTitle, fields: ManualInputDescriptor[] }`, а не перегруженное `page:required` (его payload — `StepPage`, и клиент рисует его как страницу). Сабмит — `POST /api/runs/:id/input-submit` с `{ stepIndex, values }`; пишет `pendingInput`, статус → `running`, и **мержит values в `Run.inputs`** по скоуп-ключам.

### 5. Что именно проверяется перед шагом и когда спрашивается

Порядок сборки ввода шага с путём `P`:

```
userInput = срез Run.inputs по префиксу P
          ← данные страницы шага (если есть)   // ключи переведены в локальные по target, страница перекрывает
```

Перед исполнением шага воркер берёт дескрипторы этого шага и смотрит, каких **обязательных** значений нет в `userInput`. Если таких нет — шаг идёт как обычно. Если есть — `input:required` и ожидание. Необязательные не запрашиваются никогда: пустое поле в форме — это осознанный выбор.

Добор — страховка, а не основной путь. Он срабатывает, когда: запуск создан программно без `inputs`; сценарий отредактировали между созданием и исполнением запуска; значение относится к вложенному сценарию, который перечислитель не смог развернуть.

### 6. Валидация на создании запуска: 400, а не молчаливое ожидание

`POST /api/runs` перечисляет ручные значения сценария и, если среди `inputs` нет обязательного, отвечает `400` со списком недостающих ключей. Фронт до этого не доводит (кнопка заблокирована), но программный вызов получает внятную ошибку сразу, а не висит 30 минут в `waiting_input`.

*Почему не «всегда спрашивать по ходу».* Это ровно тот сценарий, где отсутствие обратной связи и есть текущая боль: тихое ожидание вместо ошибки — та же болезнь, что тихая пустая подстановка.

### 7. Резолвер: убираем подмену отсутствующего значения

`resolved[field] = userInput?.[field] ?? userInput` → `userInput?.[field]`, а при отсутствии ключа — предупреждение в `warnings` (они уже всплывают в результат шага). Это **BREAKING** для поведения и для теста `test/mapping-resolver.spec.ts` («falls back to entire userInput when field is absent») — тест переписывается на новое поведение.

### 8. Фронт: одна форма, два места показа

Новый `components/run/ManualInputsForm.vue`, принимающий `ManualInputDescriptor[]` и рисующий поля, сгруппированные по шагам (заголовок шага → подписи параметров; для `kind: "filter"` подпись вида «Условие отбора: `field` = …»). Обязательные блокируют submit.

Под ним — новый `ui/Field.vue`, выбирающий контрол по `type` (`number` → числовой инпут, `boolean` → переключатель, остальное → текст) и отдающий **типизированное** значение наверх. Сегодня `ui/Input.vue` умеет только текст, а `required` в страницах чисто декоративный.

Форма используется дважды и в `RunPanel`, и в `PlaygroundPanel`: на экране запуска (по `GET /api/scenarios/:id/manual-inputs`, если список непуст) и по событию `input:required` посреди выполнения. Значения уходят типизированными: приведение к типу поля элемента при сравнении в фильтре резолвер уже делает сам, дублировать коэрцию на бэке не нужно.

### 9. Перечисление вложенных сценариев не должно зацикливаться

У исполнителя (`executeScenarioStep`) защиты от циклов нет: сценарий A со ссылкой на B, ссылающийся на A, повесит воркер уже сегодня. Чинить это здесь — не наша задача, но **перечислитель** обязан быть конечным: обход ведёт множество `refScenarioId` вдоль текущего пути и упирается в предел глубины; ссылку, замыкающую цикл, пропускает с пометкой. Пре-существующая дыра в исполнителе фиксируется в рисках.

### 10. Поле страницы получает явную привязку — иначе «страница закрывает свой шаг» не выполняется

Дедупликация «покрытое страницей не спрашиваем формой» опирается на то, что поле страницы связано со значением шага. Сегодня эта связь — совпадение ключей, а ключ выводится из подписи и не редактируется (см. Context). Значит, совпадение практически недостижимо: у шага со страницей параметр не считался бы покрытым, уезжал бы в общую форму, а страница при исполнении **всё равно показывалась бы** и спрашивала то же самое своими словами. Получился бы ровно тот двойной запрос, который change обязан убрать.

Поэтому у `PageField` появляется явная привязка:

```ts
export interface PageField {
  key: string;
  label: string;
  placeholder?: string;
  required: boolean;
  target?: string;   // локальный ключ значения шага: "inn" | "filter:status"
}
```

- В `PageEditor` у поля — селект «Заполняет», наполняемый ручными значениями **этого** шага. Список собирается из самого шага (`mappings`/`filters`) и уже загруженной редактором схемы шага (проп `schemas`), а **не** запросом на сервер: редактор держит шаг в памяти, поэтому селект видит и несохранённые правки — переключил параметр на «Ручной ввод» и сразу можешь привязать к нему поле. Подпись перестаёт быть носителем ключа и снова становится свободной; `syncFieldKey` остаётся только как генератор технического `key`.
- **Перечислитель**: `source: "page"`, если у страницы шага есть поле с `target === localKey`; при отсутствии `target` — прежнее правило `field.key === paramKey` (обратная совместимость со сценариями, где подпись назвали ключом параметра).
- **Воркер**: данные страницы перед мержем переводятся из ключей полей в локальные ключи значений по `target` (`{ инн_организации: "77…" }` → `{ inn: "77…" }`); поле без `target` кладётся по своему `key`, как сейчас.

Это же бесплатно делает достижимым `filter:{paramKey}`: селект предлагает и операнды условий фильтрации, чего слагифицированный ключ выразить не мог.

*Почему не отдельной фичой.* Селект нечем наполнить до этого change'а — он кормится перечислением, которого сегодня нет; параллелить нечего, только строгий порядок. А без него спека `step-pages` («не спрашивать дважды») оставалась бы невыполненной, и сводку «откуда придёт значение» в `StepConfig` пришлось бы писать дважды.

*Почему селект не ходит за списком на сервер.* `GET /manual-inputs` считается по **сохранённому** сценарию (как и `step-schema/:index`), и селект, открытый до сохранения только что переключённого на «Ручной ввод» параметра, значения бы не показал. Редактор же держит шаг в памяти — там список честнее. Серверное перечисление остаётся единственным источником правды для пути **запуска** (форма + воркер), где расхождение и опасно.

*Публичный маршрут для формы запуска.* Панель запуска видна гостю (карточка маркетплейса публична и уже отдаёт `steps`), а `/api/scenarios/*` — под JWT. Поэтому форма берёт список из `GET /api/marketplace/:id/manual-inputs` — тот же перечислитель, проверка на `published`, никакого нового раскрытия: ручные значения выводятся из тех же шагов, что маркетплейс отдаёт и так.

### 11. Сторож тишины вместо поллинга — и вопрос общей шины при масштабировании

Проверка на живом стенде вскрыла отдельную дыру, не связанную с ручным вводом, но бьющую по тому же экрану: **поток событий — единственный источник прогресса**. Локальный дев делит очередь SQS и базу с задеплоенным бэкендом, тот перехватывал сообщение, исполнял сценарий и публиковал события в **свой** gateway. Локальный клиент при этом сидел на живом сокете, получал только снапшот при подключении и висел в «Выполняем сценарий…» вечно — хотя в базе запуск давно `completed`.

Реконнект здесь не помогает: разрыва нет. Соединение здорово и молчит, а «молчит» и «ещё выполняется» для клиента неразличимы. Снапшот же присылается ровно один раз — при `connect`.

**Решение: сторож тишины** (`useRunSocket`, опция `onSilence`). Таймер сбрасывается каждым входящим событием; если событий нет дольше `SILENCE_MS` (8 с), панель **один раз** перечитывает `GET /api/runs/:id` и применяет результат тем же кодом, что и снапшот. При продолжающейся тишине интервал удваивается до 60 с — длинный шаг «Задержка» или периодический опрос не превращаются в череду запросов. В нормальном запуске (события идут потоком) сторож не делает **ни одного** запроса.

*Почему не периодический поллинг.* Он платит запросами постоянно ради редкого сбоя. Сторож платит ровно тогда, когда что-то сломалось, — цена одного запроса вместо бесконечного спиннера.

*Почему сторож молчит в фазе ожидания ввода.* Там тишина штатна (ждём пользователя), а перечитывание ещё и затёрло бы уже набранные в форме значения.

**Вопрос на будущее — backplane.** Сторож лечит симптом, но не причину класса «событие опубликовал не тот процесс, к которому подключён клиент». Сегодня worker и gateway живут в одном процессе, поэтому `emit` в комнату долетает. Требование «параллельное масштабирование воркеров» из спеки этого уже не переживёт: сокет клиента останется в одном процессе, а шаг исполнит другой. Штатное решение — общий backplane (`@socket.io/redis-adapter` либо эквивалент), и поднимать его надо **вместе с разносом воркеров**, а не после. Зафиксировано требованием в спеке `scenario-execution`; отдельной задачей в этот change не входит.

## Risks / Trade-offs

- **Сценарий отредактировали между открытием формы и стартом запуска** → ключи входов разъезжаются со схемой. Митигация: валидация на создании запуска (400 со списком недостающих) плюс добор по ходу — обязательное значение спросят, а не подставят пустоту.
- **Переименование `pageData` ломает запуски в полёте** → воркер читает `pendingInput ?? pageData` в переходный период; страницы, ожидающие ввода в момент деплоя, доживают.
- **Циклические ссылки между сценариями** → перечислитель конечен (глубина + множество посещённых по пути), но **исполнитель по-прежнему повесится** на таком сценарии. Это существующая дыра, не регресс; выносится отдельным issue.
- **Бедные схемы endpoint'ов** (нет `items`, нет `label`, нет `required`) → форма деградирует в текстовое поле с ключом вместо подписи и не обязательное. Приемлемо: сегодня там вообще нет поля.
- **Существующие страницы, где подпись поля назвали ключом параметра** (единственный способ связать их сегодня), продолжают работать по правилу совпадения ключей, но их подписи так и останутся техническими (`orgId`), пока владелец не перепривяжет поле через `target`. Ломать их нельзя; автоматически проставить `target` по совпадению ключа можно, но это молчаливая правка чужих сценариев — не делаем, ограничиваемся тем, что редактор подскажет привязку.
- **Значения шага, не покрытые страницей, спрашиваются формой заранее, а покрытые — страницей по ходу.** Для пользователя это выглядит как «часть спросили до старта, часть посреди». Неизбежно, пока страница — это экран посреди потока; митигация — честная сводка в конструкторе («откуда придёт значение»), чтобы автор сценария видел раскладку и мог увести всё на страницу или всё в форму.
- **Форма перед запуском может стать длинной** на сценарии из десятка шагов с ручным вводом. Группировка по шагам это смягчает; резать список (например, спрашивать только первые N) сознательно не будем — это вернуло бы тихую пустоту.

## Migration Plan

1. `packages/shared` — `ManualInputDescriptor`, `CreateRunDto.inputs`, `PageField.target`, событие `input:required`. Аддитивно.
2. Бэкенд — модуль перечисления, `GET /api/scenarios/:id/manual-inputs`, `Run.inputs` + `Run.pendingInput` (с чтением `pageData` для совместимости), `POST /api/runs` принимает `inputs` и валидирует обязательные, `POST /api/runs/:id/input-submit`, обобщение ожидания ввода, протаскивание `userInput` через шаги, снятие фолбэка в резолвере. Деплоится **до** фронта: старый фронт продолжает работать (не шлёт `inputs`; сценарии без обязательных ручных значений — как раньше; с обязательными — воркер спросит по ходу, чего старый фронт не покажет → отсюда порядок деплоя).
3. Фронт — `ManualInputsForm` + `ui/Field`, экран запуска в `RunPanel`/`PlaygroundPanel`, обработка `input:required`, селект привязки в `PageEditor`, сводка источника в `StepConfig`.

Откат: выкатить фронт назад достаточно, чтобы вернуться к прежнему поведению (форма не шлёт `inputs`, а `inputs` на бэке опционален). Бэкенд-откат безопасен, кроме `pendingInput` — запуски, ждущие ввода на старой сборке, придётся перезапустить.
