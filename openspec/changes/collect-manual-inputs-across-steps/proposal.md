## Why

Режим «Ручной ввод» заявлен в конструкторе, но фактически не работает: единственный источник `userInput` в системе — данные страницы ввода шага (`waitForPageInput`), а форма запуска не собирает вообще ничего. У шага без страницы ввода `mappings[key] = "user"` резолвится в пустоту: обычный параметр молча уезжает пустым, а значение условия фильтрации роняет шаг с «условие не отобрало ни одного элемента». UI при этом обещает то, чего нет («Страница не настроена — значения ручного ввода запросятся общей формой», `StepConfig.vue:738`), а прошлый change по фильтрации массивов явно отложил эту дыру на отдельный change (`specs/scenario-execution/spec.md:145`).

Разведка вскрыла, что дыра глубже описанной: `StepContext` пересоздаётся на каждой итерации цикла шагов **без** `userInput` (`worker.service.ts:249-255`), поэтому введённое на шаге 1 не видно на шаге 2; у `Run` нет поля для входов запуска, а `POST /api/runs` принимает только `{ scenarioId }`; резолвер при отсутствии ключа подставляет в параметр **весь объект входов** (`userInput?.[field] ?? userInput`, `mapping-resolver.ts:211`).

## What Changes

- **Перечисление ручных входов сценария.** Появляется единый серверный расчёт списка значений, помеченных как ручной ввод, по **всем** шагам сценария (включая вложенные сценарии): и сами параметры (`mappings[key] = "user"`), и значения условий фильтрации (`filter.value.mode = "user"`). Каждый пункт списка знает свой шаг, параметр, подпись, тип и обязательность. Список отдаётся фронтенду и используется воркером — один источник правды.
- **Форма запуска собирает всё разом.** Простой режим и playground перед стартом показывают форму со **всеми** ручными значениями сценария, сгруппированными по шагам, с подписью параметра и полем по типу значения. Обязательные поля блокируют кнопку запуска. **BREAKING**: `POST /api/runs` принимает `inputs`, а `Run` хранит их в новом поле — предыдущий контракт «`{ scenarioId }` и ничего больше» расширяется.
- **Входы доезжают до резолвера.** Воркер сеет `userInput` из `Run.inputs` и **протаскивает его через все шаги** (сейчас теряется между шагами), скоупя значения по индексу шага. Вложенный сценарий получает свой срез входов.
- **Добор по ходу выполнения.** Если на шаге обязательного ручного значения всё-таки нет (запуск через API без входов, вложенный сценарий, возобновление старого запуска), воркер не падает и не шлёт пустоту: он переводит `Run` в `waiting_input`, шлёт новое событие `input:required` и ждёт значение — тем же механизмом, что уже работает для страниц ввода. Введённое дописывается в `Run.inputs`, чтобы пережить рестарт воркера.
- **Поле страницы получает явную привязку к значению шага.** Сегодня поле страницы связано с параметром только случайным совпадением ключей: ключ выводится слагификацией подписи (`PageEditor.vue:49-56`) и не редактируется, так что связать «ИНН организации» с параметром `inn` можно, лишь назвав подпись `inn`. У `PageField` появляется `target` — что именно поле заполняет (`inn` или `filter:status`), — и селект в редакторе страниц, наполняемый тем же перечислением ручных входов. Подпись снова становится свободной. Поле без `target` работает по-старому (совпадение ключа).
- **Страница ввода закрывает свой шаг.** Значения шага, покрытые полями его страницы (по `target`), **не** попадают в общую форму — двойного запроса нет. Непокрытые собираются общей формой. Конструктор показывает по шагу, откуда придёт каждое ручное значение: со страницы шага или из общей формы.
- **BREAKING (поведение резолвера).** Убирается фолбэк «нет ключа → подставить весь `userInput`»: отсутствующее значение остаётся `undefined` с предупреждением, а обязательное — запрашивается. Существующий юнит-тест на этот фолбэк переписывается.
- Убирается ложная копирайтинг-обещалка в `StepConfig.vue` и приводится в соответствие требование спеки «форма с входами первого шага», которое никогда не было реализовано.
- **Сторож тишины на клиенте.** Поток WS-событий перестаёт быть единственным источником прогресса: если событий нет дольше порога, панель один раз перечитывает состояние запуска и применяет его как снапшот. Вскрыто проверкой на живом стенде — потерянный поток событий (запуск исполнил другой процесс) оставлял пользователя в бесконечном «Выполняем сценарий…», хотя запуск давно завершён. Заодно в спеке фиксируется, что при разносе воркеров на несколько процессов события обязаны идти через общую шину (socket.io Redis adapter) — иначе клиент, подключённый к одному процессу, не увидит событий другого.

## Capabilities

### New Capabilities
- `manual-input-collection`: перечисление ручных значений сценария (параметры и значения условий фильтрации) по всем шагам, их сбор формой перед запуском, персистентность в запуске, доставка в резолвер маппингов и добор недостающих значений по ходу исполнения.

### Modified Capabilities
- `scenario-execution`: простой режим и playground запрашивают ручные значения всех шагов (а не «входы первого шага»); движок исполнения протаскивает пользовательский ввод через все шаги и разрешает `user`-источники из входов запуска; снимается оговорка в требовании «Резолв массивных выходов шага» о том, что сбор `user`-значений у шагов без страницы вне охвата.
- `step-pages`: поле страницы получает явную привязку к значению шага (параметр или операнд условия фильтрации) вместо совпадения ключей; страница объявляется источником ручных значений **своего** шага, покрытые ею значения исключаются из общей формы, непокрытые — нет.
- `scenario-builder`: конструктор показывает, откуда придёт каждое значение ручного ввода (страница шага или общая форма), вместо текущего ложного обещания об общей форме.

## Impact

**Backend**
- `apps/backend/src/execution/mapping-resolver.ts` — резолв `user`-источника и операнда фильтра, удаление фолбэка на весь `userInput`; `test/mapping-resolver.spec.ts` — переписать затронутый кейс.
- `apps/backend/src/execution/worker.service.ts` — `StepContext.userInput` больше не теряется между шагами; проверка полноты ручных входов перед шагом; ожидание добора (обобщение `waitForPageInput`); прокидывание среза входов во вложенный сценарий.
- `apps/backend/src/execution/execution.service.ts` / `execution.controller.ts` — `POST /api/runs` принимает `inputs`, новый эндпоинт submit'а добора; валидация обязательных входов при создании запуска.
- `apps/backend/src/execution/run.schema.ts` — новое поле входов запуска; обобщение почтового ящика ожидаемого ввода.
- Новый модуль перечисления ручных входов (используется и контроллером, и воркером); новый эндпоинт `GET /api/scenarios/:id/manual-inputs` (рядом с существующим `step-schema/:index`).

**Shared**
- `packages/shared/src/types/index.ts` — тип дескриптора ручного входа, расширение `CreateRunDto`, поле `PageField.target`.
- `packages/shared/src/events/index.ts` — серверное событие `input:required`.

**Frontend**
- `components/RunPanel.vue`, `components/PlaygroundPanel.vue` — форма ручных входов перед запуском (обязательные блокируют старт), обработка `input:required` по ходу выполнения.
- Новый общий компонент рендера полей по типу значения (сейчас `ui/Input.vue` умеет только текст, а `required` в страницах чисто декоративный).
- `components/scenario/StepConfig.vue` — честная сводка «откуда придёт значение», удаление ложной подсказки.
- `components/scenario/PageEditor.vue` — селект привязки поля к значению шага; подпись перестаёт быть носителем ключа.
- `composables/useRunSocket.ts` — подписка на новое событие.

**Совместимость.** Сценарии, где ключи полей страницы совпадают с ключами параметров шага, продолжают работать как раньше. Почтовый ящик ожидаемого ввода (`Run.pageData`) переименовывается и обобщается, но воркер на переходный период читает и старое поле — запуски, ждущие ввода в момент деплоя, доживают. Бэкенд деплоится до фронта: `inputs` опционален, старый фронт продолжает работать.
