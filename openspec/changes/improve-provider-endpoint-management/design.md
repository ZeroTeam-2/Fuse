## Context

Провайдеры (в коде — «apps») импортируют OpenAPI и хранят производные endpoints в `App.endpoints[]` (`apps/backend/src/apps/app.schema.ts`). Сегодня:

- Парсер `openapi-parser.ts` **отбрасывает** `tags` операций; у `EndpointDoc` нет поля группы.
- Endpoints выводятся плоским списком (`components/ui/EndpointRow.vue`) на карточке провайдера (`pages/my/apps/[id]/index.vue`), в превью импорта (`pages/my/apps/new.vue`) и в StepPicker (`components/scenario/StepPicker.vue`). Строка endpoint не усекает длинный summary → overflow на широких описаниях, особенно в `max-w-[720px]` мастере импорта.
- `components/ui/Pagination.vue` рендерит по кнопке на каждую страницу (`Array.from({length: pageCount})`) — бесконечно растягивается.
- У приложения один `App.baseUrl`; понятия окружений нет. Base URL используется на исполнении в `execution/worker.service.ts` через `joinBaseUrl()`.

Стек: Nuxt 3 / Vue 3.5, Tailwind, Pinia, типизированный `$api` (openapi-fetch), типы в `src/types/api.ts` (генерируются из бэкенда), общие типы в `packages/shared`. Бэкенд — NestJS 11 + Mongoose 8.

## Goals / Non-Goals

**Goals:**
- Извлекать первый `tag` операции при импорте и хранить его на endpoint.
- Единый переиспользуемый UI-примитив: список endpoints, сгруппированный по тегам сворачиваемыми блоками, со сквозным поиском. Применяется в карточке провайдера, превью импорта и StepPicker.
- Устойчивая вёрстка строки endpoint (усечение/перенос длинного summary).
- Оконная пагинация с многоточиями в общем `Pagination.vue` — фиксит все места использования.
- Окружения провайдера: Prod по умолчанию (не удаляется), CRUD прочих, **набор переменных** у каждого (пока один элемент — `baseUrl`); исполнение резолвит путь относительно `baseUrl` окружения, выбранного для поставщика в сценарии.
- Вкладка «Окружения» в редакторе сценария: для каждого задействованного поставщика выбирается окружение (по умолчанию Prod); выбор хранится в сценарии.

**Non-Goals:**
- Переменные окружения помимо `baseUrl` (секреты, заголовки, произвольные ключи) — модель набора расширяема, но в этой итерации единственный элемент `baseUrl`; UI не даёт добавлять/удалять произвольные переменные.
- Выбор окружения на уровне отдельного шага или на уровне конкретного запуска — выбор per-поставщик хранится в сценарии, а не на шаге и не на запуске.
- Множественные теги на endpoint / произвольная ручная перегруппировка — берём только первый тег.
- Серверный поиск/пагинация endpoints — списки endpoints фильтруются на клиенте (как сейчас в StepPicker).

## Decisions

### 1. Модель тега endpoint
Добавляем `tag?: string` в `EndpointDoc` (`app.schema.ts`) и в `Endpoint` (`packages/shared`). Парсер берёт `operation.tags?.[0]`; при отсутствии оставляет пустым, а группировка на клиенте относит такие в блок «Прочее».
- _Почему одно поле, а не массив:_ группировка по блокам требует одной принадлежности; массив усложнит UI и модель без пользовательской выгоды на этой итерации.
- _Альтернатива:_ хранить весь массив `tags` и группировать по каждому (endpoint в нескольких блоках) — отклонено как избыточное.

Группировка вычисляется на клиенте (стабильный порядок: порядок первого появления тега; «Прочее» последним). Опционально можно вернуть `spec.tags` (описания групп) — не обязательно.

### 2. Единый компонент группированного списка endpoints
Новый компонент `components/ui/EndpointGroupList.vue` (или `EndpointExplorer.vue`): принимает `endpoints: Endpoint[]`, опционально `selectable`, эмитит `select`. Внутри — `SearchInput` (переиспользуем существующий `components/ui/SearchInput.vue` для единого стиля), группировка по `tag`, сворачиваемые блоки (`<details>`/кастомный disclosure) с заголовком тега и счётчиком, строки через обновлённый `EndpointRow`.
- Поиск фильтрует по `method`, `path`, `summary`, `tag`; блоки с совпадениями авто-раскрываются, счётчики обновляются.
- Переиспользование в трёх местах даёт «унифицированные стили сквозного поиска» из требований бесплатно.
- _Почему общий компонент, а не три реализации:_ требование прямо просит единый стиль поиска/группировки в карточке и StepPicker.

### 3. Фикс вёрстки строки endpoint
В `EndpointRow.vue`: контейнер `min-w-0`, путь и summary в flex с `truncate`/`min-w-0`, summary — `truncate` (одна строка с ellipsis) либо `line-clamp`. Это убирает overflow в узком мастере импорта.

### 4. Оконная пагинация
Переписываем `Pagination.vue`: вычисляем окно вокруг текущей страницы (например, ±2), всегда показываем первую/последнюю, вставляем маркеры-многоточия для разрывов; добавляем кнопки prev/next. Публичный контракт (`pageCount`, `v-model:page`, событие `change`) сохраняем — все вызовы (`pages/index.vue`, `my/apps/index.vue`, `my/apps/[id]/index.vue`, `my/scenarios/index.vue`, `profile/index.vue`, `ui-kit.vue`) работают без изменений.
- _Альтернатива:_ отдельный новый компонент — отклонено; проще починить общий и покрыть все места сразу.

### 5. Окружения: модель и API
Добавляем поддокумент `EnvironmentDoc { id, name, variables: VariableDoc[] }`, где `VariableDoc { key, value }`, и массив `App.environments[]` в `app.schema.ts`; при создании приложения сидим `[{ name: 'Prod', variables: [{ key: 'baseUrl', value: <derived> }] }]`. `App.baseUrl` **сохраняем** как поле для обратной совместимости и как источник Prod `baseUrl`.
- _Почему `variables` набором, а не полем `baseUrl`:_ пользователь просит расширяемый набор переменных, где `baseUrl` — первый элемент. `VariableDoc[]` с зарезервированным ключом `baseUrl` даёт задел без переусложнения; резолвер читает переменную по ключу `baseUrl`. Ключ `baseUrl` неудаляем и валидируется как абсолютный URL; прочие переменные в этой итерации через UI не добавляются.
- _Альтернатива_ `variables: Record<string,string>`: удобнее в JSON, но у Mongoose/Swagger типизация массива объектов проще и явнее для будущих метаданных переменной (тип, секретность). Оставляем `VariableDoc[]`.
- Исполнение (`worker.service.ts`) резолвит `baseUrl` так: окружение, выбранное для поставщика в сценарии (§7) → его переменная `baseUrl`; при отсутствии — Prod; крайний фолбэк — `App.baseUrl`.
- REST на `apps.controller.ts`: `POST /api/apps/:id/environments`, `PATCH /api/apps/:id/environments/:envId`, `DELETE /api/apps/:id/environments/:envId`. Валидация: Prod не удаляется; имя уникально; переменная `baseUrl` — абсолютный `http(s)` URL с хостом (переиспользуем валидацию из `base-url.ts`) и не удаляется.
- _Почему поддокумент внутри App, а не отдельная коллекция:_ окружения принадлежат приложению, их мало, читаются всегда вместе; отдельная коллекция избыточна.
- Регенерация `src/types/api.ts` (`gen:types`) после изменения Swagger DTO.

### 6. Выбор окружений в сценарии
Сценарий хранит соответствие «поставщик → окружение»: `environmentSelections: { appId: string; environmentId: string }[]` в схеме сценария (`apps/backend/src/scenarios/`) и в shared `Scenario`. Вкладка «Окружения» в редакторе (`pages/my/scenarios/[id]/edit.vue` + `stores/scenario-editor.ts`):
- Список поставщиков вычисляется из шагов (distinct `appId` у шагов с endpoint) — новый геттер в `scenario-editor.ts`.
- Для каждого поставщика `<Select>` его окружений (грузятся вместе с app), дефолт — Prod.
- Выбор сохраняется как часть сценария; при отсутствии записи для поставщика исполнение берёт Prod.
- Исполнение (`worker.service.ts`) для шага находит `environmentSelections` по `appId` шага → `environmentId` → `baseUrl` из этого окружения; иначе Prod / `App.baseUrl`.
- _Почему в сценарии, а не на шаге:_ требование — выбор per-поставщик на отдельной вкладке; одна запись на поставщика проще UI и модели, чем per-шаг.

### 7. Миграция окружений
Бэкфилл по образцу `scripts/backfill-app-base-url.mts`: для приложений без `environments` создать `[{ name:'Prod', variables:[{ key:'baseUrl', value: existing baseUrl }] }]`. Чтение допускает отсутствие массива (фолбэк на `baseUrl`), поэтому миграция не блокирующая. Сценарии без `environmentSelections` трактуются как «все поставщики → Prod» — бэкфилл не требуется.

## Risks / Trade-offs

- [Endpoints без тегов у многих спек] → блок «Прочее» вбирает всё; порядок стабилен, поиск работает. Приемлемо; UX не хуже текущего плоского списка.
- [Изменение публичного контракта `Endpoint`/`App` в shared и `api.ts`] → нужно перегенерировать типы и проверить все потребители; поля добавляемые/опциональные — ломающих удалений нет.
- [Регрессия пагинации в 6 местах] → контракт props/emits неизменен + визуальная проверка на страницах с многими и немногими страницами (ui-kit демо).
- [Исполнение читает не тот Base URL] → строгий фолбэк: выбор в сценарии → Prod → `App.baseUrl`; покрыть сценарий обратной совместимости; проверить `joinBaseUrl`.
- [«Висячий» выбор окружения после удаления окружения у app] → при резолве по `environmentId` ненайденное окружение фолбэчит на Prod; UI вкладки «Окружения» показывает только существующие окружения.
- [Overflow не устранён в др. вкладках] → `EndpointRow` — единая точка правки; используется везде, где выводятся endpoints.

## Migration Plan

1. Shared-типы (`Endpoint.tag`, `Environment`, `App.environments`) → бэкенд-схема и DTO → `gen:types`.
2. Парсер: извлечение первого тега.
3. Backfill-скрипт окружений Prod (идемпотентный, безопасный при повторе).
4. Фронтенд: `EndpointRow` truncate → `EndpointGroupList` → внедрение в карточку/превью/StepPicker; фикс `Pagination`.
5. UI управления Environments на карточке провайдера + подключение REST.
6. Rollback: код читает окружения с фолбэком на `baseUrl`, endpoints — с фолбэком «Прочее»; откат фронтенда не ломает данные (новые поля просто игнорируются).

## Open Questions

- Идентификатор окружения в `environmentSelections`: `environmentId` (стабильный) vs имя окружения. Дизайн берёт `environmentId`; удаление окружения у app требует очистки «висячих» выборов в сценариях — обрабатываем фолбэком на Prod при ненайденном id.
- Строка endpoint: усечение в одну строку (`truncate`) или `line-clamp-2`? (По умолчанию — `truncate`, уточнить при визуальной проверке.)
