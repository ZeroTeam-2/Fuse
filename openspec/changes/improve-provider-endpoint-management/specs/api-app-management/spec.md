## MODIFIED Requirements

### Requirement: Создание приложения с импортом OpenAPI
Система SHALL позволять создать «приложение» (подключённый API-провайдер): название, краткое описание и ссылку на OpenAPI-спецификацию (JSON) либо загружаемый файл спецификации. При импорте система MUST автоматически разобрать спецификацию и извлечь список endpoints (метод, путь, summary), хост API и версию.

При извлечении endpoints система MUST сохранять **тег** операции для смысловой группировки: если у операции задан массив `tags`, берётся первый тег; endpoints без тегов относятся к группе «Прочее». Теги MUST быть доступны фронтенду для группировки endpoints по сворачиваемым блокам.

При импорте система MUST вывести и сохранить **базовый URL приложения** (`baseUrl`) — абсолютный адрес со схемой, хостом и, при наличии, базовым путём. Правило вывода: значение `servers[0].url` из спецификации, разрешённое относительно URL самой спецификации (это корректно обрабатывает как абсолютные, так и относительные значения `servers`); если поле `servers` отсутствует — origin URL спецификации. Отображаемый `host` MUST быть производным от `baseUrl`. Базовый URL MUST быть достаточным для исполнения шагов сценария: путь endpoint'а, присоединённый к `baseUrl`, MUST давать вызываемый абсолютный URL. Выведенный `baseUrl` MUST становиться Base URL окружения Prod приложения (см. capability `api-app-environments`).

При разборе схемы ответа система MUST сохранять **признак массивного ответа** endpoint'а: если тело успешного ответа — коллекция, схема выходов MUST помечаться как массив, а извлечённые поля MUST описывать **элемент** коллекции. Признак MUST NOT теряться при разворачивании `items` — иначе схема шага обещает объект там, где API возвращает список. Для полей типа `array` внутри объекта система MUST сохранять схему элемента (поля элемента), если она задана в спецификации. Эти данные MUST быть доступны конструктору сценариев через схему шага — на них строится условие фильтрации массива.

Система MUST NOT сохранять полный развёрнутый снапшот спецификации инлайном в документе приложения. Сохраняются только производные данные, необходимые для работы (endpoints, схемы входов/выходов, теги, `baseUrl`, `host`, версия) — чтобы документ приложения гарантированно помещался в потолок документа хранилища независимо от размера исходной спеки.

При импорте из загружаемого файла максимальный размер файла MUST браться из конфигурации и MUST быть ограничен так, чтобы результат импорта помещался в хранилище; загрузка файла сверх лимита MUST отклоняться понятной ошибкой о превышении размера.

То же правило MUST применяться при повторном импорте спецификации.

#### Scenario: Успешный импорт спецификации
- **WHEN** пользователь указывает валидный URL OpenAPI JSON и нажимает «Импортировать»
- **THEN** система показывает «Спецификация разобрана — найдено N endpoints» и список endpoints с методами и summary

#### Scenario: Endpoints группируются по тегам в превью
- **WHEN** спецификация содержит операции с тегами (например, `Users`, `Orders`)
- **THEN** превью импорта показывает endpoints сворачиваемыми блоками по тегам, а операции без тегов — в блоке «Прочее»

#### Scenario: Спецификация без поля servers
- **WHEN** импортируется спецификация, в которой отсутствует `servers` (типично для FastAPI, где документ отдаётся по `https://host/openapi.json`)
- **THEN** система сохраняет `baseUrl` равным origin URL спецификации (`https://host`), и шаги сценария по этому приложению исполняются успешно

#### Scenario: Базовый путь из servers сохраняется
- **WHEN** спецификация содержит `servers[0].url` вида `https://api.example.com/v1`
- **THEN** система сохраняет `baseUrl` со всем базовым путём (`https://api.example.com/v1`), и вызовы шагов идут по `https://api.example.com/v1/<path>`

#### Scenario: Невалидная спецификация
- **WHEN** URL недоступен или документ не является валидным OpenAPI
- **THEN** система показывает ошибку импорта и не даёт создать приложение

#### Scenario: Создание после импорта
- **WHEN** спецификация разобрана и пользователь нажимает «Создать приложение»
- **THEN** система сохраняет приложение вместе с endpoints, их тегами и схемами входов/выходов (из параметров и примеров ответов), не сохраняя при этом полный развёрнутый снапшот спецификации, и создаёт окружение Prod с Base URL, равным `baseUrl`

#### Scenario: Импорт из файла в пределах лимита
- **WHEN** пользователь загружает файл спецификации в пределах настроенного лимита размера
- **THEN** система разбирает спеку и сохраняет приложение без ошибки сериализации хранилища

#### Scenario: Импорт из файла сверх лимита
- **WHEN** пользователь загружает файл спецификации больше настроенного лимита
- **THEN** система отклоняет загрузку с понятной ошибкой о превышении размера, а не падает с внутренней ошибкой

#### Scenario: Endpoint возвращает коллекцию
- **WHEN** успешный ответ endpoint'а описан как массив объектов (`type: array`, `items` с полями `id`, `name`)
- **THEN** система сохраняет выходы как поля элемента (`id`, `name`) вместе с признаком, что ответ — массив, и конструктор сценариев предлагает для такого источника условие фильтрации

#### Scenario: Поле-список внутри объекта
- **WHEN** ответ endpoint'а — объект с полем `items`, описанным как массив объектов
- **THEN** система сохраняет поле `items` с типом `array` и схемой его элемента, а не как тип без содержимого

### Requirement: Список и карточка приложения
Раздел «Мои API» SHALL показывать список приложений пользователя с пагинацией (имя, описание, хост, версия, число endpoints, число использующих сценариев). Карточка приложения SHALL показывать импортированные endpoints, дату последней синхронизации и число сценариев, использующих этот API.

Endpoints на карточке приложения MUST выводиться **сворачиваемыми блоками по тегам** (endpoints без тегов — в блоке «Прочее») и MUST сопровождаться **сквозным поиском** по endpoints — по методу, пути, summary и имени блока/тега. При активном поиске отображаются только совпавшие endpoints; блоки с совпадениями раскрываются. Вёрстка строки endpoint MUST устойчиво вести себя при длинном summary — описание усекается/переносится и MUST NOT выходить за пределы контейнера.

#### Scenario: Просмотр карточки приложения
- **WHEN** пользователь открывает приложение из списка
- **THEN** система показывает его метаданные, endpoints сворачиваемыми блоками по тегам и счётчик «сценариев используют этот API»

#### Scenario: Сквозной поиск по endpoints провайдера
- **WHEN** пользователь вводит запрос в поиск endpoints на карточке приложения
- **THEN** отображаются только endpoints, совпавшие по методу, пути, summary или имени блока, а блоки с совпадениями раскрываются

#### Scenario: Длинное описание endpoint не ломает вёрстку
- **WHEN** у endpoint'а очень длинный summary
- **THEN** строка endpoint остаётся в пределах контейнера (описание усекается или переносится), а метод и путь остаются читаемыми
