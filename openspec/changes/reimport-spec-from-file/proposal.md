# Реимпорт спецификации из файла

## Why

Страница «Проверить обновления» приложения работает только по сохранённому OpenAPI URL; для приложений, созданных загрузкой файла, вместо формы стоит заглушка «повторный импорт файлом скоро» — обновить спеку такого приложения невозможно вовсе. Создание при этом давно умеет оба источника (URL | Файл): реимпорт должен уметь то же самое.

## What Changes

- Страница обновления приложения получает переключатель источника **«URL | Файл»**, как на создании: URL-ветка работает как сейчас; файловая — dropzone (те же ограничения `.json/.yaml/.yml` и лимит размера) плюс необязательное поле «Базовый URL API».
- Бэкенд получает файловые ручки реимпорта: `POST /apps/:id/reimport-file` (диф) и `POST /apps/:id/reimport-file/apply` (слияние endpoints) — файл передаётся в обе, как в паре import-preview-file / from-file.
- Если из файла не выводится базовый URL (нет абсолютного `servers` и переопределение не задано), приложение сохраняет свой текущий `baseUrl` — реимпорт не должен его терять.
- Заглушка удаляется; выбор источника доступен любому приложению (у приложения с URL можно обновиться файлом и наоборот).

## Capabilities

### New Capabilities

_нет._

### Modified Capabilities

- `api-app-management`: требование повторного импорта расширяется файловым источником (сейчас спека описывает реимпорт только «тем же правилом», что импорт по URL).

## Impact

- **Backend**: `apps.service.ts` (выделение общих `diff`/`merge`, файловые варианты `reimport`/`applyReimport`), `apps.controller.ts` (две multipart-ручки с той же валидацией файла и лимитом Multer), unit-тесты.
- **Frontend**: `pages/my/apps/[id]/update.vue` (переключатель, dropzone, отправка файла в диф и в apply), типы API.
