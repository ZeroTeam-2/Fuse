## MODIFIED Requirements

### Requirement: Real-time прогресс через WebSocket
Система SHALL предоставлять real-time обновления прогресса выполнения сценария через WebSocket-соединение по `runId`. Worker публикует события (`step:start`, `step:done`, `page:required`, `run:done`, `run:error`, `run:status`) в комнату запуска; клиент, подписанный на `runId`, получает их немедленно.

При подключении клиента к `runId` сервер MUST немедленно отправить этому клиенту снапшот текущего состояния `Run` (статус, `currentStep`, накопленные `stepResults`, текст ошибки при наличии) — до и независимо от последующих инкрементальных событий. Клиент, подключившийся после начала исполнения (в том числе после завершения запуска, после реконнекта или перезагрузки страницы), MUST получить актуальное состояние и MUST NOT остаться без обратной связи из-за событий, отправленных до его входа в комнату.

WebSocket-соединение SHALL аутентифицироваться при handshake: сервер MUST извлечь JWT access-токен из `handshake.auth.token` или из `handshake.headers.cookie` (`access_token` cookie), верифицировать его и получить `userId`. После получения `runId` из `handshake.query.runId` сервер MUST загрузить документ `Run` и проверить, что `run.userId === decoded.userId`; при несовпадении или при отсутствии/невалидном токене сервер MUST отклонить подключение (socket disconnect с причиной `unauthorized` или `forbidden`). Снапшот состояния MUST содержать только sanitized данные (без upstream error bodies длиннее 200 символов, без внутренних `consts` шагов).

#### Scenario: Подключение после старта исполнения владельцем
- **WHEN** worker начал (и, возможно, уже завершил) исполнение до того, как клиент-владелец подключился к `runId`
- **THEN** клиент при подключении получает снапшот с текущим статусом и результатами шагов и отображает актуальное состояние вместо бесконечного «Выполняем сценарий…»

#### Scenario: Быстрый запуск завершился до подключения
- **WHEN** запуск падает или завершается за миллисекунды — раньше, чем браузер успел открыть WebSocket
- **THEN** клиент при подключении получает снапшот со статусом `failed` (с текстом ошибки) либо `completed` (с результатами) и показывает соответствующий экран

#### Scenario: Reconnect после обрыва WebSocket
- **WHEN** WebSocket-соединение обрывается во время выполнения
- **THEN** клиент переподключается по `runId`, повторно аутентифицируется (с тем же JWT), получает снапшот текущего статуса и результаты выполненных шагов без потери прогресса

#### Scenario: Подключение без JWT
- **WHEN** клиент подключается к `runs` namespace без `access_token` cookie и без `handshake.auth.token`
- **THEN** сервер немедленно отключает сокет с причиной `unauthorized` и не отправляет снапшот

#### Scenario: Подключение к чужому запуску
- **WHEN** аутентифицированный пользователь подключается к `runId`, где `run.userId !== decoded.userId`
- **THEN** сервер отключает сокет с причиной `forbidden` и не раскрывает данные запуска

### Requirement: Движок исполнения шагов
Worker SHALL исполнять шаги сценария последовательно: для каждого шага разрешать входы по маппингу (выходы предыдущих шагов, ввод пользователя, константы, включая подстановку `{{token}}`-шаблонов), выполнять HTTP-вызов endpoint стороннего API и сохранять ответ как выходы шага в документ `Run`.

Для шагов, привязанных к приложению (`api`, `periodic`, `file`), worker MUST разрешать **абсолютный URL** вызова: загружать `App` по `appId` шага и склеивать `App.baseUrl` с путём шага (`path` / `pollPath` / `uploadPath`). Относительный путь шага MUST NOT передаваться в HTTP-клиент напрямую. Если приложение не найдено либо у него не задан `baseUrl`, шаг MUST завершаться доменной ошибкой с диагностикой, указывающей на приложение и на необходимость переимпорта спецификации. Резолвнутый URL MUST проходить проверку SSRF перед вызовом — с DNS-pinning (однократный резолвинг, проверка IP через `ipaddr.js` RFC 6890, блокировка decimal/octal/hex форм и метаданных endpoint'ов) и блокировкой внешних редиректов на приватные диапазоны.

Шаг «Задержка» MUST приостанавливать поток на заданное время; «Периодический запрос» MUST опрашивать endpoint с интервалом до признака завершения, публикуя прогресс; «Другой сценарий» MUST исполнять вложенный сценарий как один шаг. Ошибка шага MUST переводить `Run` в статус `failed` с диагностикой (шаг, endpoint, статус ответа). Upstream error body MUST обрезаться до 200 символов перед записью в `stepResults[].error` и `run.error`; полный текст сохраняется только в server-side логах с correlation ID.

Запись результатов шагов MUST быть идемпотентной: повторная обработка одного и того же `runId` (повторная доставка сообщения очереди) MUST NOT создавать дублирующиеся записи `stepResults` с одинаковым `stepIndex` и MUST NOT переводить `Run` из терминального статуса (`completed`, `failed`, `cancelled`) обратно в незавершённый. Worker SHALL валидировать формат `runId` (MongoDB ObjectId `/^[a-f\d]{24}$/`) перед передачей в обработку; невалидные SQS-сообщения MUST перемещаться в DLQ без retry.

#### Scenario: Резолв базового URL приложения
- **WHEN** worker исполняет `api`-шаг с `path: "/collections"` у приложения с `baseUrl: "https://dadata-fake.cloud.astral-dev.ru"`
- **THEN** worker вызывает `https://dadata-fake.cloud.astral-dev.ru/collections` и сохраняет тело ответа как результат шага

#### Scenario: Base path приложения сохраняется
- **WHEN** у приложения `baseUrl: "https://api.example.com/v1"`, а путь шага — `/collections`
- **THEN** worker вызывает `https://api.example.com/v1/collections`

#### Scenario: У приложения не задан базовый URL
- **WHEN** worker исполняет шаг приложения, у которого `baseUrl` пуст
- **THEN** шаг завершается ошибкой с сообщением, называющим приложение и предлагающим переимпортировать спецификацию, а `Run` переходит в `failed` — без низкоуровневой ошибки разбора URL

#### Scenario: SSRF block на metadata endpoint
- **WHEN** атакующий настраивает приложение с `baseUrl: "http://169.254.169.254"` (AWS metadata)
- **THEN** worker отклоняет вызов с ошибкой `URL resolves to a blocked private range`, шаг завершается как `failed`, и реальный metadata-ответ не возвращается

#### Scenario: DNS rebinding отклонён
- **WHEN** атакующий DNS возвращает публичный IP при SSRF-чеке, но перенаправляет соединение на `127.0.0.1` в момент фактического fetch
- **THEN** undici Agent с pinned IP отклоняет соединение, и шаг завершается как `failed`

#### Scenario: Подстановка шаблона в константе
- **WHEN** константа шага содержит `Bearer {{access_token}}` и предыдущий шаг вернул `access_token`
- **THEN** worker подставляет значение токена в заголовок Authorization перед HTTP-вызовом

#### Scenario: Периодический опрос до завершения
- **WHEN** шаг «Периодический запрос» опрашивает endpoint каждые 5 сек до признака завершения в ответе
- **THEN** worker публикует прогресс (значение поля прогресса) на каждой итерации и завершает шаг при получении признака завершения

#### Scenario: Повторная доставка сообщения не дублирует результаты
- **WHEN** сообщение с одним и тем же `runId` доставляется worker'у повторно
- **THEN** результаты шагов перезаписываются по индексу, дубликаты `stepResults` не появляются, а уже терминальный `Run` не возвращается в `running` или `waiting_input`

#### Scenario: Доменная ошибка шага не уходит на повторную доставку
- **WHEN** шаг падает детерминированно (недоступный endpoint, ошибка стороннего API)
- **THEN** worker фиксирует `failed` в `Run`, публикует ошибку и подтверждает сообщение очереди — повторные доставки того же детерминированного сбоя не выполняются

#### Scenario: Невалидный runId в SQS-сообщении
- **WHEN** worker получает SQS-сообщение с `{ runId: "not-an-objectid" }`
- **THEN** сообщение отклоняется без повторной доставки, попадает в DLQ, в лог пишется warning с correlation ID

## ADDED Requirements

### Requirement: Проверка владельца на state-changing эндпоинтах выполнения
Все state-changing эндпоинты управления запуском (`POST /api/runs/:id/cancel`, `POST /api/runs/:id/page-submit`, `POST /api/runs/:id/input-submit`) SHALL принимать `userId` из JWT-пayload и проверять, что `run.userId === userId`. При несовпадении система MUST отвечать `403 Forbidden` без раскрытия существования запуска.

#### Scenario: Владелец отменяет свой запуск
- **WHEN** аутентифицированный пользователь отправляет `POST /api/runs/<own-run-id>/cancel`
- **THEN** система проверяет ownership, отменяет запуск и возвращает обновлённый документ `Run`

#### Scenario: IDOR-попытка отменить чужой запуск
- **WHEN** аутентифицированный пользователь отправляет `POST /api/runs/<other-user-run-id>/cancel`
- **THEN** система отвечает `403 Forbidden` и не меняет статус запуска

#### Scenario: IDOR-попытка инъекции данных в чужой запуск
- **WHEN** аутентифицированный пользователь отправляет `POST /api/runs/<other-user-run-id>/page-submit` с произвольным payload
- **THEN** система отвечает `403 Forbidden` и не модифицирует `stepResults` или `inputs` чужого запуска
