## MODIFIED Requirements

### Requirement: Карточка сценария
Система SHALL позволять создать сценарий (use-case карточку) с названием, обложкой (загрузка/drag-and-drop изображения), rich-text описанием для маркетплейса (жирный, курсив, подчёркивание, заголовки, код) и категорией из справочника с поиском. Tagline карточки MUST выводиться из первых ~90 символов текстового описания. Сценарий — самостоятельная сущность и MAY собирать шаги из разных API.

Поле `steps` в Mongoose-схеме сценария SHALL быть типизировано через явный subschema (или валидироваться zod-схемой перед записью), а не `Mixed`. Структура каждого шага (включая `appId`, `endpointId`, `mappings`, `consts`, `filter`, `page`) MUST валидироваться перед `$set`. Размер шага и общее число шагов MUST быть ограничены (например, не более 100 шагов и не более 100 КБ JSON на шаг), чтобы предотвратить document-growth DoS.

#### Scenario: Создание сценария
- **WHEN** пользователь заполняет детали и нажимает «Создать сценарий»
- **THEN** система создаёт неопубликованный сценарий без шагов и открывает редактор шагов

#### Scenario: Редактирование метаданных
- **WHEN** пользователь меняет обложку, описание или категорию на вкладке «Основная»
- **THEN** изменения сохраняются и отражаются в карточке маркетплейса после публикации

#### Scenario: Слишком большой шаг отклонён
- **WHEN** пользователь сохраняет шаг, сериализованный JSON которого превышает 100 КБ (например, огромный объект `consts`)
- **THEN** система отвечает `400 Bad Request` с сообщением `Step payload exceeds size limit` и не сохраняет шаг

### Requirement: Публикация сценария
Система SHALL позволять публиковать и скрывать сценарий через выделенный эндпоинт `POST /api/scenarios/:id/publish` (toggle). Публикация MUST требовать хотя бы один шаг и MUST блокироваться, если сценарий или один из его шагов ссылается на заблокированное приложение. Опубликованный сценарий появляется в маркетплейсе, скрытый — виден только владельцу.

Поле `published` MUST NOT редактироваться через generic `PATCH /api/scenarios/:id`; `UpdateScenarioDto` SHALL исключать поле `published`. Все изменения статуса публикации MUST проходить через `togglePublish` с валидацией (проверка наличия шагов, проверка блокировок приложений).

#### Scenario: Публикация без шагов
- **WHEN** пользователь пытается опубликовать сценарий без шагов через `POST /api/scenarios/:id/publish`
- **THEN** система блокирует публикацию с подсказкой «Добавьте хотя бы один шаг»

#### Scenario: Публикация сценария с заблокированным приложением
- **WHEN** пользователь пытается опубликовать сценарий, в котором хотя бы один шаг ссылается на приложение, помеченное как `blocked`
- **THEN** система блокирует публикацию с указанием имени заблокированного приложения

#### Scenario: Байпас публикации через PATCH заблокирован
- **WHEN** пользователь отправляет `PATCH /api/scenarios/:id` с телом `{ "published": true }`
- **THEN** система отвечает `400 Bad Request` с сообщением `published field is not allowed in PATCH; use POST /api/scenarios/:id/publish`, либо молча игнорирует поле (в зависимости от конфигурации `forbidNonWhitelisted`)

## ADDED Requirements

### Requirement: Проверка владельца на manual-inputs эндпоинте
Эндпоинт `GET /api/scenarios/:id/manual-inputs` SHALL принимать `userId` из JWT и возвращать дескрипторы ручных вводов только если `scenario.ownerId === userId` ИЛИ `scenario.published === true`. При невыполнении любого из условий система MUST отвечать `403 Forbidden` (или `404 Not Found`, чтобы не раскрывать существование сценария).

#### Scenario: Владелец запрашивает manual-inputs своего черновика
- **WHEN** владелец отправляет `GET /api/scenarios/<own-draft-id>/manual-inputs`
- **THEN** система возвращает полный список дескрипторов вводов (включая `paramKey`, типы, обязательность)

#### Scenario: Гость запрашивает manual-inputs опубликованного сценария
- **WHEN** неаутентифицированный пользователь открывает карточку опубликованного сценария и frontend запрашивает manual-inputs
- **THEN** система возвращает дескрипторы (для отображения превью формы запуска)

#### Scenario: IDOR на чужой черновик отклонён
- **WHEN** аутентифицированный пользователь отправляет `GET /api/scenarios/<other-user-draft-id>/manual-inputs`
- **THEN** система отвечает `404 Not Found` и не раскрывает структуру чужого черновика
