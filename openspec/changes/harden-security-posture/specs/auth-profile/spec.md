## MODIFIED Requirements

### Requirement: Вход через Яндекс (Yandex OAuth)
Система SHALL предоставлять вход через Yandex OAuth (Yandex ID) без отдельной регистрации по email. При первом входе система автоматически создаёт пользователя на основе профиля Яндекса (ФИО, email, аватар).

Вход SHALL быть модальным окном, доступным с любой страницы, а не отдельным маршрутом: пользователь не теряет контекст, в котором ему потребовалась авторизация. Модалка MUST открываться по кнопке «Войти» в шапке, при попытке запустить сценарий и при заходе гостя в приватный раздел. Модалка MUST показывать причину, по которой потребовался вход, и фирменную кнопку Яндекс ID.

Гость (неаутентифицированный пользователь) SHALL иметь доступ к публичному маркетплейсу и к страницам карточек сценариев, включая просмотр шагов сценария. Запуск сценария и приватные разделы (мои API, мои сценарии, профиль) MUST требовать аутентификации. Пункты меню приватных разделов MUST NOT показываться гостю.

Система SHALL реализовывать защиту от login CSRF через OAuth `state`-параметр: при старте входа система генерирует случайный `state`, сохраняет его в короткоживущей (≤ 10 минут) httpOnly cookie `oauth_state` и передаёт в URL авторизации Яндекса. При callback система MUST сверить `state` из query со значением cookie; при несовпадении или отсутствии система отклоняет callback и не выдаёт сессию. После успешной сверки cookie `oauth_state` MUST удаляться.

Auth-куки (`access_token`, `refresh_token`) SHALL устанавливаться с атрибутами `HttpOnly`, `Secure` (в production), `SameSite=Lax` (во всех окружениях), `Path=/api`. Система SHALL ротировать refresh-токены: каждый refresh-токен имеет уникальный `jti`, при обновлении старый `jti` инвалидируется, а новый access/refresh выдаётся. Access-токен содержит поле `sId` (session id), валидируемое при каждом запросе. Logout инвалидирует все сессии пользователя (или текущую по выбору). При "выйти на всех устройствах" или смене password инкрементируется `tokenVersion`, что инвалидирует все ранее выданные access-токены.

#### Scenario: Успешный вход через Яндекс
- **WHEN** пользователь нажимает «Войти с Яндекс ID» в модалке входа и подтверждает доступ в Яндекс ID
- **THEN** система создаёт сессию (JWT в httpOnly cookie с `SameSite=Lax`, `Path=/api`), при необходимости создаёт пользователя, и показывает данные пользователя (аватар/инициалы в шапке), а приватные разделы становятся видимыми

#### Scenario: Отказ от авторизации
- **WHEN** пользователь отменяет доступ в Яндекс ID
- **THEN** система не создаёт сессию, возвращает пользователя в маркетплейс и показывает в модалке входа сообщение «Вход отменён»

#### Scenario: Недействительный OAuth-код
- **WHEN** callback содержит недействительный или истекший код
- **THEN** система показывает в модалке входа ошибку авторизации и предлагает повторить вход

#### Scenario: Login CSRF отклонён
- **WHEN** атакующий инициирует OAuth flow со своим аккаунтом и пытается заставить жертву перейти по `GET /api/auth/callback?code=ATTACKER_CODE` без корректного `state` cookie
- **THEN** система отвечает `400 Bad Request` с сообщением `Invalid OAuth state`, не создаёт сессию и не устанавливает auth-куки

#### Scenario: Гость пытается запустить сценарий
- **WHEN** неаутентифицированный пользователь открывает карточку сценария и нажимает кнопку запуска
- **THEN** система не создаёт запуск, а открывает модалку входа с пояснением, что вход нужен именно для запуска

#### Scenario: Гость открывает приватный раздел по прямой ссылке
- **WHEN** неаутентифицированный пользователь переходит по ссылке на приватный раздел (мои сценарии, мои API, профиль)
- **THEN** система возвращает его в маркетплейс и открывает модалку входа

#### Scenario: Выход из аккаунта
- **WHEN** пользователь нажимает «Выйти» в меню аватара и подтверждает выход (POST-запрос с CSRF-токеном)
- **THEN** система инвалидирует сессию (отмечает `jti` как revoked в session store), очищает auth-куки с теми же `Path`/`SameSite` атрибутами, что и при установке, и показывает маркетплейс как гостю

#### Scenario: Replay старого refresh-токена
- **WHEN** после ротации атакующий использует украденный старый refresh-токен (с уже инвалидированным `jti`)
- **THEN** система отвечает `401 Unauthorized`, инвалидирует всю цепочку сессий (detected reuse), и пользователь должен заново пройти OAuth flow

### Requirement: Профиль пользователя
Система SHALL хранить и позволять редактировать личные данные: ФИО, аватар (загрузка и удаление; при первом входе подставляется аватар из профиля Яндекса). Email пользователя immutable после создания и MUST NOT редактироваться через `PATCH /api/users/me`. Личные данные MUST быть видны только самому пользователю и владельцам API, чьи сценарии он запускает.

Аватары MUST валидироваться по magic-bytes (через `file-type` библиотеку), а не по клиентскому mimetype. Допустимые форматы: PNG, JPEG, WebP. SVG MUST быть запрещён (потенциальный stored XSS через `<script>`). Размер файла MUST быть ограничен 5 МБ на уровне multer `limits`.

#### Scenario: Изменение личных данных
- **WHEN** пользователь меняет ФИО и нажимает «Сохранить изменения»
- **THEN** система сохраняет данные и подтверждает сохранение; поле email в форме редактирования доступно только для чтения

#### Scenario: Загрузка аватара с валидным PNG
- **WHEN** пользователь загружает PNG-файл размером 2 МБ
- **THEN** система валидирует magic-bytes, сохраняет аватар в MinIO с метаданными `Content-Disposition: attachment` и `X-Content-Type-Options: nosniff`, и отображает его вместо инициалов

#### Scenario: Загрузка SVG отклонена
- **WHEN** пользователь загружает файл `evil.svg` с mimetype `image/svg+xml` или `evil.png` с SVG-содержимым
- **THEN** система отвечает `400 Bad Request` с сообщением «SVG is not allowed»

#### Scenario: Загрузка слишком большого файла
- **WHEN** пользователь загружает PNG размером 10 МБ
- **THEN** система отвечает `413 Payload Too Large` без полной буферизации файла в памяти
