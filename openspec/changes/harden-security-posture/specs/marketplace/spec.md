## MODIFIED Requirements

### Requirement: Страница карточки
Карточка сценария SHALL иметь вкладки: «Обзор» (описание в rich-формате, число запусков, число поставщиков API, число endpoints, категория, кнопка «Перейти к запуску»), «Запуск» и «Сервисы и endpoints» (список поставщиков API и задействованных endpoints с методами и путями).

Страница карточки SHALL быть публичной: гость видит её целиком, включая вкладку «Запуск» с превью шагов сценария. Публичное представление опубликованного сценария MUST включать его шаги в **санитизированном виде**: показывается тип шага, его название и видимое для пользователя описание; внутренние поля `appId`, `endpointId`, `consts` (потенциально содержащие секреты и токены) и текст `blockedReason` MUST быть исключены из публичного ответа. Сам запуск MUST требовать аутентификации (см. capability `auth-profile`).

#### Scenario: Просмотр обзора
- **WHEN** пользователь открывает карточку из каталога
- **THEN** система показывает вкладку «Обзор» с отрендеренным описанием (через parser-based DOMPurify санитайзер) и метриками карточки

#### Scenario: Гость смотрит вкладку «Запуск»
- **WHEN** неаутентифицированный пользователь открывает вкладку «Запуск» карточки
- **THEN** система показывает панель запуска с превью шагов сценария (без внутренних `appId`/`endpointId`/`consts`), но запуск остаётся недоступным до входа

#### Scenario: Публичный ответ не раскрывает константы шагов
- **WHEN** неаутентифицированный клиент запрашивает `GET /api/marketplace/:id`
- **THEN** ответ не содержит полей `consts`, `blockedReason`, и каждый шаг не раскрывает `appId`/`endpointId` (только публичное название провайдера и endpoint)

### Requirement: Поиск
Маркетплейс SHALL поддерживать текстовый поиск по названию карточки и имени провайдера. Поиск MUST обновлять счётчики дерева категорий и показывать пустое состояние «По вашему запросу ничего не найдено».

Поисковый запрос пользователя MUST быть экранирован перед передачей в MongoDB `$regex` (или заменён на текстовый индекс), чтобы предотвратить катастрофическую backtracking (ReDoS). Запросы вида `(a+)+$`, `(?:a|a)+$` и подобные патологически сложные паттерны MUST NOT вызывать зависание worker-а базы данных.

#### Scenario: Пустой результат
- **WHEN** запрос не совпадает ни с одной карточкой
- **THEN** система показывает пустое состояние и нулевые счётчики, сбрасываемые очисткой запроса

#### Scenario: Malicious regex pattern экранирован
- **WHEN** атакующий отправляет `?search=(a%2B)%2B$`
- **THEN** система экранирует спецсимволы, выполняет литеральный поиск по строке `(a+)+$` и возвращает пустой результат за ≤ 100 мс (без DoS)
