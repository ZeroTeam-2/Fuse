## Context

Импорт OpenAPI-спецификаций выполняется в `SsrfGuard.fetchSpec()` (`apps/backend/src/apps/ssrf-guard.ts:98`). Метод скачивает документ по URL, ограничивает размер 10 МБ и разбирает тело — но разбор захардкожен в `JSON.parse(text)` (`ssrf-guard.ts:149-153`). Любой YAML-ответ падает с `BadRequestException("Spec response is not valid JSON")`, хотя `Accept`-заголовок уже заявляет `application/yaml` (`ssrf-guard.ts:112`).

Дальнейший конвейер — `OpenApiParserService.parse()` (`openapi-parser.ts:77`) — работает с уже разобранным объектом `Record<string, unknown>` и к строковому формату чувствительности не имеет: `@readme/openapi-parser` дерезолвит `$ref` и валидирует структуру независимо от исходной кодировки. Таким образом, единственное место, требующее изменений, — преобразование «текст → объект» в `fetchSpec()`.

Прямой YAML-зависимости в backend нет (`apps/backend/package.json`), но `js-yaml@4` уже присутствует транзитивно через `@readme/openapi-parser` → `@apidevtools/json-schema-ref-parser`.

## Goals / Non-Goals

**Goals:**
- Принимать OpenAPI-спецификации в YAML наравне с JSON при импорте (создание, preview, reimport).
- Корректно определять формат ответа, в т.ч. когда сервер отдаёт YAML без явного `Content-Type` (распространённый случай — `text/plain` или `application/octet-stream`).
- Сохранить полную обратную совместимость для существующих JSON-спецификаций.
- Покрыть новый путь тестами.

**Non-Goals:**
- Загрузка спецификации файлом / вставкой текста (ввод остаётся только по URL).
- Изменение `OpenApiParserService`, `base-url.ts`, `apps.service.ts` — они работают с объектом и не затрагиваются.
- Валидация схемы YAML за пределами того, что уже делает `@readme/openapi-parser` при `dereference()`.
- Поддержка форматов, отличных от JSON и YAML (XML, TOML и т.п.).

## Decisions

### 1. Библиотека парсинга: `js-yaml`

Выбран `js-yaml` (v4), а не альтернативный пакет `yaml`.

- Уже присутствует транзитивно в `node_modules` через зависимости `@readme/openapi-parser` — добавление явной зависимости фиксирует версию, но не вводит новый артефакт в дерево.
- Используется тем же `@readme/openapi-parser` для разрешения YAML-编码ированных внешних `$ref`, т.е. семантика разбора согласована с тем, что парсер ожидает.
- Меньшая и более стабильная поверхность API, чем у `yaml`, для нашего единственного сценария (`load(text)` → объект).

Альтернатива `yaml` рассмотрена и отвергнута: не даёт преимуществ для одноразового `parse()` и добавляет второй YAML-движок в дерево, уже содержащее `js-yaml`.

Добавляются: `js-yaml` в `dependencies`, `@types/js-yaml` в `devDependencies` (`apps/backend/package.json`).

### 2. Определение формата: Content-Type + body sniffing

Стратегия разрешения формата в `fetchSpec()` — двухуровневая:

1. **Content-Type ответа** (приоритет): если содержит `json` → JSON; если содержит `yaml` (вкл. `application/yaml`, `application/x-yaml`, `text/yaml`, `text/vnd.yaml`) → YAML.
2. **Body sniffing** (fallback, когда Content-Type неинформативен — `text/plain`, `*/*`, отсутствует): первый непробельный символ тела. `{` или `[` → JSON; иначе → YAML. Это надёжно разделяет JSON-объекты/массивы от YAML-документов (YAML не может начинаться с `{`/`[` как верхнеуровневый блок-документ — flow-mapping технически допустим, но на практике OpenAPI-YAML всегда block-style).

Альтернатива «всегда пробовать JSON, при ошибке — YAML» отвергнута: `JSON.parse` выбрасывает на любом YAML, но его сообщение бесполезно, а двойной парсинг размывает источник ошибок. Явное определение даёт детерминизм и точное сообщение.

### 3. Изоляция изменений в `fetchSpec()`

Вся логика определения и разбора инкапсулируется в `fetchSpec()` (или приватном помощнике `parseSpecBody(text, contentType)` в том же файле). Возвращаемый тип остаётся `Record<string, unknown>`. `OpenApiParserService.parse()` и нижележащие методы не меняются — это сохраняет контракт и гарантирует, что reimport/diff работают идентично для обоих форматов.

### 4. Уточнение сообщения об ошибке

При провале разбора выбрасывается `BadRequestException("Spec response is not valid JSON or YAML")` — отражает оба поддерживаемых формата и не вводит новое кодовое поведение. Гранулярные сообщения (JSON-specific vs YAML-specific) не добавляются: пользователь всё равно видит только текст исключения, а детальное логирование уже не ведётся.

## Risks / Trade-offs

- **[Ложное определение YAML для повреждённого JSON]** → Если повреждённый JSON не начинается с `{`/`[`, sniffing направит его в YAML-парсер, который может выдать строку/примитив вместо объекта. Смягчение: после разбора проверять `typeof result === "object" && result !== null`; иначе — ошибка. Это уже косвенно покрывается `dereference()` дальше, но явная проверка даёт ранний и понятный отказ.
- **[YAML-документ с flow-style верхнего уровня `{...}`]** → Будет определён как JSON и провалит `JSON.parse`. Смягчение: при ошибке JSON-разбора для тела, определённого как JSON по sniffing, делать повторную попытку через YAML (мягкий fallback). Это компромисс между детерминизмом решения 2 и устойчивостью; fallback ограничен одним повтором.
- **[Расхождение версий `js-yaml`]** → Явная зависимость может разойтись с транзитивной. Смягчение: pnpm дедуплицирует при совместимости semver; фиксируем `^4` как уже используемую мажорную версию.
- **[Безопасность YAML]** → `js-yaml.load()` по умолчанию (v4) безопасен — не выполняет JS-теги (в отличие от старого `safeLoad`). Дополнительных мер не требуется.
