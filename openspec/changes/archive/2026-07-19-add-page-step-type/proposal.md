# Страница как отдельный тип шага

## Why

Сегодня страница — это скрытая настройка любого шага (`BaseStep.page`), и показывается она всегда **до** исполнения шага-носителя. Из-за этого невозможно построить финальный экран: показать после последнего шага страницу с текстом ответа. Кроме того, наличие страницы не видно в потоке шагов — система непрозрачна: чтобы понять, где сценарий останавливается и ждёт пользователя, нужно открывать настройки каждого шага.

Страница как самостоятельный тип шага решает обе проблемы: её место в потоке явно видно и свободно выбирается (в том числе последним шагом — экран результата), а данные страницы становятся обычными выходами шага, доступными следующим шагам штатным маппингом «Из шага».

## What Changes

- Новый тип шага **«Страница»** (`type: "page"`) в палитре редактора: шаг несёт раскладку страницы (`StepPage`) как свою основную настройку и открывает тот же полноэкранный конструктор.
- **Семантика данных**: блоки ввода страницы-шага становятся **выходами** этого шага (ключ выхода — привязка блока либо его id); следующие шаги берут эти значения обычным маппингом `s{idx}:{key}`. Блоки отображения привязываются к переменным (выходам) любых пройденных шагов (`s{idx}:{outKey}`).
- **Исполнение**: шаг-страница с блоками ввода приостанавливает запуск (`waiting_input` + `page:required`), собранные значения записываются результатом шага. Страница только с блоками отображения не блокирует поток: она публикуется для показа и остаётся финальным экраном, если стоит последним шагом.
- **BREAKING**: встроенные страницы выпиливаются целиком — поле `BaseStep.page`, легаси-ветка исполнения в worker'е и секция «Страница» в панелях настройки остальных типов шагов удаляются. Обратная совместимость со старыми документами не поддерживается: локальная база сбрасывается (`pnpm reset` / пересев).
- Форма запуска не спрашивает значения, которые собирает страница-шаг (это её собственные выходы, а не `user`-маппинги других шагов).

## Capabilities

### New Capabilities

_нет — изменение перераспределяет существующие возможности._

### Modified Capabilities

- `step-pages`: страница перестаёт быть привязкой к произвольному шагу и становится содержимым шага типа «Страница»; привязка блока ввода теперь задаёт **ключ выхода** шага-страницы (а не входной параметр шага-носителя); блоки отображения привязываются к выходам любых пройденных шагов.
- `scenario-builder`: пятый тип шага «Страница» в палитре; его выходы (блоки ввода) доступны следующим шагам в маппинге «Из шага»; секция «Страница» исчезает из настройки остальных типов.
- `scenario-execution`: исполнение шага `page` (пауза/сабмит для ввода, неблокирующий показ для чисто-отображающих страниц, результат шага = собранные значения); финальная страница как экран результата; удаление легаси-исполнения встроенных страниц.

## Impact

- **Shared** (`packages/shared`): `StepType` + новый `PageStep` в union `Step`; хелперы схемы шага-страницы (выходы из блоков ввода); удаление `BaseStep.page`.
- **Backend**: `worker.service.ts` (ветка исполнения `page`-шага; удаление легаси-ветки `step.page`), `manual-inputs.ts` (страница-шаг не даёт дескрипторов формы запуска; удаление `pageCovers` и старого `mapPageDataToLocalKeys`), `scenarios.service.ts` (схема шага `page` для маппинга следующих шагов), валидация шагов, `seed-local.mts`.
- **Frontend**: `AddStepMenu`/`StepPicker` (новый тип), `StepConfig.vue` (конфиг page-шага = конструктор; удаление секции «Страница» у остальных), `PageEditor.vue` (инспектор привязок блока ввода → ключ выхода), `scenario-editor.ts` (модель шага, выходы page-шага в источниках маппинга), `RunPanel.vue`/`PageRunner.vue` (неблокирующие display-страницы, финальный экран), e2e-тесты.
- **Данные**: без миграции и без легаси — локальная база сбрасывается (`pnpm reset`), сид пересобирает демо-данные уже с page-шагами.
