# Design — страница как отдельный тип шага

## Context

Сейчас страница живёт в `BaseStep.page` (`packages/shared/src/types/index.ts`): любой шаг может нести раскладку, и воркер (`worker.service.ts`, `executeStep`) перед исполнением шага-носителя приостанавливает запуск (`waiting_input`, событие `page:required`), ждёт `page:submit` и вливает собранные значения во входы шага через привязки блоков (`mapPageDataToLocalKeys`). Дескрипторы ручного ввода (`manual-inputs.ts`) помечают покрытые страницей значения `source: "page"`, чтобы их не спрашивала форма запуска.

Следствия модели:
- страница показывается только **до** исполнения шага-носителя — финальный экран с ответом сценария построить нельзя;
- наличие страницы не видно в списке шагов, поток непрозрачен;
- привязки блоков ввода замкнуты на входы одного шага-носителя — значение страницы нельзя переиспользовать в других шагах.

Проект на стадии разработки, продовых данных нет: обратная совместимость со старыми документами не требуется. Встроенные страницы выпиливаются целиком, локальная база сбрасывается (`pnpm reset`) и пересеивается.

## Goals / Non-Goals

**Goals:**
- Тип шага `page`: страница — первоклассный шаг, видимый в потоке, вставляемый в любую позицию (включая последнюю).
- Значения блоков ввода — выходы шага-страницы, доступные следующим шагам штатным маппингом `s{idx}:{key}` (включая `UploadedFileRef` от dropzone → multipart в api-шаге).
- Финальный экран: страница только с блоками отображения, стоящая последним шагом, остаётся видимой после завершения запуска.
- Полное удаление встроенных страниц: поле `BaseStep.page`, легаси-ветка исполнения, `pageCovers` и связанная с ними логика формы запуска.

**Non-Goals:**
- Поддержка и миграция старых документов со встроенными страницами — не нужны: локальная база сбрасывается (`pnpm reset`) и пересеивается.
- Новые типы блоков, условная маршрутизация, несколько страниц на шаг.
- Изменение протокола WebSocket (`page:required`/`page:submit` переиспользуются как есть).

## Decisions

### D1. Модель: `PageStep` с обязательной раскладкой

```ts
export interface PageStep extends BaseStep {
  type: "page";
  page: StepPage;
}
```

`StepType` получает `PAGE: "page"`, `Step`-union — `PageStep`. Поле `BaseStep.page` **удаляется** — страницу несёт только `PageStep`. У `PageStep` нет `mappings`/`filters` по назначению: его входной контекст — результаты пройденных шагов, и блоки отображения резолвятся из них напрямую (как сейчас в `resolvePageBindings`).

*Альтернатива — отдельное поле `layout` вместо переиспользования `page`*: отвергнута, формат `StepPage` не меняется, а совпадение поля позволяет переиспользовать весь код резолва/рендера без ветвления.

### D2. Семантика данных: блоки ввода = выходы шага

Для шага-страницы `binding` блока ввода меняет смысл: это **ключ выхода** шага (латиница, как ключи параметров), а не входной параметр шага-носителя. Блок без привязки отдаёт значение под своим `id` (та же деградация, что сейчас).

Направление данных у страницы двустороннее, но асимметричное: **входной контекст** страницы — результаты пройденных шагов (блоки отображения и `optionsSource` селектов привязываются к переменным любых прошлых шагов, `s{idx}:{outKey}`); **выходы** — значения блоков ввода, которые следующие шаги забирают маппингом.

Схема шага-страницы (`ManualInputsService.loadStepSchema`, `ScenariosService.getStepSchema`, зеркально во фронтенд-сторе):
- `inputs: []` — у страницы нет входов в смысле маппинга;
- `outputs` — по блокам ввода: `key = binding ?? id`, `label = block.label ?? key`, тип: `dropzone → "file"`, остальные → `"string"`;
- `outputIsArray: false`.

Хелпер `pageStepSchema(page: StepPage): StepSchema` кладётся в `packages/shared` рядом с `pageBlocks`/`isInputBlock`, чтобы бэкенд и фронтенд не разъехались.

*Альтернатива — оставить семантику «страница заполняет входы следующего шага»*: отвергнута, это воспроизводит текущую непрозрачную связь и не даёт переиспользовать значения в нескольких шагах.

### D3. Исполнение: блокирующая страница ввода, неблокирующая страница отображения

В `executeStep` появляется ветка `case "page"`:

- **Есть блоки ввода** → как сейчас: `requestUserInput` публикует `page:required` (payload: `page`, `resolved` из `resolvePageBindings`), запуск в `waiting_input`; после `page:submit` данные маппятся `blockId → binding ?? id` и **записываются результатом шага** (объект «ключ выхода → значение»). Обязательность блоков валидирует PageRunner (как сейчас) + воркер проверяет непустоту обязательных ключей.
- **Только блоки отображения** → воркер публикует то же `page:required` с резолвнутыми данными, результатом шага пишет `{}` и продолжает **не дожидаясь** сабмита. Если такой шаг последний — RunPanel оставляет страницу видимой как финальный экран (событие `run:done` не сбрасывает её, сводка доступна рядом/ниже).

Легаси-ветка `if (step.page)` в `executeStep` удаляется вместе с `pageCovers` и старой семантикой `mapPageDataToLocalKeys` (маппинг «blockId → ключ выхода» для page-шага пишется заново рядом с новой веткой).

*Альтернатива — display-only страница тоже ждёт подтверждения («Продолжить»)*: отвергнута для основного кейса «финальный экран с ответом» — запуск завершался бы только после лишнего клика; страницу посреди потока, требующую подтверждения, при необходимости можно собрать, добавив на неё блок ввода.

### D4. Ручной ввод и форма запуска

`enumerateManualInputs` пропускает шаги типа `page` (у них нет `mappings`) — дескрипторы для них не создаются, форма запуска ничего не спрашивает: значения соберёт сама страница по ходу. `pageCovers` и признак `source: "page"` у `ManualInputDescriptor` теряют смысл и удаляются: все оставшиеся дескрипторы спрашивает форма запуска.

### D5. Редактор

- `AddStepMenu`/`StepPicker`: пятый тип «Страница» (без выбора приложения/endpoint).
- `StepConfig` для `page`-шага: название + кнопка «Открыть редактор» (тот же полноэкранный `PageEditor`) + сводка «N элементов»; секция «Страница» у остальных типов удаляется без следа.
- `PageEditor`, инспектор блока ввода: вместо селекта «привязать к параметру шага» — текстовое поле «Ключ выхода» (валидация: латиница/цифры/underscore, уникальность в пределах страницы; пусто → используется `id`). Инспектор блока отображения не меняется по механике: источники — переменные (выходы) **любых** шагов до шага-страницы. Сайд-эффект «привязка назначает `mappings[key] = "user"`» исчезает вместе со старой семантикой.
- Стор `scenario-editor`: источники маппинга «Из шага» для последующих шагов включают выходы page-шага из `pageStepSchema`.

### D6. Валидация и смежные места

- Валидация шагов на сервере: `page`-шаг обязан нести непустую страницу (хотя бы один блок); ключи выходов не дублируются.
- `cycle-guard`, вкладка «Окружения», индикатор мультисервисности: `page`-шаг не имеет `appId` и просто не участвует (по аналогии с `delay`).
- Вложенный сценарий, заканчивающийся page-шагом: `loadStepSchema`/`getStepSchema` для `scenario`-шага рекурсивно берут схему последнего шага — `pageStepSchema` там работает без доработок. Страницы вложенного сценария показываются при его исполнении как один шаг.

## Risks / Trade-offs

- [Старые документы со встроенными страницами перестают исполняться корректно] → приемлемо: стадия разработки, локальная база сбрасывается `pnpm reset` и пересеивается; сид обновляется на page-шаги.
- [Display-only страница посреди потока «мелькает»] → осознанный компромисс (D3); контент останется в стеке страниц RunPanel/истории шагов. Если понадобится пауза — добавить на страницу блок ввода или в будущем флаг «требует подтверждения».
- [Ключ выхода вводится вручную — опечатки ломают маппинг] → валидация в инспекторе + автоподстановка источника по совпадению ключа уже есть в сторе; выходы показываются в UI маппинга следующих шагов.
- [Файл со страницы теперь едет через результат шага (`UploadedFileRef` в `stepResults`)] → воркер уже детектит `isUploadedFileRef` по значениям входов; убедиться, что резолв `s{idx}:{key}` возвращает объект целиком (не разворачивает), и что `objectName` не светится в UI результата (сводка должна показывать имя/размер файла, не внутренние ссылки).

## Migration Plan

Миграций нет — вместо них сброс локальной базы (`pnpm reset`) и пересев (`seed-local.mts` обновляется: демо-сценарий со страницей ввода и финальной display-страницей). Удалённая засеянная БД, если используется, пересеивается тем же сидом. Откат кода возможен обычным revert'ом; данные после отката снова пересеваются.

## Open Questions

- Нужна ли кнопка «Продолжить» на display-only странице посреди потока (пауза-подтверждение)? Пока — нет (D3), финальный кейс важнее; вернуться при первом реальном запросе.
- Показывать ли выходы page-шага в сводке шага в списке (как у api-шагов показываются выходные поля)? Предлагается да, дёшево и добавляет прозрачности.
