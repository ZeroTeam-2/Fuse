# Proposal: merge-file-step-into-api-step

## Why

Отдельный тип шага «Файл» — устаревшая концепция. С появлением визуальной привязки блоков страницы к параметрам (dropzone → body-вход endpoint'а) файловый шаг полностью дублирует конфигурацию api-шага (приложение, endpoint, входы по местам, маппинги, страница), но конфигуратора в билдере у него нет (`StepPicker` создаёт пустышку без `appId`/`uploadPath`), а подмешать к файлу другие данные из UI невозможно. Живой кейс (llm-doc-recognizer, поля `doc` + `data`) показал: пользователь корректно настраивает всё через **api-шаг**, и не хватает единственного — чтобы worker закодировал тело `multipart/form-data`, когда среди входов есть загруженный файл. Способ кодирования тела — свойство endpoint'а, а не отдельного типа шага.

## What Changes

- **Worker**: `executeApiStep` отправляет `multipart/form-data`, когда среди разрешённых входов шага есть файловая ссылка (`UploadedFileRef` из dropzone): файл — в поле по схеме endpoint'а (`type: "file"`) либо по ключу привязки, остальные body-входы — текстовыми полями рядом; без файла — JSON, как сейчас. Логика переносится из `executeFileStep`.
- **BREAKING (доменно): тип шага `file` выводится из употребления** — исполнение файлового шага заменяется доменной ошибкой с подсказкой использовать api-шаг; «Файл» удаляется из палитры шагов билдера. Тип `FileStep` остаётся в union помеченным `@deprecated` (старые документы сценариев читаются).
- **Стадия обработки** (бывший `statusEndpoint` файлового шага) — композиция с существующим шагом «Периодический запрос»: api-шаг возвращает `task_id`, periodic-шаг опрашивает статус. Новых механизмов не требуется.
- **Опрос**: `pollUntilComplete` получает терминальные статусы ошибки (`status: "error" | "failed"` в ответе) — шаг падает доменной ошибкой сразу, а не крутится до 5-минутного таймаута (перенесённый follow-up 5.5 прошлого change).
- **Миграция данных**: единственный рабочий файловый шаг (сценарий «Распознать документ», сконвертированный вручную) переводится на пару api + periodic.

## Capabilities

### New Capabilities

_нет._

### Modified Capabilities

- `scenario-builder`: «Типы шагов» — пункт «Файл» удаляется; загрузка файла описывается как свойство api-шага с файловым body-входом и dropzone-привязкой.
- `scenario-execution`: «Движок исполнения шагов» — api-шаг MUST слать multipart при файловом входе; тип `file` — доменная ошибка; терминальные статусы ошибки опроса. «Загрузка файлов (single и chunked)» — формулировка «для файловых шагов» заменяется на «для api-шагов с файловым входом»; требование «Исполнение файлового шага» удаляется (поглощено движком).

## Impact

- **Код**: `apps/backend/src/execution/worker.service.ts` (`executeApiStep`, `executeFileStep`, `pollUntilComplete`), `apps/frontend/components/scenario/AddStepMenu.vue` + `StepPicker.vue` (палитра), `packages/shared/src/types/index.ts` (`@deprecated FileStep`), тесты worker'а.
- **Не меняется**: загрузка из PageRunner (`useFileUpload`, single/chunked, прогресс/resume), uploads API, MinIO, валидация dropzone (`accept`/`maxFileMb`), парсер (`format: binary` → `file`) — всё из archived change `implement-file-step-upload` переживает слияние, меняется только точка доставки файла провайдеру.
- **Данные**: сценарии со старыми file-шагами (до этой недели — нерабочие заглушки) при запуске получают понятную доменную ошибку; локальный сценарий «Распознать документ» мигрируется на api + periodic.
