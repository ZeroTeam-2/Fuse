# scenario-execution — delta (merge-file-step-into-api-step)

## ADDED Requirements

### Requirement: Multipart-вызов api-шага с файловым входом
Если среди разрешённых входов api-шага есть ссылка на загруженный файл (`UploadedFileRef` из dropzone-блока страницы), worker MUST отправлять запрос телом `multipart/form-data`: объект читается из хранилища платформы и уходит файловой частью, остальные body-входы — текстовыми полями рядом, path/query/header-входы — по своим местам как обычно. Имя multipart-поля файла MUST определяться по body-входу типа `file` из схемы endpoint'а, при его отсутствии — по ключу привязки dropzone-блока, если он совпадает с body-входом; крайний фолбэк — `file`. Заголовок `Content-Type` MUST выставляться средой исполнения (boundary), а не жёстко. Без файловой ссылки во входах api-шаг MUST отправлять JSON-тело, как прежде. Внутренние ссылки хранилища (`objectName`, presigned URL) MUST NOT попадать в результат шага.

Шаг типа `file` MUST завершаться доменной ошибкой с подсказкой использовать api-шаг с файловым входом: тип выведен из употребления, его функционал поглощён api-шагом, а стадия обработки строится композицией со следующим шагом «Периодический запрос».

#### Scenario: Файл и обычные поля в одном запросе
- **WHEN** api-шаг вызывает endpoint с файловым входом `doc` (привязан к dropzone) и текстовым body-входом `data` (константа), и пользователь загрузил файл со страницы шага
- **THEN** worker отправляет один `multipart/form-data`-запрос: файл в поле `doc`, значение константы — текстовым полем `data`, и сохраняет ответ провайдера как результат шага

#### Scenario: Api-шаг без файла не меняет поведения
- **WHEN** api-шаг исполняется без файловых ссылок во входах
- **THEN** запрос уходит с JSON-телом, как до изменения

#### Scenario: Обработка после загрузки — композиция с periodic-шагом
- **WHEN** сценарий состоит из api-шага загрузки файла (возвращает `task_id`) и следующего periodic-шага, опрашивающего `/task/{task_id}` с маппингом `task_id` из выхода первого шага
- **THEN** первый шаг доставляет файл multipart-запросом, второй опрашивает статус до признака завершения, и результатом сценария становится финальный ответ опроса

#### Scenario: Легаси-шаг типа file
- **WHEN** worker исполняет шаг устаревшего типа `file`
- **THEN** шаг завершается доменной ошибкой, называющей шаг и предлагающей использовать api-шаг с файловым входом, а запуск переходит в `failed` без повторной доставки

### Requirement: Терминальные статусы ошибки опроса
Цикл опроса (периодический шаг) MUST завершаться доменной ошибкой немедленно, если ответ опроса содержит статус ошибки (`status: "error"` или `status: "failed"`), с диагностикой из тела ответа — вместо продолжения опроса до таймаута.

#### Scenario: Провайдер сообщил об ошибке задачи
- **WHEN** периодический опрос получает ответ `{"status": "error", "error": "Failed to open stream"}`
- **THEN** шаг сразу завершается доменной ошибкой с текстом причины, запуск переходит в `failed`, опрос не продолжается до таймаута

## MODIFIED Requirements

### Requirement: Загрузка файлов (single и chunked)
Для api-шагов с файловым body-входом (dropzone-блок страницы, привязанный к входу) система SHALL выбирать режим загрузки по размеру файла: файлы ≤ 10 МБ (настраиваемый порог `FILE_SINGLE_UPLOAD_MAX_MB`, публикуемый фронтенду через runtime-конфиг) загружаются одним запросом `multipart/form-data`; файлы > 10 МБ загружаются чанками (5–10 МБ) с отображением прогресса (проценты и байты), поддержкой паузы/продолжения и отмены. При обрыве соединения загруженные чанки MUST сохраняться, и пользователь MAY возобновить загрузку с места обрыва. Файлы MUST проверяться на допустимый формат и лимит размера, заданные страницей шага.

Загрузка MUST выполняться из страницы шага в хранилище платформы (существующее API `/api/uploads/*`) **до** сабмита страницы: пока обязательный dropzone-блок не завершил загрузку, продолжение шага MUST быть заблокировано. В данные сабмита страницы MUST попадать ссылка на загруженный объект (`objectName`, имя, размер, тип файла), а не содержимое файла и не голые метаданные.

#### Scenario: Single-загрузка до порога
- **WHEN** пользователь выбирает в dropzone файл размером не больше порога `FILE_SINGLE_UPLOAD_MAX_MB`
- **THEN** файл уходит одним `multipart/form-data`-запросом с отображением прогресса, и после завершения сабмит страницы передаёт шагу ссылку на объект хранилища

#### Scenario: Chunked-загрузка сверх порога
- **WHEN** пользователь выбирает файл больше порога
- **THEN** система инициирует чанковую сессию, грузит части по 5–10 МБ с прогрессом в процентах и байтах, позволяет паузу/продолжение и отмену, а по завершении передаёт шагу ссылку на объект

#### Scenario: Возобновление после обрыва
- **WHEN** соединение обрывается на середине загрузки
- **THEN** система показывает «Соединение прервалось на X% — загруженные чанки сохранены» и после «Возобновить загрузку» продолжает с того же места

#### Scenario: Файл не проходит валидацию страницы
- **WHEN** выбранный файл не подходит по формату или превышает лимит размера, заданные dropzone-блоком страницы шага
- **THEN** загрузка не начинается, пользователь видит понятную ошибку и может выбрать другой файл

## REMOVED Requirements

### Requirement: Исполнение файлового шага
**Reason**: Тип шага `file` выведен из употребления — доставка файла провайдеру стала свойством api-шага (см. ADDED «Multipart-вызов api-шага с файловым входом»), а стадия обработки (`statusEndpoint`) — композицией со существующим шагом «Периодический запрос».
**Migration**: Сценарии с файловыми шагами пересобираются как api-шаг с dropzone-привязкой к файловому входу + (опционально) periodic-шаг опроса статуса с маппингом идентификатора задачи из выхода api-шага.
