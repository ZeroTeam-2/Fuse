## Context

Текущая модель маппинга плоская: `step.mappings[fieldKey]` = `"user" | "const" | "s{idx}:{outKey}"`, значение константы — в `step.consts[fieldKey]`. Резолвер (`apps/backend/src/execution/mapping-resolver.ts:53-62`) для ссылки `s{idx}:{key}` делает ровно одно: `(stepResults[idx].result as Record<string, unknown>)[key]`.

Три факта из кода, определяющие дизайн:

1. **Факт массива теряется на импорте.** `openapi-parser.ts:extractSchemaFields` делает `const target = schema.type === "array" ? schema.items : schema` — то есть массивный ответ молча разворачивается в поля элемента. Схема шага утверждает «объект с полем `id`», рантайм получает `[{id}, …]`, `result["id"]` → `undefined`. Без восстановления этого признака фильтр не на чем строить: UI не знает, когда его показывать.
2. **Поля-массивы внутри объекта пусты.** `mapSchemaType` даёт `type: "array"`, но `SchemaField` не имеет `items` — что внутри списка, неизвестно. Второй по частоте REST-шейп (`{items: [...], total}`) сейчас неописуем.
3. **Ветка «ручной ввод» мертва.** `worker.service.ts` вызывает `resolveMappings(step, ctx.stepResults)` — без третьего аргумента `userInput`. Значение страницы шага (`waitForPageInput`) возвращается, но результат отбрасывается. Значит режим «ручной ввод» у значения фильтра не заработает, пока это не починено.

Поблажки, которые упрощают работу: `steps` в Mongoose — `[Mixed]`, а `UpdateScenarioDto.steps` — `unknown[]` без `@ValidateNested`. Новое поле шага доедет до БД и обратно **без правок бэкенда**. А вот `SchemaFieldDoc` (`app.schema.ts`) — strict-сабсхема: `items` без правки схемы не сохранится.

## Goals / Non-Goals

**Goals:**
- Схема шага честно сообщает, что выход — массив, и описывает поля элемента.
- Входной параметр с источником «Из шага» получает условие: поле элемента + оператор + значение (ручной ввод / константа / из шага).
- Worker отбирает по условию ровно один элемент; пустая выборка — внятная ошибка, а не тихий `undefined`.
- Старые сценарии не ломаются.

**Non-Goals:**
- Фан-аут: исполнить шаг N раз по числу элементов массива. Здесь только «сузить до одного».
- Составные условия (`AND` / `OR`, несколько предикатов). Один предикат на параметр.
- Вложенные пути (`s0:data.items[2].id`) и глубина схемы > 1 уровня. Парсер и сейчас плоский; углубление — отдельная задача.
- Сортировка/агрегация («самый свежий», «максимальный»). Оператор отбирает, а не ранжирует.
- ~~Починка группировки `resolved` по `loc`~~ — **пришлось включить в change**. Расхождение (worker читает `resolved.path` / `resolved.query`, а резолвер отдаёт плоский объект по ключам полей) оказалось не «известным неудобством», а полным блокером: path-параметр не подставлялся вообще, в URL уезжал литеральный `{guid}`, и любой шаг с path-параметром падал с 422 — то есть отобранный фильтром элемент физически не мог доехать до запроса. См. `groupInputsByLocation` и требование «Подстановка входов по месту в HTTP-запросе» в дельте `scenario-execution`.

## Decisions

### 1. Признак массива — на уровне схемы выходов, а не через новый тип поля

Схема шага (`GET /api/scenarios/:id/step-schema/:i`) начинает отдавать `outputIsArray: boolean` рядом с `inputs` / `outputs`; `outputs` по-прежнему описывают **поля элемента**. Тот же флаг сохраняется у endpoint'а приложения при импорте.

*Почему так.* Альтернатива — представить массивный ответ одним полем `{key: "", type: "array", items: [...]}` — ломает и текущий UI («Поле результата» перечисляет `outputs`), и резолвер (`s{i}:{key}` по ключу элемента). Флаг сохраняет обе структуры нетронутыми: `outputs` остаются списком полей, к которым можно маппиться, и просто получают смысл «поля одного элемента». Плюс это ровно тот бит, который парсер сегодня выбрасывает — мы его не изобретаем, а перестаём терять.

Для полей-массивов **внутри** объекта — `SchemaField.items?: SchemaField[]` (глубина 1, как и весь остальной парсер). Это покрывает шейп `{items: [...], total}`.

### 2. Фильтр — отдельная запись на шаге, ключ = ключ входного параметра

```ts
export type FilterOperator = "eq" | "ne" | "gt" | "lt" | "gte" | "lte" | "contains";

export interface StepFilterValue {
  mode: "user" | "const" | "ref";
  const?: string;   // mode=const, поддерживает {{s0:key}}-шаблон
  ref?: string;     // mode=ref, "s{idx}:{key}"
}

export interface StepFilter {
  arrayPath?: string;   // пусто/отсутствует => массив — это сам результат шага;
                        // иначе ключ поля-массива в результате (напр. "items")
  field: string;        // поле элемента, по которому сравниваем
  op: FilterOperator;
  value: StepFilterValue;
}

// BaseStep
filters?: Record<string, StepFilter>;   // ключ = key входного параметра шага
```

*Почему параллельная запись, а не расширение строки маппинга.* Строковый формат (`s0:id?inn=eq:7707083893`) пришлось бы парсить в трёх местах и он не выражает «значение из шага» в качестве операнда. Отдельная структура — прямое расширение уже существующего приёма «`mappings` + параллельный `consts`, оба по ключу поля». Резолвер по-прежнему сначала читает `mappings[field]`, и только для ветки `s{idx}:` заглядывает в `filters[field]`.

*Почему `arrayPath` хранится, а не выводится в рантайме.* Worker не имеет доступа к схемам шагов (он читает `Run` и `App`, а `getStepSchema` живёт в `ScenariosService`). Фронт знает схему в момент настройки — он и фиксирует, где лежит массив. Рантайм остаётся автономным.

### 3. Семантика резолва (ветка `s{idx}:{key}`)

```
value = stepResults[idx].result
arr   = filter?.arrayPath ? value[filter.arrayPath] : value
если arr — не массив  → поведение как сегодня: value[key]
если arr — массив:
    нет filter        → arr[0]?.[key]                       // мягкая деградация
    есть filter       → matches = arr.filter(предикат)
                        0        → доменная ошибка шага (Run → failed)
                        >1       → matches[0][key] + предупреждение в результат шага
                        1        → matches[0][key]
```

Ошибка при 0 совпадений и «первый + предупреждение» при >1 — решение владельца: пустая выборка почти всегда означает сломанный сценарий, и честнее упасть с текстом «фильтр по `inn` = 7707083893 не дал совпадений», чем тихо подставить `undefined` и сломаться тремя шагами позже, в неочевидном месте.

Массив без фильтра сегодня даёт `undefined` — брать первый элемент строго лучше и не ломает ни один работающий сценарий.

### 4. Сравнение — по типу поля элемента, приведение в резолвере

Тип поля берётся из схемы… которой у worker'а нет (см. решение 2). Поэтому тип **не читается из схемы в рантайме**, а выводится из фактического значения элемента:

- элемент даёт `number` → операнд приводится `Number(...)`, сравнение числовое;
- `boolean` → операнд приводится по `"true"/"false"`;
- иначе → строковое сравнение, `contains` = регистронезависимая подстрока.

*Почему так, а не тащить тип из схемы.* Прокидывать `SchemaField.type` в фильтр — дублировать схему в документе сценария и ловить рассинхрон при переимпорте спеки. Значение элемента — источник истины в момент сравнения. UI при этом всё равно ограничивает набор операторов по типу из схемы (спека `scenario-builder`), так что бессмысленные комбинации не создаются; рантайм лишь не обязан им доверять.

### 5. Значение условия — те же три источника, что и у маппинга

- `const` → `resolveTemplate(value.const, templateContext)`, тот же путь, что у обычной константы.
- `ref` → `s{idx}:{key}` из результатов шагов. Вложенный фильтр не поддерживается: операнд обязан быть скаляром. Если по ссылке лежит массив — берётся первый элемент (то же правило мягкой деградации).
- `user` → значение из пользовательского ввода. Ключ — `filter:{paramKey}`, чтобы не сталкиваться с ключом самого параметра (он может одновременно быть в режиме `user`).

### 6. Прокидывание пользовательского ввода в резолвер (предпосылка)

`worker.service.ts` начинает передавать третий аргумент: данные страницы шага (результат `waitForPageInput`, который сейчас отбрасывается) и/или входные данные запуска. Без этого режим `user` у значения фильтра — мёртвая ветка, ровно как сегодня мёртв режим «Ручной ввод» у самого маппинга.

Правку держим минимальной: собрать `userInput` в `StepContext` и передать в `resolveMappings`. Переделка формы простого режима запуска (она собирает входы только первого шага) — **за рамками**; в этой итерации `user`-значение фильтра работает там, где данные реально доходят (страница шага, входы первого шага).

## Risks / Trade-offs

- **Старые приложения импортированы без `outputIsArray` / `items`** → UI не покажет блок фильтра, пока спеку не переимпортируют. Митигация: механизм «Обновление спецификации» уже существует; рантайм деградирует мягко (работает по фактическому типу результата, флаг ему не нужен). В задачах — явный шаг: проверить на реальном приложении после переимпорта.
- **`SchemaFieldDoc` — strict-сабсхема** → забыть `items` в `app.schema.ts` = поле молча не сохранится, и парсер «вроде работает», а в БД пусто. Митигация: тест на круговой путь парсер → БД → `getStepSchema`.
- **Фронт фиксирует `arrayPath` в момент настройки; спека приложения может измениться** → после переимпорта поле-массив может исчезнуть/переехать, а `arrayPath` останется. Тогда `value[arrayPath]` = `undefined`, `arr` не массив → сработает старый путь `value[key]` → `undefined`. Это не хуже сегодняшнего поведения, но и не диагностируется. Митигация: при не-массиве по заданному `arrayPath` писать предупреждение в результат шага.
- **«Первый элемент» при >1 совпадении зависит от порядка ответа API** → недетерминированность, если API не гарантирует сортировку. Осознанный компромисс (альтернатива — падать, отвергнута владельцем); предупреждение в результате шага делает это видимым.
- **Оператор `contains` для чисел/булевых** UI не предлагает, но документ сценария можно собрать руками (DTO шагов не валидируется). Рантайм в этом случае сравнивает строки — не падает, но результат может удивить. Валидацию DTO шагов не вводим: это отдельная, более широкая задача.
- **Тестового покрытия worker'а нет вообще** (`worker.service.ts` не тестируется). Новую логику отбора кладём в `mapping-resolver.ts` — чистую функцию, покрываемую vitest'ом, а не в worker.
