# scenario-execution (delta)

## MODIFIED Requirements

### Requirement: Движок исполнения шагов
Worker SHALL исполнять шаги сценария последовательно: для каждого шага разрешать входы по маппингу (выходы предыдущих шагов, ввод пользователя, константы, включая подстановку `{{token}}`-шаблонов), выполнять HTTP-вызов endpoint стороннего API и сохранять ответ как выходы шага в документ `Run`.

Для шагов, привязанных к приложению (`api`, `periodic`, `file`), worker MUST разрешать **абсолютный URL** вызова: загружать `App` по `appId` шага и склеивать `App.baseUrl` с путём шага (`path` / `pollPath` / `uploadPath`). Относительный путь шага MUST NOT передаваться в HTTP-клиент напрямую. Если приложение не найдено либо у него не задан `baseUrl`, шаг MUST завершаться доменной ошибкой с диагностикой, указывающей на приложение и на необходимость переимпорта спецификации. Резолвнутый URL MUST проходить проверку SSRF перед вызовом.

Ответ API-шага worker MUST сохранять по типу содержимого: `application/json` — как распарсенный JSON в `stepResults[i].result`; `text/*` — как строку; иное (файловое содержимое, а также ответы с `Content-Disposition: attachment`) — бинарно в S3 как артефакт запуска, записывая в `stepResults[i].result` файловую ссылку (`RunFileRef`) и регистрируя объект в реестре файлов запуска. Сырое бинарное тело MUST NOT записываться в Mongo.

Шаг «Задержка» MUST приостанавливать поток на заданное время; «Периодический запрос» MUST опрашивать endpoint с интервалом до признака завершения, публикуя прогресс; «Другой сценарий» MUST исполнять вложенный сценарий как один шаг. Ошибка шага MUST переводить `Run` в статус `failed` с диагностикой (шаг, endpoint, статус ответа).

Запись результатов шагов MUST быть идемпотентной: повторная обработка одного и того же `runId` (повторная доставка сообщения очереди) MUST NOT создавать дублирующиеся записи `stepResults` с одинаковым `stepIndex` и MUST NOT переводить `Run` из терминального статуса (`completed`, `failed`, `cancelled`) обратно в незавершённый.

При переводе `Run` в терминальный статус, а также в `waiting_input`, worker MUST инициировать создание уведомления владельцу запуска (см. capability `run-notifications`).

#### Scenario: Резолв базового URL приложения
- **WHEN** worker исполняет `api`-шаг с `path: "/collections"` у приложения с `baseUrl: "https://dadata-fake.cloud.astral-dev.ru"`
- **THEN** worker вызывает `https://dadata-fake.cloud.astral-dev.ru/collections` и сохраняет тело ответа как результат шага

#### Scenario: Base path приложения сохраняется
- **WHEN** у приложения `baseUrl: "https://api.example.com/v1"`, а путь шага — `/collections`
- **THEN** worker вызывает `https://api.example.com/v1/collections`

#### Scenario: У приложения не задан базовый URL
- **WHEN** worker исполняет шаг приложения, у которого `baseUrl` пуст
- **THEN** шаг завершается ошибкой с сообщением, называющим приложение и предлагающим переимпортировать спецификацию, а `Run` переходит в `failed` — без низкоуровневой ошибки разбора URL

#### Scenario: Подстановка шаблона в константе
- **WHEN** константа шага содержит `Bearer {{access_token}}` и предыдущий шаг вернул `access_token`
- **THEN** worker подставляет значение токена в заголовок Authorization перед HTTP-вызовом

#### Scenario: Периодический опрос до завершения
- **WHEN** шаг «Периодический запрос» опрашивает endpoint каждые 5 сек до признака завершения в ответе
- **THEN** worker публикует прогресс (значение поля прогресса) на каждой итерации и завершает шаг при получении признака завершения

#### Scenario: Файловый ответ API уходит в S3
- **WHEN** api-шаг (или финальный запрос periodic-шага) возвращает `Content-Type: application/zip`
- **THEN** worker сохраняет тело в S3, в `stepResults[i].result` попадает `RunFileRef`, объект регистрируется в реестре файлов запуска, а бинарное тело в Mongo не пишется

#### Scenario: Повторная доставка сообщения не дублирует результаты
- **WHEN** сообщение с одним и тем же `runId` доставляется worker'у повторно
- **THEN** результаты шагов перезаписываются по индексу, дубликаты `stepResults` не появляются, а уже терминальный `Run` не возвращается в `running` или `waiting_input`

#### Scenario: Доменная ошибка шага не уходит на повторную доставку
- **WHEN** шаг падает детерминированно (недоступный endpoint, ошибка стороннего API)
- **THEN** worker фиксирует `failed` в `Run`, публикует ошибку и подтверждает сообщение очереди — повторные доставки того же детерминированного сбоя не выполняются
