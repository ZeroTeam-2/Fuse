# Proposal: add-background-run-history

## Why

Запуски сценариев уже исполняются асинхронно (SQS + воркер), но пользователь вынужден сидеть на странице запуска и ждать результата — уйдя со страницы, он теряет к нему путь (нет ни списка запусков, ни уведомлений). Для длительных сценариев (periodic-шаги, страницы с вводом) это делает платформу практически непригодной: нужен раздел «Запуски» с историей, уведомление о завершении и надёжное хранение файлов-результатов.

## What Changes

- **Раздел «Запуски»** в навигации: список активных и завершённых запусков текущего пользователя с пагинацией, статусами, переходом к результатам (существующая страница запуска уже умеет восстанавливаться по `?run=<id>`) и повторным запуском с теми же входами. UI — по готовому эталону дизайн-системы `assets/Fuse Design System/ui_kits/fuse-app/Runs.jsx` (экран «Запуски» + `NotificationBell`).
- **Список и удаление на API**: новые endpoint'ы `GET /api/runs` (список своих запусков с фильтрами) и `DELETE /api/runs/:id`; удаление стирает документ `Run` **и все связанные файлы в S3** (и загруженные пользователем входы, и файлы-результаты).
- **Файлы-результаты API в S3**: воркер при получении от внешнего API бинарного/файлового ответа сохраняет его в S3 (сейчас такой ответ портится через `response.text()` и оседает строкой в Mongo) и кладёт в результат шага файловую ссылку; скачивание — по presigned URL.
- **Уведомления («колокольчик»)**: при завершении запуска (успех/ошибка/отмена) пользователю создаётся уведомление; колокольчик в шапке показывает непрочитанные, доставка в реальном времени через существующий socket.io-паттерн (комната `user:{userId}`).

## Capabilities

### New Capabilities

- `run-history`: раздел и API истории запусков — список активных/завершённых запусков пользователя, просмотр результатов, удаление запуска с каскадным удалением его файлов из S3.
- `run-artifacts`: хранение файлов, полученных от внешних API в ходе запуска, в S3; файловая ссылка в результате шага; скачивание по presigned URL; учёт файлов запуска для каскадного удаления.
- `run-notifications`: уведомления о завершении запусков — модель уведомления, колокольчик с непрочитанными, realtime-доставка, отметка прочитанности.

### Modified Capabilities

- `scenario-execution`: требование «Движок исполнения шагов» меняется — ответ API-шага с файловым содержимым MUST сохраняться в S3 как артефакт запуска, а в `stepResults[i].result` MUST записываться файловая ссылка (а не сырое тело ответа); завершение запуска MUST порождать уведомление пользователю.

## Impact

- **Backend** (`apps/backend/src`):
  - `execution/` — `execution.controller.ts` (+`GET /api/runs`, +`DELETE /api/runs/:id`), `execution.service.ts` (список, удаление с каскадом), `worker.service.ts` (детект файлового ответа в `executeApiStep`/`pollUntilComplete`, выгрузка в S3), `run.schema.ts` (индексы `userId+createdAt`, учёт файлов запуска).
  - `minio/minio.service.ts` — используется как есть (`uploadFile`, `getPresignedUrl`, `deleteFile`).
  - Новый модуль `notifications/` — схема Mongo, сервис, контроллер, gateway (или расширение `websocket/`).
  - `websocket/run.gateway.ts` — образец; появится per-user комната.
- **Shared** (`packages/shared`): типы `RunListItem`/`RunFileRef`/`Notification`, события WS для уведомлений, возможно расширение `UploadedFileRef`.
- **Frontend** (`apps/frontend`): новая страница `pages/my/runs/index.vue`, пункт в `navItems` (`layouts/default.vue`), компонент колокольчика в шапке, стор/composable уведомлений, отображение и скачивание файловых результатов в `RunPanel.vue`; регенерация `src/types/api.ts`.
- **Инфраструктура**: без изменений — используются существующие MinIO/S3, SQS, socket.io.
- **Данные**: старые запуски без списка файлов остаются валидными (каскад удаляет только известные файлы); breaking-изменений API нет.
