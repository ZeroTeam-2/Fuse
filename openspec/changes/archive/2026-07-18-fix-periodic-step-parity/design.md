# fix-periodic-step-parity — design

## Context

Периодический запрос задумывался как «Endpoint API + повтор раз в N секунд до признака завершения», но реализация разошлась с этой идеей на трёх уровнях:

1. **Создание** — [StepPicker.vue](../../../apps/frontend/components/scenario/StepPicker.vue) при добавлении periodic-шага копирует из выбранного endpoint только `method` и `path` (в `pollMethod`/`pollPath`), а `endpointId` выбрасывает. У `PeriodicStep` в shared-типах этого поля просто нет.
2. **Схема** — `ScenariosService.getStepSchema` умеет строить схему только для `api` (по `endpointId`) и `scenario`; periodic попадает в `default` и получает `EMPTY_SCHEMA`. Панель `StepConfig.vue` рендерит секции Path/Query/Header/Body из `schema.inputs` — при пустой схеме секций нет, выходов для следующих шагов тоже нет.
3. **Исполнение** — `WorkerService.executePeriodicStep` передаёт в `groupInputsByLocation` пустой список входов (спасают только плейсхолдеры пути) и опрашивает `fetch(url, { method })` — без заголовков, тела и настроенных query-параметров. `executeApiStep` всё это делает.

Панель настройки (`StepConfig.vue`) уже полностью универсальна: она показывает `pollMethod`/`pollPath` в шапке и группирует входы по `loc`. Чинить нужно данные (тип + схема + исполнение), а не UI.

## Goals / Non-Goals

**Goals:**
- Periodic-шаг настраивается идентично api-шагу: все секции параметров по местам, выходные поля, автоподстановка маппингов.
- Выходы periodic-шага доступны следующим шагам (и вложенным сценариям через `getStepSchemaForStep`).
- Воркер шлёт каждый опрос с полным набором входов: path в путь, query в строку запроса, header в заголовки, body в тело.
- Существующие сценарии с periodic-шагами (без `endpointId`) продолжают работать без миграции данных.
- Сквозная проверка шагов «Другой сценарий» и «Задержка» (входит в верификацию change'а).

**Non-Goals:**
- Изменение UI-панели `StepConfig.vue` (не требуется — рендер идёт из схемы).
- Изменение условия завершения опроса (`isPollComplete`), таймаута `POLL_TIMEOUT_MS`, поля прогресса.
- Паритет для `file`-шага (у него своя модель с `uploadPath`/`statusEndpoint`) — отдельный change, если понадобится.
- Автоматическая миграция легаси-шагов в БД.

## Decisions

### 1. `endpointId` у `PeriodicStep` — опциональное поле

`PeriodicStep` получает `endpointId?: string`. Опциональность — осознанная: в БД уже лежат шаги без него, и жёсткий тип заставил бы врать на чтении. `StepPicker` при создании пишет `endpointId: ep.id` (endpoint и так уже выбран — терялась только ссылка). `pollMethod`/`pollPath` остаются как есть: их использует шапка панели и воркер, и они же — ключ фолбэка для легаси.

*Альтернатива — заменить `pollMethod`/`pollPath` на `endpointId` полностью*: чище, но ломает существующие документы и требует миграции; отклонено.

### 2. Схема periodic-шага — переиспользование логики api-шага + фолбэк по методу и пути

В `getStepSchema` и `getStepSchemaForStep` добавляется ветка `case "periodic"`, которая:
1. при наличии `endpointId` ищет endpoint по нему (та же логика, что `getApiStepSchema`);
2. для легаси-шагов без `endpointId` ищет endpoint приложения по совпадению `method === pollMethod && path === pollPath`;
3. если endpoint не найден (удалён, переимпортирован) — возвращает `EMPTY_SCHEMA`, **не** кидает 404.

Пункт 3 отличается от api-ветки (та кидает `NotFoundException`): для легаси-шага пустая схема — это текущее поведение, и превращать его в ошибку редактора нельзя. Для шага с явным `endpointId`, у которого endpoint пропал, пустая схема тоже мягче: редактор продолжает работать, а сломанность шага уже покрывается флагом `broken`.

### 3. Воркер: входы endpoint и полный запрос на каждой итерации

`executePeriodicStep` повторяет подготовку `executeApiStep`:
- загрузить приложение (`loadApp` — уже кэширован на запуск), найти endpoint (по `endpointId`, фолбэк по `pollMethod`/`pollPath`);
- передать `endpoint.inputs` в `groupInputsByLocation(resolved, inputs, step.pollPath)` — приоритет плейсхолдеров пути сохраняется, поэтому легаси-шаги с пустой схемой не регрессируют;
- собрать URL (path + query) и опции запроса (Content-Type, header-входы, body для не-GET/DELETE) **один раз до цикла** — входы между итерациями не меняются;
- каждый опрос выполняется с этими опциями вместо голого `{ method }`.

Общую часть «endpoint → located → url + RequestInit» стоит вынести в приватный хелпер, чтобы api- и periodic-ветки не разъезжались снова.

### 4. Фронтенд не трогаем

`StepConfig.vue` рендерит секции из `schemas[stepIndex].inputs`, шапку — из `pollMethod`/`pollPath`, выходы — из `schema.outputs`. Автоподстановка маппингов и фильтры массивных выходов работают поверх той же схемы. Единственное изменение фронтенда — одна строка в `StepPicker.vue`.

## Risks / Trade-offs

- [Легаси-шаг, чей `pollPath` разошёлся с endpoint после переимпорта спецификации] → фолбэк не найдёт endpoint, схема останется пустой — это текущее поведение, не регрессия; исправляется пересозданием шага (тогда запишется `endpointId`).
- [Body в периодическом POST-опросе шлётся на каждой итерации] → это и есть ожидаемое поведение (например, опрос статуса задачи по POST с телом-фильтром); тело собирается один раз, побочных мутаций между итерациями нет.
- [Дублирование логики сборки запроса между api- и periodic-ветками] → выносим общий хелпер (см. решение 3); полное слияние веток не делаем — у periodic свой цикл, прогресс и признак завершения.
- [`getStepSchemaForStep` (для вложенных сценариев) забыт при правке] → правится в той же задаче и покрывается сценарием «выходы periodic-шага видны как выход вложенного сценария».

## Migration Plan

Миграция данных не требуется: новые шаги пишутся с `endpointId`, старые обслуживаются фолбэком по `pollMethod`/`pollPath`. Деплой обычный (backend + frontend вместе); откат безопасен — поле `endpointId` в документах старый код просто игнорирует.
