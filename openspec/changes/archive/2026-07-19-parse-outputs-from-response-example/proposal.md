# Выходы endpoint'а из примера ответа, когда схемы нет

## Why

Авто-сгенерированные спеки (типично FastAPI без `response_model`) описывают успешный ответ пустой схемой (`schema: {}`), а реальные поля лежат только в `example` рядом. Импортёр строит выходы строго из `schema.properties`, поэтому такой endpoint получает «нет выходных данных» — в редакторе сценариев к его ответу нельзя привязать ни маппинг «Из шага», ни блок отображения страницы, хотя рантайм эти поля реально возвращает (живой случай: `GET /task/{task_id}` сервиса llm-doc-recognizer с `example: {status, result}`).

## What Changes

- Парсер OpenAPI получает фолбэк: если схема успешного ответа не дала ни одного поля, выходы выводятся из **примера ответа** (`content.example`, `schema.example` либо первый из `content.examples[*].value`).
- Пример-объект даёт поля по ключам (тип — по JS-типу значения, значение — в `ex`); пример-массив объектов помечает ответ коллекцией и описывает элемент; вложенные массивы объектов получают схему элемента на один уровень, как и при разборе схемы.
- Схема с настоящими `properties` ведёт себя как раньше — фолбэк включается только при пустом результате разбора схемы.

## Capabilities

### New Capabilities

_нет._

### Modified Capabilities

- `api-app-management`: требование импорта дополняется правилом вывода схемы выходов из примера ответа при пустой/отсутствующей схеме.

## Impact

- **Backend**: `apps/backend/src/apps/openapi-parser.ts` (`extractOutputs`, чтение примера из content-объекта), unit-тесты `openapi-parser.spec.ts`.
- Фронтенд и рантайм не меняются: они уже работают с готовыми `outputs`.
- Данные: существующие приложения получают поля после переимпорта спеки.
