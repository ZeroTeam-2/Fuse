# Design — реимпорт из файла

## Context

`AppsService.reimport`/`applyReimport` завязаны на `requireOpenapiUrl` + `ssrfGuard.fetchSpec`; диф и слияние endpoints написаны инлайном в каждом методе. Файловый импорт при создании уже существует парой ручек (`import-preview-file` — превью, `from-file` — создание): файл уходит в обе, разбор — `parseSpecText` + `openapiParser.parse(raw, "", {baseUrlOverride})`, лимит размера — Multer из `SPEC_FILE_MAX_MB`. Фронтенд-страница `update.vue` показывает форму только при `app.openapiUrl`, иначе заглушку.

## Goals / Non-Goals

**Goals:** реимпорт из файла с тем же дифом и слиянием, что у URL-ветки; переключатель источника на странице обновления; сохранение текущего `baseUrl` приложения, когда файл его не сообщает.

**Non-Goals:** хранение загруженной спеки между «проверить» и «сохранить» (файл просто шлётся в обе ручки — ровно как при создании); изменение алгоритма дифа/слияния; обновление `openapiUrl` при файловом реимпорте (URL остаётся прежним, если был).

## Decisions

- **D1. Файл в обе ручки, состояния на сервере нет.** `POST :id/reimport-file` → диф, `POST :id/reimport-file/apply` → слияние. Симметрично паре создания; сервер остаётся stateless, «сохранить» применяет ровно тот файл, который пользователь видит в dropzone.
- **D2. Общий код выносится**: `diffEndpoints(current, parsed)` и `mergeEndpoints(current, parsed)` — приватные хелперы, используемые обеими ветками (URL и файл). Поведение URL-ветки не меняется.
- **D3. Базовый URL при apply из файла**: `parsed.baseUrl ?? app.baseUrl` (то же для `host`). Файл без абсолютного `servers` и без переопределения не должен затирать рабочий `baseUrl` приложения пустотой; ошибки «не удалось определить базовый URL» здесь нет — в отличие от создания, URL уже есть у приложения. Поле «Базовый URL API» в форме остаётся для явного переопределения.
- **D4. Валидация и лимиты — те же**: `validateSpecFile` (расширения) и Multer-лимит из конфига переиспользуются как есть.
- **D5. Фронтенд**: `SegmentedControl` «URL | Файл» (дефолт — `url`, если у приложения есть `openapiUrl`, иначе `file`), файловая ветка = `Dropzone` + необязательный «Базовый URL API»; «Сохранить обновление» в файловом режиме шлёт тот же файл в apply. Смена режима сбрасывает диф.

## Risks / Trade-offs

- [Файл между дифом и apply можно подменить] → apply применяет то, что в dropzone на момент клика; это то же окно, что и у пары создания. Диф носит информационный характер, слияние всегда пересчитывается по фактическому файлу.
- [Реимпорт файлом у приложения с URL «расщепляет» источник] → осознанно: `openapiUrl` сохраняется, следующая проверка по URL продолжит работать; файл — разовый способ обновления.
