## Context

Изменение `local-docker-infra-dev` дало воспроизводимую локальную инфраструктуру, но оставило среду пустой: после `infra:reset` в БД нет ни пользователя, ни данных. Единственный вход — Yandex OAuth (`JwtStrategy` читает JWT из куки `access_token`, секрет — `JWT_SECRET`). Playwright в проекте не установлен: `apps/frontend/e2e/smoke.spec.ts` импортирует `@playwright/test`, которого нет в зависимостях, — файл не запускается ни одной командой.

Ключевые факты, на которые опирается дизайн:

- `JwtStrategy` (`apps/backend/src/auth/strategies/jwt.strategy.ts:16`) берёт токен **только** из куки `access_token`, payload — `{ userId, email }`, подпись проверяется `JWT_SECRET`. Значит, валидную сессию можно получить, подписав токен тем же секретом, — без OAuth и без правок в коде.
- Воркер исполняет `api`-шаг реальным `fetch` на `app.baseUrl + endpoint.path` (`worker.service.ts:562`), поэтому сид-приложению нужен достижимый эндпоинт.
- В модели шагов есть `DelayStep` (`packages/shared/src/types/index.ts`, `type: "delay"`, `seconds`), который не ходит по сети. Такой шаг даёт герметичный запуск, доходящий до `completed` без внешних зависимостей.
- Запуск создаётся `POST /api/runs` с телом `{ scenarioId }`, статусы — `pending | running | completed | failed` (`RunStatus` в `@fuse/shared`).

## Goals / Non-Goals

**Goals:**

- После `pnpm infra:reset && pnpm seed` в UI есть с чем работать: пользователь, приложение, сценарии.
- Запуск сид-сценария доходит до терминального статуса локально — это закрывает задачу 4.4 предыдущего изменения.
- Регрессия «дубли сценариев» ловится тестом, а не руками.
- Тесты входят под сид-пользователем без OAuth, **не ослабляя авторизацию в приложении**.

**Non-Goals:**

- Дамп/миграция данных из прод-БД.
- Покрытие e2e всего UI — только ключевые сценарии из спеки.
- Тесты в CI (настройка пайплайна) — отдельная задача; здесь только команды, запускаемые локально.
- Замена существующих unit-тестов (vitest) чем-либо ещё.

## Decisions

### 1. Вход по подписанной куке, а не «выключатель авторизации»

Тест подписывает JWT (`{ userId, email }`) секретом `JWT_SECRET` из окружения и кладёт его в куку `access_token` — ровно ту, которую читает `JwtStrategy`. Для Playwright это `storageState` с готовой кукой, для скриптов — заголовок `Cookie`.

*Почему так:* предлагавшийся вариант «отключить проверку авторизации в dev» создаёт в продакшен-коде ветку, которая при неверном значении переменной окружения превращается в полный обход аутентификации. Это классический источник инцидентов, и выигрыша он не даёт: подпись куки решает ту же задачу, не касаясь guard'а. Спека это фиксирует явным требованием: режима, отключающего проверку токена, в коде быть не должно.

*Следствие:* тесты обязаны знать `JWT_SECRET`. Берём его из того же окружения, что и приложение (`.env` / `.env.local`).

### 2. Герметичный `delay`-сценарий как основной прогон

Сид создаёт сценарий с одним шагом `type: "delay"` (2 секунды). Запуск такого сценария проходит полный тракт — `POST /api/runs` → SQS → воркер → терминальный статус — **не обращаясь наружу**. Это делает e2e-тест детерминированным: он проверяет инфраструктуру и тракт исполнения, а не доступность чужого API.

*Альтернатива (отклонена):* строить основной прогон на `api`-шаге к внешнему сервису. Тест начинает падать от чужой недоступности и перестаёт быть сигналом о нашем коде.

### 3. Мок-API как сервис локальной инфраструктуры

Для `api`-шагов в `docker-compose.dev.yml` добавляется контейнер-мок (`mccutchen/go-httpbin`), `baseUrl` сид-приложения указывает на него (`http://localhost:<порт>`). Второй сид-сценарий делает `api`-шаг к моку — он нужен, чтобы UI выглядел реалистично и чтобы `api`-тракт (сборка URL, маппинг ответа) тоже был проверен.

*Почему контейнер, а не стаб внутри теста:* мок должен быть доступен и когда разработчик просто кликает руками после `pnpm seed`, а не только во время прогона тестов. Инфраструктурный сервис даёт это бесплатно.

### 4. Сид отказывается работать против удалённой БД

Перед записью сид проверяет `MONGODB_URL`: хост обязан быть локальным (`localhost`/`127.0.0.1`). Иначе — ошибка и выход.

*Почему:* сид пишет данные и идемпотентно удаляет/переписывает свои записи. Запуск такого скрипта с боевым `.env` (в котором `MONGODB_URL` смотрит на прод) — ровно тот класс ошибок, ради предотвращения которого затевалась вся локальная инфраструктура.

### 5. Playwright с готовым `storageState`, без UI-логина

`playwright.config.ts` поднимает проект с `baseURL` фронтенда; глобальный setup генерирует `storageState` с кукой `access_token` для сид-пользователя. Тесты стартуют уже авторизованными. Существующий `e2e/smoke.spec.ts` остаётся как есть (он проверяет неавторизованные страницы) и наконец начинает запускаться.

Тесты из спеки: (а) создание сценария создаёт ровно один — прямая регрессия на дубли, (б) запуск сид-сценария доходит до терминального статуса.

### 6. Smoke-тест инфраструктуры — на vitest, без браузера

Отдельный тест (`test:infra`) проверяет очередь + DLQ, бакет и round-trip (SQS send/receive, S3 put/get) через те же SDK, что использует бэкенд. Быстрый, не требует поднятого приложения — только `pnpm infra`. Таймауты короткие: при опущенной инфраструктуре тест падает с понятным сообщением, а не висит.

## Risks / Trade-offs

- **Тестам нужен `JWT_SECRET`.** → Берётся из окружения; в отсутствие переменной тест падает с явным сообщением. Секрет не хардкодится.
- **Playwright тянет браузеры (сотни МБ) и удлиняет `pnpm install`.** → Ставится как dev-зависимость; браузеры скачиваются отдельной командой, а не автоматически при установке.
- **Сид-данные ломаются при эволюции схем.** → Сид пишется поверх типов из `@fuse/shared`, так что расхождение всплывает на typecheck, а не в рантайме теста.
- **Мок-контейнер — ещё один сервис в compose.** → Образ крошечный, запускается за секунды; альтернатива (стаб внутри теста) не покрывает ручные клики.
- **E2E зависят от запущенных фронта и бэка.** → Playwright-конфиг может поднимать их сам (`webServer`), но по умолчанию рассчитываем на уже запущенный `pnpm dev`; в README это фиксируется.

## Migration Plan

1. `pnpm infra` — инфраструктура (включая мок-API).
2. `pnpm seed` — данные.
3. `pnpm dev` — приложение.
4. `pnpm test:infra` и `pnpm test:e2e` — проверки.

Откат: удалить `.env.local` (возврат к удалённым сервисам); сид-данные живут только в локальной БД и сносятся через `pnpm infra:reset`.
