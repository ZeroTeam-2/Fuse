# Fuse Platform MVP — Design

## Context

Репозиторий — pnpm-монорепо: `apps/frontend` (Nuxt), `apps/backend` (NestJS), `packages/shared`, MongoDB и Redis через docker-compose. Кода фактически нет — это founding change. Источник требований — дизайн-прототип `Fuse.html` (все экраны и интеракции) и шесть спецификаций этого change. Скрипт `gen:types` уже настроен на генерацию типов фронтенда из OpenAPI-схемы backend (`/api/schema.json`).

Инфраструктура docker-compose: **MongoDB** (данные), **Redis** (очереди задач + pub/sub для WebSocket-событий), **MinIO** (S3-совместимое файловое хранилище). Backend запускается в двух режимах: **main process** (API-сервер) и **worker process** (исполнитель сценариев).

Домен: **App** (подключённый API-провайдер с импортированной OpenAPI-спецификацией) → **Endpoint** (метод, путь, схемы входов/выходов) → **Scenario** (карточка с упорядоченными **Step**'ами пяти типов и **Page**'ами) → **Run** (исполнение сценария с состоянием шагов).

## Goals / Non-Goals

**Goals:**
- Рабочий вертикальный срез MVP: импорт OpenAPI → сборка сценария → публикация → запуск из маркетплейса.
- Исполнение сценариев на backend через очередь задач (main process + workers); данные и ключи сторонних API не утекают в браузер.
- Real-time прогресс выполнения через WebSocket.
- Chunked-загрузка файлов с resume; порог single/chunked по размеру.
- Типобезопасный контракт frontend↔backend через shared-типы и генерацию из OpenAPI.
- Опциональные интеграции: Yandex Metrica, сбор логов (Monium) — behind feature flags.

**Non-Goals (MVP):**
- Биллинг, тарифы, ключи доступа для потребителей API.
- Версионирование сценариев, командные воркспейсы, роли сложнее «владелец/пользователь».
- Ветвления/условия в сценариях — поток строго линейный (как в дизайне).
- Маршаллинг форматов ответов сверх JSON.
- Модерация маркетплейса.

## Decisions

**D1. Архитектура исполнения: main process + workers + очереди + WebSocket.**

Backend работает в двух режимах процесса из одной кодовой базы NestJS:

- **Main process** (API-сервер): принимает HTTP/WebSocket-запросы, CRUD-операции, создаёт документ `Run` со статусом `pending`, помещает job в очередь **BullMQ** (Redis), отдаёт клиенту WebSocket-канал по `runId` для real-time событий.
- **Worker process** (исполнитель): забирает job из очереди, исполняет шаги последовательно, после каждого шага сохраняет промежуточное состояние в документ `Run` (результат, статус, тайминги) и публикует события в **Redis pub/sub**. Main process подписан на pub/sub-канал `run:<runId>` и ретранслирует события в WebSocket-клиента.

Статусы `Run`: `pending` → `running` → `waiting_input` (страница шага) → `completed` | `failed` | `cancelled`. Промежуточные состояния сохраняются в БД, поэтому обрыв WebSocket-соединения не теряет прогресс: клиент переподключается и получает текущий статус + накопленные результаты шагов.

События через **WebSocket** (двунаправленный канал): сервер шлёт `step:start`, `step:done`, `page:required`, `run:done`, `run:error`; клиент шлёт `page:submit` (данные формы), `run:cancel`. Выбор WebSocket вместо SSE обоснован: двунаправленность позволяет отправлять данные страниц шага и команды отмены без дополнительных POST-запросов; reconnect по `runId` подтягивает пропущенные события из журнала.

Очереди (BullMQ) дают: сохранение промежуточных состояний (worker падает → job перезапускается с последнего сохранённого шага), статусы работы сценария (`pending`, `running`, etc.), горизонтальное масштабирование воркеров, приоритизацию, retry с backoff, ограничение конкурентности.

**D2. OpenAPI-импорт: парсим на backend, храним снимок спецификации + нормализованные endpoints.**
Библиотека `@readme/openapi-parser` (или `@apidevtools/swagger-parser`) для валидации/dereference. Из каждой операции извлекаем: метод, путь, summary, параметры (path/query/header/body) и схему ответа с примерами — ровно те данные, на которых построен маппинг в дизайне (`ENDPOINT_SCHEMA`: `in[]`, `out[]` с `ex`). Обновление спецификации — новый снимок + diff по `method+path`; удалённые из спеки endpoints помечаем `deprecated`, но не удаляем (требование «существующие карточки продолжат работать»).

**D3. Модель шага — дискриминированный union в `packages/shared`.**
`Step = ApiStep | ScenarioStep | DelayStep | FileStep | PeriodicStep` с общими полями (`title`, `mappings`, `consts`, `page?`). Маппинг входа: `{ [inputKey]: 'user' | 'const' | 's<idx>:<outputKey>' }` — токен-формат из прототипа сохраняем, он компактен и сериализуем. Подстановка шаблонов `{{token}}` в константах (например `Bearer {{access_token}}`) — на этапе резолва входов. Валидация циклов вложенных сценариев — DFS по графу ссылок при сохранении шага и при исполнении (guard, как в прототипе).

**D4. Файлы: S3-совместимое хранилище (MinIO), порог размера для single/chunked.**
Лимит single-upload — **10 МБ**: файлы ≤ 10 МБ грузятся одним запросом `multipart/form-data` напрямую. Файлы > 10 МБ грузятся чанками (5–10 МБ) через backend-эндпоинты `upload/init | part | complete | abort`, состояние частей хранится в Mongo — pause/resume и восстановление после обрыва (докачка с последней подтверждённой части). Порог настраивается через env (`FILE_SINGLE_UPLOAD_MAX_MB`, по умолчанию 10). После загрузки backend вызывает целевой endpoint провайдера согласно режиму шага (`multipart/form-data` целиком или чанки `application/octet-stream`).

**D5. Frontend: Nuxt 3, страницы 1:1 к экранам прототипа, Pinia для стейта редакторов.**
Роуты: `/` (маркетплейс), `/cards/:id` (обзор/запуск/endpoints), `/my/scenarios`, `/my/scenarios/:id/edit`, `/my/apps`, `/my/apps/:id`, `/my/apps/new`, `/my/apps/:id/update`, `/profile`. Редактор шагов и редактор страниц — самые «толстые» компоненты; их состояние (draft шага, маппинги, draft страницы) живёт в Pinia-сторах, персист — явными save-вызовами API. Rich-text описания — Tiptap (лёгкий, headless); в прототипе хватает B/I/U/H1-H3/code.

**D6. Auth — Yandex OAuth (Yandex ID), JWT-сессии в httpOnly cookie.**
Регистрация и вход по email/паролю заменены на OAuth-авторизацию через Яндекс. Flow: пользователь нажимает «Войти через Яндекс» → redirect на `https://oauth.yandex.ru/authorize` → callback с `code` на backend → backend обменивает `code` на access-токен Yandex, запрашивает профиль (`https://login.yandex.ru/info`) → создаёт/обновляет пользователя (email, ФИО, аватар из профиля Яндекса) → выпускает собственный JWT (access+refresh) в httpOnly cookie. Модуль NestJS `auth` + guards. Пароли не хранятся. Магазин публичен только на чтение каталога; запуск сценариев требует аккаунта. Настройки OAuth (`YANDEX_CLIENT_ID`, `YANDEX_CLIENT_SECRET`, `YANDEX_REDIRECT_URI`) — в env.

**D7. Категории — статический справочник в shared с подкатегориями.**
Полный список из прототипа (17 категорий + подкатегории). Счётчики дерева считаются агрегацией Mongo на запросе каталога. Выносить в БД-справочник с админкой — после MVP.

**D8. Backend отдаёт собственную OpenAPI-схему (`@nestjs/swagger`)** — питает существующий `gen:types`; DTO аннотируются сразу.

**D9. Observability — Yandex Metrica и сбор логов (опционально).**
Yandex Metrica подключается на frontend (счётчик `ym(...)`), включается env-переменной `YANDEX_METRIKA_ID` — при отсутствии переменной скрипт не загружается. Сбор логов backend (structured logging) — через интеграцию с системой сбора логов (Monium), включается env-переменной `LOG_COLLECTOR_URL` / `MONIUM_ENABLED`; при отсутствии логи пишутся только в stdout/stderr. Обе интеграции — behind feature flags, MVP работает без них.

## Risks / Trade-offs

- [Произвольные OpenAPI-спеки разнообразнее демо-данных: `$ref`-циклы, oneOf, отсутствие примеров] → dereference парсером, деградация до «поле без примера»; MVP поддерживает JSON body + примитивные параметры, остальное помечаем «не поддерживается в конструкторе».
- [SSRF: backend делает запросы по URL из пользовательских спецификаций] → allowlist схем (https), запрет приватных IP-диапазонов, таймауты и лимиты размера ответа на прокси-вызовах.
- [Хранение чужих API-ключей/токенов для шагов авторизации] → в MVP токен добывается шагом сценария (как в дизайне — `POST /auth/token` первым шагом), секреты-константы шифруются at rest; полноценный secret-manager — вне MVP.
- [Длинные запуски (поллинг до минут)] → очередь BullMQ изолирует воркеры от API-сервера; лимит длительности run (конфигурируемый, TTL job), интервал поллинга 1–600 с; при падении воркера job перезапускается с последнего сохранённого шага (промежуточные состояния в документе Run).
- [Резолв выходов шага зависит от реального ответа, а конструктор — от примеров из спеки] → расхождение схем: маппинг по ключу из фактического JSON-ответа, отсутствующий ключ → ошибка шага с внятной диагностикой.
- [Счётчик запусков и «популярные» уязвимы к накрутке] → в MVP принимаем; считаем только завершённые запуски аутентифицированных пользователей.

## Migration Plan

Greenfield — миграций нет. Порядок деплоя: docker-compose (Mongo + Redis + MinIO) → backend (main process) → worker process → frontend. Изменения схем данных до релиза — без миграций (пересоздание dev-базы). В production main и worker запускаются как отдельные процессы (или контейнеры) из одного образа с разными entrypoint-флагами.

## Open Questions

- Нужен ли публичный запуск сценариев без аккаунта (в дизайне шапка всегда с пользователем — принимаем «нет» для MVP)?
- Как владелец API ограничивает потребление (квоты/rate limits на сценарии поверх его API) — вне MVP, но повлияет на модель Run.
- Хранить ли полные ответы шагов в Run (удобно для отладки, но PII/объём) — предлагается TTL-коллекция с усечением тел до N КБ.
