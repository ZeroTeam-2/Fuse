## Context

Лимиты размера файлов заданы магическими числами в разрозненных местах:

| Значение | Где | Назначение |
| --- | --- | --- |
| `15 * 1024 * 1024` | `apps.controller.ts:33` (`SPEC_FILE_SIZE_LIMIT`) | лимит загрузки файла спеки (multer) |
| `10 * 1024 * 1024` | `Dropzone.vue:13` (`maxSize` default) | клиентская валидация |
| `15 * 1024 * 1024` | `new.vue:75` (`:max-size`) | клиентская валидация |
| `10 * 1024 * 1024` | `ssrf-guard.ts:6` (`MAX_RESPONSE_BYTES`) | лимит ответа при fetch спеки по URL |
| `5 * 1024 * 1024` | `uploads.service.ts:18` (`DEFAULT_PART_SIZE`) | размер part для chunked upload |
| `FILE_SINGLE_UPLOAD_MAX_MB` | env=15, сервис `?? 10`, nuxt=10 | порог single/chunked upload — **три разных дефолта** |
| `maxMb: 15` | `page-validator.ts:70` | дефолт файловой страницы (спека говорит 10) |

Root cause BSON-ошибки: `App.specSnapshot` (`@Prop` Mixed) хранит **развёрнутый** OpenAPI-документ инлайном. Он пишется в `create`, `createFromFile`, `applyReimport`, но **не читается нигде** (`grep specSnapshot` по `src` даёт только записи и seed `{}`). После dereference спека раздувается (все `$ref` инлайнятся), и документ `App` пробивает жёсткий потолок MongoDB 16 МБ → `ERR_OUT_OF_RANGE` (offset 17825792 = 17 МБ, внутренний буфер BSON).

Пользователь выбрал стратегию: **убрать `specSnapshot` из Mongo** (write-only поле). Это снимает раздувание и делает лимит 15 МБ безопасным без переноса в S3.

Инфраструктура конфига: NestJS `ConfigModule.forRoot({ isGlobal: true, validate: validateEnv })`, env читается zod-схемой `env.schema.ts`. Фронт — Nuxt `runtimeConfig.public`.

## Goals / Non-Goals

**Goals:**
- Устранить BSON-ошибку, убрав неиспользуемый `specSnapshot` из документа `App`.
- Единый источник правды для лимитов размера файлов: env-переменные → один конфиг-модуль → чтение по всему коду.
- Один канонический дефолт на каждый лимит (env ↔ сервис ↔ nuxt совпадают).
- Публичные лимиты доступны фронту через `runtimeConfig.public`; `Dropzone`/`new.vue` не хардкодят.
- Лимит спеки ограничен вместимостью документа хранилища.

**Non-Goals:**
- Перенос снапшота спеки в S3/MinIO (отклонён в пользу удаления поля).
- Chunked-upload флоу спеки (спека грузится одним запросом; общий upload-флоу для больших файлов уже есть).
- Миграция существующих документов `App` (поле опционально, старые читаются как есть).
- Пересмотр самих значений лимитов сверх того, что нужно, чтобы влезть в потолок хранилища.

## Decisions

### 1. Убрать `specSnapshot`, а не переносить его
Поле write-only. Удаляем `@Prop specSnapshot` из `app.schema.ts`, поле `specSnapshot` из результата `openapi-parser` (`ParsedSpec`) и его запись в `apps.service` (3 места) и `seed.ts`. Оставляем `@Prop` опциональным чтением не нужно — просто перестаём писать; схема без поля не ломает чтение старых документов (Mongoose игнорирует неописанные поля).
- _Альтернатива (S3):_ больше кода, новый failure-mode (рассинхрон Mongo↔S3), ради данных, которые никто не читает. Отклонено.

### 2. Один конфиг-модуль лимитов на бэкенде
Расширяем `env.schema.ts`:
- `SPEC_FILE_MAX_MB` (дефолт `15`) — лимит файла спеки.
- `SPEC_URL_FETCH_MAX_MB` (дефолт `10`) — лимит ответа fetch по URL (бывш. `MAX_RESPONSE_BYTES`).
- `UPLOAD_PART_SIZE_MB` (дефолт `5`) — размер part (бывш. `DEFAULT_PART_SIZE`).
- `FILE_SINGLE_UPLOAD_MAX_MB` — оставляем, но сводим дефолт к **одному** значению везде.

Добавляем `config/file-limits.config.ts` — NestJS `registerAs("fileLimits", ...)` (или простой провайдер), отдающий значения уже в байтах (`mb * 1024 * 1024`). Контроллер/guard/сервис инжектят его вместо литералов. `SPEC_FILE_SIZE_LIMIT` в контроллере, `MAX_RESPONSE_BYTES` в ssrf-guard, `DEFAULT_PART_SIZE` в uploads → читаются отсюда.
- _Альтернатива (константы в shared-пакете):_ не проходят через env, оператор не переопределит. Конфиг лучше для «оператор меняет из одного места».

### 3. Единый дефолт `FILE_SINGLE_UPLOAD_MAX_MB`
Сейчас 15/10/10. Приводим к одному значению (**10**, как в сервисе и nuxt — консервативный порог включения chunked upload); правим дефолт в `env.schema.ts` и убираем фолбэк-расхождение в `uploads.service` (`?? 10` становится сонаправлен env-дефолту).

### 4. Публичные лимиты на фронт
В `nuxt.config.ts` `runtimeConfig.public` добавляем `specFileMaxMb` (из `NUXT_PUBLIC_SPEC_FILE_MAX_MB`) и оставляем/выравниваем `fileSingleUploadMaxMb`. `Dropzone.vue` перестаёт хардкодить `maxSize` дефолт и подпись `{{ maxSize / (1024*1024) }}` — принимает МБ/байты сверху; `new.vue` передаёт `useRuntimeConfig().public.specFileMaxMb` вместо `15 * 1024 * 1024`.
- Значения env фронта и бэка задаются оператором согласованно; расхождение допустимо (клиент — UX-подсказка, бэк — жёсткая граница), но дефолты держим равными.

### 5. Лимит спеки ограничен хранилищем
После удаления `specSnapshot` документ `App` содержит только endpoints+схемы — на порядок меньше исходной спеки, 15 МБ файла безопасны. Дефолт `SPEC_FILE_MAX_MB=15` сохраняем (цель пользователя достигнута). Фиксируем инвариант в спеке: лимит не должен задаваться значением, которое хранилище не примет.

## Risks / Trade-offs

- **[Удаление specSnapshot ломает будущего потребителя]** → Сейчас потребителей нет (проверено grep). Если понадобится diff «файл↔файл» при reimport из файла — это отдельная фича; reimport по URL перечитывает спеку заново и в снапшоте не нуждается.
- **[Крупная спека всё ещё может раздуть endpoints[]]** → Теоретически спека с тысячами endpoints даст большой (но не гигантский) массив; это на порядки меньше развёрнутого документа. При необходимости — отдельный лимит на число endpoints, вне рамок этой задачи.
- **[Рассинхрон env фронта и бэка]** → Клиентский лимит — только UX; жёсткая граница на бэке (multer `limits.fileSize` + валидация) остаётся источником истины. Дефолты держим равными, чтобы не путать.
- **[Смена дефолта FILE_SINGLE_UPLOAD_MAX_MB 15→10]** → Тест `upload-mode.spec.ts` завязан на конкретные значения; правим тест на чтение из конфига/ожидаемый дефолт.

## Migration Plan

1. Расширить `env.schema.ts`, добавить `config/file-limits.config.ts`.
2. Перевести контроллер/guard/uploads на конфиг, убрать литералы.
3. Убрать `specSnapshot` (schema, parser, service ×3, seed).
4. Nuxt `runtimeConfig.public` + `Dropzone`/`new.vue` на конфиг.
5. Обновить `page-validator` дефолт из конфига (или общего значения) и `upload-mode.spec.ts`.
6. Прогнать бэкенд-тесты и `verify` флоу импорта файла спеки.

Rollback: изменения аддитивны в конфиге и удаляют write-only поле — откат = revert коммита; данных не теряем (старые `App.specSnapshot` остаются в БД нетронутыми, просто перестают писаться).

## Open Questions

- Значение единого дефолта `FILE_SINGLE_UPLOAD_MAX_MB` — берём **10** (текущее поведение сервиса/фронта). Если продукт хочет 15 — меняется одна env-переменная.
- Нужен ли `page-validator` дефолт (`maxMb`) читать из рантайм-конфига бэкенда или достаточно общей константы МБ — при apply выберем минимально инвазивный вариант (валидатор — чистая функция без DI, вероятно общая константа).
