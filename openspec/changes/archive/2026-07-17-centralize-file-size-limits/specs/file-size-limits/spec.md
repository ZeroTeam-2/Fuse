## ADDED Requirements

### Requirement: Единый источник правды для лимитов размера файлов

Система SHALL хранить все лимиты размера загружаемых файлов в конфигурации (env-переменные и производный от них конфиг-модуль), а не в виде магических чисел в коде. Код бэкенда и фронтенда MUST читать лимиты из этого единого источника; дублирующие захардкоженные значения (в контроллерах, компонентах, guard-ах, сервисах) MUST быть устранены.

Каждый лимит MUST иметь ровно один канонический дефолт: значение по умолчанию в env-схеме, фолбэк в сервисе и значение в `runtimeConfig` фронтенда MUST совпадать.

#### Scenario: Изменение лимита из одного места

- **WHEN** оператор меняет env-переменную лимита (например, `SPEC_FILE_MAX_MB`) и перезапускает приложение
- **THEN** новый лимит применяется и на бэкенде (валидация загрузки), и на фронтенде (валидация в дропзоне и подпись «макс. N МБ») без правок кода

#### Scenario: Отсутствие магических чисел

- **WHEN** разработчик ищет захардкоженные размеры файлов (`* 1024 * 1024`) в контроллерах, компонентах загрузки, SSRF-guard и uploads-сервисе
- **THEN** значения лимитов приходят из конфига, а не заданы литералами в этих местах

### Requirement: Лимит загрузки спецификации ограничен вместимостью хранилища

Система MUST ограничивать максимальный размер загружаемого файла спецификации значением, при котором результат импорта гарантированно помещается в документ хранилища (жёсткий потолок MongoDB — 16 МБ на документ), с запасом на разворачивание `$ref`. Лимит MUST браться из конфигурации и MUST NOT задаваться значением, которое хранилище физически не способно принять.

#### Scenario: Загрузка спеки в пределах лимита

- **WHEN** пользователь загружает файл спецификации размером в пределах настроенного лимита
- **THEN** система разбирает спеку и сохраняет приложение без ошибок сериализации хранилища

#### Scenario: Загрузка спеки сверх лимита

- **WHEN** пользователь загружает файл спецификации больше настроенного лимита
- **THEN** система отклоняет загрузку с понятной ошибкой о превышении размера, а не падает с внутренней ошибкой сериализации

### Requirement: Публичные лимиты доступны фронтенду

Система SHALL публиковать лимиты, влияющие на клиентскую валидацию (лимит файла спецификации, лимит single-upload), через публичную runtime-конфигурацию фронтенда. Компоненты загрузки MUST читать лимит и его отображение («макс. N МБ») из конфига, а не из литералов.

#### Scenario: Дропзона использует лимит из конфига

- **WHEN** страница создания приложения рендерит дропзону загрузки спеки
- **THEN** максимальный размер и подпись «макс. N МБ» соответствуют настроенному публичному лимиту, а клиентская проверка размера использует то же значение, что и бэкенд
