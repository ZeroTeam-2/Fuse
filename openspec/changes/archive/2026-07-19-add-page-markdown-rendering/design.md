# Design: Блок «Текст» — рендеринг Markdown

## Context

Блок `paragraph` — единственный блок категории «Отображение» на страницах шагов. Рантайм-рендер живёт в одном компоненте [PageRunner.vue](../../../apps/frontend/components/run/PageRunner.vue): статический текст блока либо значение из `resolved` (данные пройденного шага, разрешённые воркером) выводится как `{{ displayValue(block) }}` c `whitespace-pre-wrap`. Из внешних API текст часто приходит как Markdown, причём с литеральными последовательностями `\n` вместо переводов строк.

В проекте уже есть прецедент безопасного `v-html`: описания маркетплейса рендерятся через allowlist-санитайзер [sanitizeRichText.ts](../../../apps/frontend/utils/sanitizeRichText.ts) (режет все атрибуты и теги вне списка, выкусывает содержимое `script/style/iframe`). Markdown-парсера в зависимостях нет.

## Goals / Non-Goals

**Goals:**
- Настройка формата у блока `paragraph` в инспекторе конструктора: «Обычный текст» (по умолчанию) или «Markdown».
- В режиме Markdown — рендер содержимого блока на странице исполнения: и статического текста, и разрешённого значения из `resolved`.
- Нормализация экранированных последовательностей в строках из API до парсинга.
- Безопасность: весь HTML после парсера проходит санитайзер; поддержать безопасные ссылки.
- Обратная совместимость: существующие страницы без поля `format` рендерятся как раньше.

**Non-Goals:**
- WYSIWYG-редактирование Markdown в конструкторе — автор редактирует исходный текст как раньше (contenteditable в `PageParagraph.vue`).
- Markdown в других местах (лейблы блоков, заголовки страниц, результаты шагов вне страниц, KeyValueGrid).
- Изменения модели данных, backend или worker.

## Decisions

### 0a. Переименование — только UI-имя, тип не трогаем
В палитре конструктора блок называется «Текст» вместо «Абзац» (одна строка в `PALETTE` в `PageEditor.vue`). Тип `paragraph` в модели, спеках данных и валидаторе остаётся: переименование типа потянуло бы миграцию сохранённых сценариев и правки backend/shared ради чисто косметического эффекта. Основной спек `step-pages` ссылается на тип `paragraph`, а не на русское имя, — конфликт не возникает.

### 0. Формат — явная настройка блока, по умолчанию «Обычный текст»
`PageBlock` получает опциональное поле `format?: "text" | "markdown"`; отсутствие поля означает `"text"`. Инспектор конструктора показывает переключатель формата только для блока `paragraph` (рядом с существующей секцией «Текст» в `PageEditor.vue`). Альтернатива — авто-детект Markdown по содержимому — отвергнута: эвристика непредсказуема (звёздочки и `#` встречаются в обычном тексте), а данные из API приезжают только на рантайме. Явный выбор автора надёжнее и снимает риск изменения вида существующих страниц. Серверный `page-validator` неизвестные поля не режет — правки бэкенда не нужны; строгая валидация значения поля не требуется (неизвестное значение читается как `"text"`).

### 1. Парсер: `marked`
Берём `marked` — компактный, без DOM-зависимостей, работает в SSR (Nuxt) и браузере. Альтернативы: `markdown-it` (тяжелее, плагинная система не нужна для одного блока); самописный парсер (жирный/списки/заголовки быстро разрастается и остаётся неполным). Опция `breaks: true` — одиночный перевод строки становится `<br>`, что соответствует нынешнему `whitespace-pre-wrap`-поведению и ожиданиям авторов.

### 2. Санитизация: существующий `sanitizeRichText` + ссылки
HTML из `marked` прогоняется через `sanitizeRichText` — не пишем второй санитайзер и не тащим DOMPurify. Санитайзер расширяется тегом `a`: `href` сохраняется только со схемами `http/https/mailto` (протокол-относительные и `javascript:` — отбрасываются вместе с атрибутом), на выходе принудительно `rel="noopener noreferrer" target="_blank"`. Это единственное отступление от правила «резать все атрибуты», локализованное в одном месте. Сырые HTML-теги в исходном Markdown до allowlist не доживают — parser передаёт их как текст либо санитайзер их режет.

### 3. Нормализация экранированных последовательностей
Перед парсингом строка из `resolved` нормализуется: литеральные `\n`, `\t`, `\"`, `\\` → реальные символы. Нормализация применяется только если строка содержит такие последовательности (эвристика не нужна — замена идемпотентна для строк без них; `\\` обрабатывается первым, чтобы не породить ложные `\n`). Статический текст блока автор пишет руками — его тоже прогоняем той же функцией: хуже не станет, а поведение единообразно.

### 4. Место: утилита + условная ветка в PageRunner
Новая утилита `renderMarkdown(text: string): string` в `apps/frontend/utils/` (нормализация → marked → sanitizeRichText). В `PageRunner.vue` ветка `paragraph` раздваивается по `block.format`: при `"markdown"` — `v-html="renderMarkdown(displayValue(block))"` с typography-классами для вложенных элементов (заголовки, списки, код, ссылки) в духе существующего DS; иначе — прежняя интерполяция с `whitespace-pre-wrap`.

## Risks / Trade-offs

- [XSS через результат внешнего API] → единая точка входа `renderMarkdown` всегда завершается санитайзером; тест на `javascript:`-ссылки, `<script>` внутри Markdown, event-атрибуты.
- [Автор ждёт Markdown, но забыл переключить формат] → блок покажет сырой текст; переключатель в инспекторе рядом с полем «Текст», подписан понятно. Принимаем: явный выбор дешевле ложных срабатываний авто-детекта.
- [`breaks: true` отличается от строгого CommonMark] → осознанно: сохраняет привычное поведение переводов строк.
- [Новая зависимость `marked`] → маленькая, zero-deps, широко используемая; фиксируем мажорную версию.
