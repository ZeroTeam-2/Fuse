## Why

Текущий редактор страниц (`PageEditor.vue`) — форма-модалка с тремя фиксированными типами страницы (`fields` / `file` / `text`). Дизайн в `assets/Fuse Design System` (кнопка «Открыть редактор» в настройке шага → `FormBuilder.jsx`) заменяет её принципиально другим инструментом: полноэкранным визуальным конструктором с палитрой UIKit-элементов, drag & drop на сетку из 4 колонок и привязкой каждого элемента к данным сценария. Текущая модель `StepPage` этого не выражает: одна страница = один тип, поля — плоский список без раскладки, а «отображение результата» ограничено статическим текстом и не умеет показывать значения из **уже пройденных шагов**. Пользователь явно подтвердил: старый редактор deprecated, его надо заменить.

## What Changes

- **Новая модель страницы шага (`StepPage`).** **BREAKING**: единый union `fields | file | text` заменяется композиционной раскладкой — страница держит `title` и список строк (`rows`), каждая строка — набор блоков на сетке из 4 колонок, каждый блок имеет тип, ширину (`span` 1–4) и опциональную привязку к данным. Прежние три типа поглощаются набором блоков (поле → `input`, файл → `dropzone`, текст → `paragraph`), так что выразительность не теряется, а растёт.
- **Палитра UIKit-элементов в двух категориях.** Блоки делятся на **Ввод** (`input`, `select`, `dropzone`, `richtext` — заполняет пользователь) и **Отображение** (`paragraph` — показывает данные). Категория элемента определяет, к какому концу данных он привязывается.
- **Полноэкранный конструктор вместо модалки.** Слева — палитра (перетащить на страницу), в центре — «лист страницы» с drag & drop создания строк и вставки в строки, перетаскиванием блоков для переупорядочивания и ручкой изменения ширины (снап 1–4 колонки), справа — инспектор свойств выделенного блока. Открывается по кнопке «Открыть редактор» из панели настройки шага.
- **Привязка Ввода → входной параметр шага.** Блок ввода привязывается к ручному значению **этого** шага (параметр `mappings[key] = "user"` или операнд условия `filter:key`). Это замена нынешнего `PageField.target`: то, что пользователь ввёл в блок, подставляется во вход шага.
- **Привязка Отображения → данные пройденных шагов.** **NEW**: блок отображения привязывается к полю результата **предыдущего** (уже исполненного) шага (`s{idx}:{outKey}`). Страница показывается до запуска шага, поэтому во время выполнения значение подставляется из результата шага-источника. Это ключевая новая способность — «привязывать элементы к данным, которые имеются на уже пройденных шагах».
- **Рантайм рендерит раскладку и подставляет данные.** Панель запуска (`RunPanel`/`PlaygroundPanel`) рендерит строки/сетку обобщённо: блоки ввода собирают значения, блоки отображения показывают подставленные данные. Воркер при `page:required` разрешает значения для блоков отображения из результатов предыдущих шагов и кладёт их в payload события.
- **Инспектор + сохранение.** Инспектор редактирует подпись, placeholder, текст, ширину и привязку; счётчик элементов и «Готово» в шапке сохраняют страницу в состав шага. Валидация страницы на бэкенде переписывается под новую модель (допустимые типы блоков, `span` 1–4, непустые строки, существующие привязки).
- **Миграция существующих страниц.** Ранее сохранённые `fields`/`file`/`text` страницы конвертируются в новую модель (best-effort: поля → строки блоков `input`, файл → `dropzone`, текст → `paragraph`), чтобы засеянные и уже настроенные сценарии не сломались.

### Modified Capabilities
- `step-pages`: модель страницы становится композиционной раскладкой из блоков на сетке 4 колонок с per-блочной привязкой; редактор — полноэкранный конструктор с палитрой, drag & drop и инспектором; типы блоков делятся на Ввод и Отображение; привязка Ввода заменяет `PageField.target`, привязка Отображения — новая (к выходам пройденных шагов); валидация и live-поведение переопределяются.
- `scenario-execution`: при показе страницы шага движок разрешает значения блоков отображения из результатов предыдущих шагов и доставляет их в payload `page:required`; панель запуска рендерит раскладку страницы обобщённо (ввод собирает значения, отображение показывает подставленные данные).

## Impact

**Shared**
- `packages/shared/src/types/index.ts` — новая модель `StepPage` (строки/блоки/`span`/`binding`), типы блоков и категорий; `PageField` заменяется/переопределяется под блок; возможный тип для payload `page:required` с разрешёнными данными отображения.
- `packages/shared/src/events/index.ts` — payload события `page:required` расширяется картой разрешённых значений блоков отображения.

**Frontend**
- `apps/frontend/components/scenario/PageEditor.vue` — полностью переписывается в полноэкранный конструктор (палитра, канвас-сетка, drag & drop, resize, инспектор) по образцу `FormBuilder.jsx`.
- `apps/frontend/pages/my/scenarios/[id]/edit.vue` — редактору передаются все шаги и их схемы (не только текущий шаг), чтобы инспектор предлагал выходы предыдущих шагов.
- `apps/frontend/components/scenario/StepConfig.vue` — кнопка/секция «Страница шага» под новую модель (счётчик элементов вместо «типа страницы»).
- `apps/frontend/components/RunPanel.vue`, `apps/frontend/components/PlaygroundPanel.vue` — обобщённый рендер раскладки страницы вместо трёх веток по типам; блоки отображения выводят подставленные данные.

**Backend**
- `apps/backend/src/scenarios/page-validator.ts` — валидация новой модели.
- `apps/backend/src/execution/manual-inputs.ts` — `pageFields`/`pageCovers`/`mapPageDataToLocalKeys` читают привязку блоков ввода вместо `PageField.target`.
- `apps/backend/src/execution/worker.service.ts` — при `page:required` разрешает данные блоков отображения из результатов предыдущих шагов и кладёт их в payload.
- Тесты: `test/page-validation.spec.ts`, `test/manual-inputs.spec.ts`, `test/worker-manual-inputs.spec.ts` — под новую модель; данные сидов/фикстур со старой моделью — мигрировать.
