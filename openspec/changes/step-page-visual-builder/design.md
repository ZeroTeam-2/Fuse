## Context

Страница шага — экран, который пользователь видит **перед** исполнением шага при запуске сценария. Сегодня это одна из трёх фиксированных форм (`StepPage = fields | file | text`), настраиваемых модалкой `PageEditor.vue`. Дизайн (`assets/Fuse Design System/ui_kits/fuse-app/FormBuilder.jsx`, открывается кнопкой из `ScenarioEditor.jsx → StepPanel`) заменяет её полноэкранным визуальным конструктором: палитра UIKit-элементов, «лист страницы» с сеткой из 6 колонок, drag & drop, ручка ширины, инспектор свойств и **per-блочная привязка к данным сценария**.

Текущая цепочка данных, которую затрагивает переход:
- **Модель**: `packages/shared/src/types/index.ts` — `StepPage`, `PageField`, `Step.page`.
- **Редактор**: `PageEditor.vue`, открывается из `StepConfig.vue` (`emit('edit-page')`) через `pages/my/scenarios/[id]/edit.vue` (`ScenarioPageEditor`), которому передаётся только `:step` и `:step-schema` текущего шага.
- **Рантайм**: `worker.service.ts::executeStep` шлёт `page:required` с `step.page`; `RunPanel.vue`/`PlaygroundPanel.vue` рендерят страницу тремя ветками по типу; ввод возвращается через `submitPage`.
- **Ручные входы**: `manual-inputs.ts` (`pageFields`/`pageCovers`/`mapPageDataToLocalKeys`) связывает поле страницы со значением шага по `PageField.target`; `source: "page"` исключает значение из общей формы запуска.
- **Валидация**: `scenarios/page-validator.ts` проверяет три типа.

Ограничения: проект предпродакшн, но удалённая dev-БД засеяна сценариями со старой моделью страницы — их нельзя молча ломать. Один и тот же список ручных входов строит и форму запуска, и проверку полноты в воркере — они не должны разъехаться.

## Goals / Non-Goals

**Goals:**
- Композиционная модель страницы: строки × блоки на сетке из 6 колонок, `span` 1–6, per-блочная привязка.
- Полноэкранный конструктор с палитрой (2 категории), drag & drop создания/переупорядочивания, resize и инспектором — по образцу `FormBuilder.jsx`, но на Vue и на реальных схемах шага.
- Привязка **Ввода** к ручному входу текущего шага (замена `PageField.target`).
- Привязка **Отображения** к выходу **уже пройденного** шага (`s{idx}:{outKey}`) с подстановкой реального значения в рантайме.
- Обобщённый рендер раскладки в панели запуска; воркер разрешает данные отображения и кладёт их в payload `page:required`.
- Миграция ранее сохранённых страниц в новую модель без потери смысла.

**Non-Goals:**
- Вложенные строки/колонки глубже одного уровня (сетка плоская: строка → блоки, как в референсе).
- Новые типы блоков сверх набора референса (`input`, `select`, `dropzone`, `richtext`, `paragraph`).
- Условная видимость блоков, вычисляемые выражения, стилизация блоков пользователем.
- Изменение механики ручного добора значений посреди выполнения (остаётся как есть).
- Привязка отображения к выходам **следующих** или **текущего** шага (страница показывается до шага — таких данных ещё нет).

## Decisions

### D1. Форма модели `StepPage`
Единый union заменяется одной формой раскладки:

```ts
type PageBlockType = "input" | "select" | "dropzone" | "richtext" | "paragraph";

interface PageBlock {
  id: string;
  type: PageBlockType;
  span: 1 | 2 | 3 | 4 | 5 | 6;
  label?: string;        // input | select | dropzone | richtext
  placeholder?: string;  // input | select
  text?: string;         // paragraph
  binding?: string;      // input-категория: ключ входа шага (inn | filter:status)
                         // display-категория: выход пройденного шага (s{idx}:{key})
}

interface PageRow { id: string; items: PageBlock[]; }

interface StepPage {
  title: string;
  rows: PageRow[];
}
```

Категория блока выводится из типа (`input/select/dropzone/richtext` → ввод; `paragraph` → отображение), как `META`/`CATS` в референсе, — отдельного поля категории в модели нет.

**Почему не сохранить union с новым вариантом `builder`.** Пользователь подтвердил, что старая модель deprecated; тянуть оба варианта — двойной рантайм-рендер, двойная валидация и двойная логика ручных входов навсегда. Чистая замена + разовая миграция дешевле в поддержке. Совместимость закрывается миграцией (D6), а не параллельным типом.

### D2. Привязка Ввода = входной параметр текущего шага (и она же назначает источник)
`PageBlock.binding` у блока-ввода указывает **входной параметр этого шага** (`inn`) либо операнд его условия фильтрации (`filter:status`). Инспектор наполняет список привязок **всеми** входами шага из схемы (`schemas[stepIndex].inputs`), а не только заранее помеченными «Ручной ввод»: пользователь не должен идти в панель параметров, чтобы сперва пометить вход ручным, — сама привязка это и объявляет.

Привязка **назначает источник значения**. На сохранении страницы (`edit.vue::savePage`) для каждого параметра, к которому привязан блок ввода, проставляется `mappings[key] = "user"` (для операнда условия — `filters[key].value.mode = "user"`). Прежняя константа/ссылка на другой шаг при этом **переопределяется**: привязать поле к параметру значит явно перенаправить его источник на страницу. Снятие привязки источник **не откатывает** (параметр остаётся `user`, его спросит форма запуска) — вернуть константу/ref можно в панели параметров шага. Так остаётся один источник правды — `mappings`/`filters`, — а бэкенд (перечисление ручных входов, резолвер) не меняется.

`manual-inputs.ts` читает блоки ввода вместо `PageField[]`:
- `pageInputBlocks()` → блоки-ввода страницы;
- `pageCovers(step, localKey)` → есть ли блок-ввод с `binding === localKey`;
- `mapPageDataToLocalKeys(step, data)` → ключом данных страницы становится `block.id`, значение кладётся под `binding` (или под собственный ключ, если привязки нет — как деградация).

Форма запуска по-прежнему не спрашивает значения, покрытые страницей (`source: "page"`).

**Почему источник назначается на сохранении фронтом, а не проверяется на рантайме.** Единственный источник правды о том, откуда берётся вход, — `mappings`/`filters`; их читают и форма запуска, и воркер. Если бы страница объявляла ручной ввод «в обход» `mappings`, логика «откуда значение» размазалась бы на два места. `savePage` уже владеет и шагом, и страницей — там и сводим привязки в источник, оставляя бэкенд нетронутым.

**Почему `block.id` как ключ данных страницы.** У блока нет пользовательского «ключа поля» (в отличие от старого `PageField.key`); `id` уникален в пределах страницы и стабилен при сохранении — им и адресуем ввод от рантайма к воркеру.

### D3. Привязка Отображения = выход пройденного шага
Блок `paragraph` с `binding = "s{idx}:{outKey}"` показывает значение поля результата шага `idx`. Инспектор наполняет список только шагами **до** текущего, у которых есть выходы (та же логика, что `prevStepOptions` + `fieldsForStep` в `StepConfig.vue`). Значение подставляется **в рантайме**, а не в конструкторе.

**Почему предыдущие шаги, а не текущий.** Страница рендерится до вызова шага (`executeStep` шлёт `page:required` перед HTTP-вызовом), поэтому результата **текущего** шага ещё нет. Референс (`FormBuilder.jsx`) прокидывает в `outputs` выходы текущего шага — это артефакт мока; требование пользователя явно про «уже пройденные шаги». Берём выходы предшественников.

### D4. Разрешение данных отображения в воркере
`executeStep` перед отправкой `page:required` собирает карту `{ blockId → value }` для блоков отображения: по `binding = s{idx}:{key}` читает результат шага-источника из уже накопленного контекста запуска (те же данные, что видит `mapping-resolver` для `ref`-маппингов), учитывая массивные выходы так же, как резолвер (берётся одно значение/поле). Карта уходит в `payload.resolved` события `page:required`. Панель запуска рендерит блок отображения этим значением; при отсутствии — плейсхолдер/пусто.

**Почему разрешать на сервере, а не слать сырые ссылки на клиент.** Результаты предыдущих шагов живут на бэкенде (в состоянии запуска), клиент их целиком не имеет и не должен. Сервер уже умеет читать выход шага по `s{idx}:{key}` — переиспользуем его, а на клиент отдаём только готовые к показу значения.

### D5. Редактор: полноэкранный оверлей на Vue
`PageEditor.vue` переписывается по структуре `FormBuilder.jsx`: `fixed inset-0` оверлей, шапка (назад/заголовок/счётчик/Отмена/Готово), палитра слева (`input/select/dropzone/richtext` в «Ввод», `paragraph` в «Отображение»), канвас-лист по центру (строки, сетка `grid-cols-6`, `GapZone` между строками, вставка внутрь строки по позиции курсора, ручка ресайза со снапом 1–6), инспектор справа (привязка, подпись/placeholder/текст, ширина, удаление). Локальное состояние — реактивная копия `rows`; сохранение эмитит новую `StepPage`.

Интеграция: `edit.vue` передаёт редактору **все** `steps` и `schemas` (не только текущий шаг) — инспектору нужны и входы текущего шага (для Ввода), и выходы предыдущих (для Отображения). `StepConfig.vue` секция «Страница шага» показывает счётчик элементов (`сумма items`) вместо «типа страницы».

**Почему свой D&D, а не библиотека.** Референс реализует D&D нативными HTML5-событиями и pointer-capture для ресайза без зависимостей; переносим этот же подход на Vue, чтобы не тянуть новый пакет ради одного экрана и сохранить поведение 1:1.

### D6. Миграция существующих страниц
Разовый конвертер `StepPage(old) → StepPage(new)`:
- `fields`: `title` сохраняется; каждое поле → строка с блоком `input` (`label`, `placeholder`, `span: 4`, `binding = field.target`); `hint` → ведущий `paragraph`; `buttonText` отбрасывается (Q1 — кнопка в рантайме).
- `file`: `title`; блок `dropzone` (`label = hint`, `span: 4`); `accept`/`maxMb` отбрасываются, рантайм применяет дефолты (Q2).
- `text`: `title`; блок `paragraph` (`text = body`).

Конвертер применяется best-effort при чтении (нормализация в шейпе `Step`) либо разовой миграцией сидов — см. Migration Plan.

## Risks / Trade-offs

- **[Потеря атрибутов старой модели при миграции: `buttonText`, `accept`, `maxMb`.]** → Осознанно отбрасываются (Q1/Q2): `title`/`hint`/поля/текст переносятся точно, кнопка и лимиты файла уходят в дефолты рантайма. Ни один старый сценарий не должен стать невалидным после конверсии — покрыть тестом миграции.
- **[Рассинхрон привязок при переупорядочивании/удалении шагов.]** Привязка Отображения `s{idx}:{key}` адресует шаг по индексу; удаление/перестановка шагов делает индекс несогласованным. → Валидация страницы на бэкенде отсекает битые привязки (несуществующий шаг/поле) в предупреждение, рантайм показывает пусто, а не падает; в идеале конструктор чинит индексы при drag-reorder шагов (как это уже делается для `ref`-маппингов — свериться с текущим поведением).
- **[Обобщённый рантайм-рендер сложнее трёх статических веток.]** → Рендер строк/сетки изолируется в одном компоненте, переиспользуемом `RunPanel` и `PlaygroundPanel`; блоки ввода мапятся на существующий `Field`/`Input`, отображение — на текст.
- **[Сложность D&D → баги каретки/drop.]** Референс уже решил тонкие места (модульная `GapZone` во избежание ремоунта, `applyDrop` по явному таргету, seed текста `contentEditable`) — переносим вместе с комментариями, не переизобретаем.
- **[Расширение payload `page:required` — контракт события.]** Добавляется `resolved`; старые клиенты его игнорируют, новый рантайм без него показывает пусто — аддитивно, не ломающе для транспорта.

## Migration Plan

1. Ввести новую модель `StepPage` и типы блоков в `@fuse/shared`; собрать пакет.
2. Бэкенд: новый `page-validator`, переписанные `manual-inputs`, разрешение данных отображения в воркере, расширение payload `page:required`.
3. Конвертер старой модели: применить к сид-данным (`scripts`/seed) и, при необходимости, добавить нормализацию на чтении шага, чтобы уцелевшие старые записи в БД поднимались в новую форму.
4. Фронтенд: переписать `PageEditor.vue`, обновить `edit.vue` (передача всех шагов/схем), `StepConfig.vue` (счётчик), обобщить рендер в `RunPanel`/`PlaygroundPanel`.
5. Обновить тесты (`page-validation`, `manual-inputs`, `worker-manual-inputs`) и фикстуры.
6. Ручная проверка end-to-end: собрать страницу с блоком ввода (привязка к параметру) и блоком отображения (привязка к выходу шага 1), запустить сценарий, убедиться, что ввод уходит в шаг, а отображение показывает данные шага 1.

**Rollback:** изменения фронтенд-редактора и рантайм-рендера обратимы откатом коммита; риск только в конвертированных сид-данных — сиды перегенерируются, боевых данных нет.

## Resolved Decisions

- **Q1 → Решено: кнопки нет в модели.** `buttonText` в новую модель не переносится. Кнопка отправки страницы — часть рантайма запуска: панель рисует стандартную «Продолжить». Миграция `fields`/`file` отбрасывает `buttonText`.
- **Q2 → Решено: дефолты сейчас, поля блока — позже.** `accept`/`maxMb` на блоке `dropzone` в первой итерации не хранятся и не редактируются: рантайм применяет дефолты (напр. лимит 10 МБ, форматы без ограничения). Поля блока и их правка в инспекторе — отдельным инкрементом. Миграция `file` задаёт дефолты, не перенося старые `accept`/`maxMb`.
- **Q3 → Решено: минимум-валидация, выровненная под `ref`-маппинги.** Автопочинку индексов привязок Отображения при перестановке/удалении шагов не вводим отдельно: поведение выравнивается по тому, как сейчас ведут себя `ref`-маппинги при reorder в конструкторе. Битые привязки (несуществующий шаг/поле) отсекаются валидацией страницы в предупреждение, а рантайм показывает блок пустым и не падает.
