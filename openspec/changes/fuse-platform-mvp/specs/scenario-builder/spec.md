# scenario-builder

## ADDED Requirements

### Requirement: Карточка сценария
Система SHALL позволять создать сценарий (use-case карточку) с названием, обложкой (загрузка/drag-and-drop изображения), rich-text описанием для маркетплейса (жирный, курсив, подчёркивание, заголовки, код) и категорией из справочника с поиском. Tagline карточки MUST выводиться из первых ~90 символов текстового описания. Сценарий — самостоятельная сущность и MAY собирать шаги из разных API.

#### Scenario: Создание сценария
- **WHEN** пользователь заполняет детали и нажимает «Создать сценарий»
- **THEN** система создаёт неопубликованный сценарий без шагов и открывает редактор шагов

#### Scenario: Редактирование метаданных
- **WHEN** пользователь меняет обложку, описание или категорию на вкладке «Основная»
- **THEN** изменения сохраняются и отражаются в карточке маркетплейса после публикации

### Requirement: Типы шагов
Редактор шагов SHALL поддерживать пять типов шага:
1. **Endpoint API** — вызов метода выбранного приложения (выбор приложения с поиском → выбор endpoint);
2. **Другой сценарий** — вложение готового сценария (его входы → выходы); система MUST предотвращать циклические вложения;
3. **Задержка** — пауза на заданное число секунд перед следующим шагом (пресеты 1/3/5/10 сек и произвольное значение);
4. **Файл** — загрузка файла пользователем с режимами «разовая загрузка» (multipart/form-data) и «мультичанковая загрузка S3» (application/octet-stream, рекомендуемый), Content-Type MUST определяться режимом; опционально — endpoint опроса статуса обработки с интервалом и полем прогресса (в байтах);
5. **Периодический запрос** — опрос выбранного endpoint с интервалом (1–600 сек) и полем прогресса из схемы ответа.

#### Scenario: Добавление API-шага
- **WHEN** пользователь выбирает тип «Endpoint API», приложение и endpoint, задаёт название шага и нажимает «Добавить шаг»
- **THEN** шаг появляется в потоке с методом, путём и названием приложения

#### Scenario: Вложение сценария
- **WHEN** пользователь выбирает тип «Другой сценарий» и указывает сценарий
- **THEN** шаг принимает входы вложенного сценария и отдаёт его выходы; сценарии, приводящие к циклу, не предлагаются

#### Scenario: Настройка файлового шага
- **WHEN** пользователь выбирает режим «Мультичанковая загрузка S3» и включает опрос статуса
- **THEN** система выставляет Content-Type: application/octet-stream и требует выбрать endpoint статуса, интервал и поле прогресса

### Requirement: Маппинг данных между шагами
Для каждого входного поля шага система SHALL позволять выбрать источник: выход одного из предыдущих шагов, ввод пользователя или константа. Система MUST автоматически подставлять источник по совпадению ключа или метки с выходами предыдущих шагов. Заголовки авторизации MUST по умолчанию получать константу `Bearer {{access_token}}`. Панель шага SHALL показывать выходные поля endpoint с примерами значений. Шаг без входных данных MAY стоять первым в потоке.

#### Scenario: Автоподстановка источника
- **WHEN** пользователь добавляет шаг, у которого поле `id` совпадает с выходом предыдущего шага
- **THEN** система автоматически маппит поле на выход этого шага, оставляя возможность сменить источник вручную

### Requirement: Управление потоком шагов
Редактор SHALL позволять вставлять шаг в любую позицию, редактировать и удалять шаги, менять порядок drag-and-drop. Для мультисервисных сценариев система SHALL показывать индикатор числа задействованных API.

#### Scenario: Перестановка шагов
- **WHEN** пользователь перетаскивает шаг на новую позицию
- **THEN** порядок исполнения обновляется и сохраняется

### Requirement: Публикация сценария
Система SHALL позволять публиковать и скрывать сценарий. Публикация MUST требовать хотя бы один шаг. Опубликованный сценарий появляется в маркетплейсе, скрытый — виден только владельцу.

#### Scenario: Публикация без шагов
- **WHEN** пользователь пытается опубликовать сценарий без шагов
- **THEN** система блокирует публикацию с подсказкой «Добавьте хотя бы один шаг»
