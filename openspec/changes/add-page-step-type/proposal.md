# Страница как отдельный тип шага

## Why

Сегодня страница — это скрытая настройка любого шага (`BaseStep.page`), и показывается она всегда **до** исполнения шага-носителя. Из-за этого невозможно построить финальный экран: показать после последнего шага страницу с текстом ответа. Кроме того, наличие страницы не видно в потоке шагов — система непрозрачна: чтобы понять, где сценарий останавливается и ждёт пользователя, нужно открывать настройки каждого шага.

Страница как самостоятельный тип шага решает обе проблемы: её место в потоке явно видно и свободно выбирается (в том числе последним шагом — экран результата), а данные страницы становятся обычными выходами шага, доступными следующим шагам штатным маппингом «Из шага».

## What Changes

- Новый тип шага **«Страница»** (`type: "page"`) в палитре редактора: шаг несёт раскладку страницы (`StepPage`) как свою основную настройку и открывает тот же полноэкранный конструктор.
- **Семантика данных**: блоки ввода страницы-шага становятся **выходами** этого шага (ключ выхода — привязка блока либо его id); следующие шаги берут эти значения обычным маппингом `s{idx}:{key}`. Блоки отображения, как и раньше, привязываются к выходам предыдущих шагов (`s{idx}:{outKey}`).
- **Исполнение**: шаг-страница с блоками ввода приостанавливает запуск (`waiting_input` + `page:required`), собранные значения записываются результатом шага. Страница только с блоками отображения не блокирует поток: она публикуется для показа и остаётся финальным экраном, если стоит последним шагом.
- **BREAKING** (редактор): секция «Страница» удаляется из панели настройки остальных типов шагов — новые страницы создаются только шагом «Страница». Легаси-шаги со встроенной страницей продолжают исполняться по-старому (паттерн деградации как у типа `file`), редактор показывает у них пометку и позволяет только удалить встроенную страницу; автоконвертация в шаг-страницу — вне объёма изменения.
- Форма запуска не спрашивает значения, которые собирает страница-шаг (это её собственные выходы, а не `user`-маппинги других шагов).

## Capabilities

### New Capabilities

_нет — изменение перераспределяет существующие возможности._

### Modified Capabilities

- `step-pages`: страница перестаёт быть привязкой к произвольному шагу и становится содержимым шага типа «Страница»; привязка блока ввода теперь задаёт **ключ выхода** шага-страницы (а не входной параметр шага-носителя); привязка блоков отображения не меняется.
- `scenario-builder`: пятый тип шага «Страница» в палитре; его выходы (блоки ввода) доступны следующим шагам в маппинге «Из шага»; секция «Страница» исчезает из настройки остальных типов; легаси-шаги со встроенной страницей открываются без ошибок.
- `scenario-execution`: исполнение шага `page` (пауза/сабмит для ввода, неблокирующий показ для чисто-отображающих страниц, результат шага = собранные значения); финальная страница как экран результата; сохранение легаси-ветки исполнения встроенных страниц.

## Impact

- **Shared** (`packages/shared`): `StepType` + новый `PageStep` в union `Step`; хелперы схемы шага-страницы (выходы из блоков ввода); депрекация `BaseStep.page` (остаётся для чтения легаси-документов).
- **Backend**: `worker.service.ts` (ветка исполнения `page`-шага, легаси-ветка `step.page`), `manual-inputs.ts` (страница-шаг не даёт дескрипторов формы запуска; `pageCovers` — только для легаси), `scenarios.service.ts` (схема шага `page` для маппинга следующих шагов), валидация шагов, `seed-local.mts`.
- **Frontend**: `AddStepMenu`/`StepPicker` (новый тип), `StepConfig.vue` (конфиг page-шага = конструктор; удаление секции «Страница» у остальных), `PageEditor.vue` (инспектор привязок блока ввода → ключ выхода), `scenario-editor.ts` (модель шага, выходы page-шага в источниках маппинга), `RunPanel.vue`/`PageRunner.vue` (неблокирующие display-страницы, финальный экран), e2e-тесты.
- **Данные**: существующие сценарии со встроенными страницами не мигрируются — работают по легаси-ветке.
