## Context

Сквозной поток запуска сценария сегодня: `RunPanel.startRun()` → `POST /api/runs` → `ExecutionService.createRun` (документ `Run` в статусе `pending` + сообщение `{ runId }` в очередь) → `WorkerService` (тот же процесс, `sqs-consumer`) → `executeRun` → шаги → события в `RunGateway` (socket.io, namespace `runs`, комната `run:<id>`).

Транспорт исправен: сообщения доставляются и исполняются. Ломается движок шагов и контур доставки состояния в UI.

Опорные факты, полученные на живой БД и на реальном приложении:

- В `test.runs` 11 запусков: 7 `failed`, 4 `waiting_input`, ни одного `completed`.
- Три последних запуска содержат `error: "Failed to parse URL from /collections"` — шаг с `path: "/collections"` и `appId: "6a5279373ea93f687f66a19f"`.
- Приложение `Fake Dadata` (`openapiUrl: https://dadata-fake.cloud.astral-dev.ru/openapi.json`) имеет `host: undefined`; в его спецификации (OpenAPI 3.0.2, FastAPI) отсутствует поле `servers`.
- Проверено вживую: `new URL("/collections", "https://dadata-fake.cloud.astral-dev.ru")` → `GET` возвращает `200 application/json` со списком организаций. То есть после резолва базового URL шаг работает.
- Засеянные демо-приложения (`DataHub`, `VisionText`) имеют `host` вида `api.datahub.example.com` — без схемы и без base path.
- Есть запуски с тремя записями `stepResults` для одного `stepIndex: 0` и запуск, вернувшийся из `failed` в `waiting_input` — следы повторной доставки сообщения.

## Goals / Non-Goals

**Goals:**
- Сценарий из одного `api`-шага доходит до `completed`, и пользователь видит данные ответа.
- Ошибка исполнения всегда доходит до UI (в том числе если она произошла до подключения сокета).
- Повторная доставка сообщения не создаёт дубликатов в `stepResults` и не откатывает терминальный статус.
- Приложения без `servers` в спецификации импортируются с рабочим базовым URL.

**Non-Goals:**
- Реализация `file`-шага (сейчас заглушка `{ placeholder: true }`) — отдельная работа.
- Замена транспорта очереди или его настроек (visibility timeout, DLQ).
- Освобождение слота consumer'а на время ожидания ввода пользователя (`waitForPageInput` держит слот до 30 минут, при concurrency 5 это узкое место) — известный долг, выносится отдельно.
- Переработка playground.
- Чистка исторических «грязных» документов `runs`.

## Decisions

### 1. Базовый URL живёт на `App` как `baseUrl`, а не собирается из `host`

Хранить `host` (только имя хоста) недостаточно: теряются схема и base path. У `servers[0].url` вида `https://api.example.com/v1` префикс `/v1` обязателен для корректного вызова, а у спецификаций без `servers` хоста нет вовсе.

Решение: `App.baseUrl: string` — абсолютный URL со схемой и, при наличии, base path. Правило вывода в `OpenApiParser`:

1. Если есть `spec.servers[0].url` — разрешить его **относительно `openapiUrl`**: `new URL(servers[0].url, openapiUrl)`. Это корректно обрабатывает и абсолютные (`https://api.example.com/v1`), и относительные (`/v1`, `../api`) значения — последние в OpenAPI допустимы и разрешаются относительно места публикации документа.
2. Иначе (нет `servers`, как у FastAPI) — origin документа: `new URL(openapiUrl).origin`.
3. Нормализовать: убрать хвостовой `/`.

Это требует передавать `openapiUrl` в `parse()` — сейчас метод принимает только сырую спеку. Сигнатура меняется на `parse(rawSpec, openapiUrl)`.

`host` сохраняется как производное от `baseUrl` (`new URL(baseUrl).host`) — его читает UI карточки приложения, ломать не нужно.

**Альтернатива (отклонена):** резолвить базу в worker'е из `openapiUrl` на лету. Тогда каждый запуск повторяет разбор, а оператор не может переопределить адрес (staging/prod) — база должна быть явным атрибутом приложения.

### 2. Бэкфилл существующих приложений — одноразовый скрипт, не миграция схемы

Документов `apps` немного, поле опционально в схеме. Скрипт `apps/backend/scripts/backfill-app-base-url.ts` проходит по всем `apps` и проставляет `baseUrl` по тому же правилу (`specSnapshot.servers[0].url` относительно `openapiUrl`, иначе origin `openapiUrl`). Идемпотентен: пропускает документы с уже заполненным `baseUrl`.

### 3. Worker резолвит URL через `AppUrlResolver` с кэшем на запуск

`api`, `periodic` и `file`-шаги содержат `appId`. Вводится приватный резолвер в `WorkerService`:

```
resolveUrl(appId, path, query) →
  app = cache.get(appId) ?? await appModel.findById(appId)
  if (!app)          throw StepError(`Приложение ${appId} не найдено`)
  if (!app.baseUrl)  throw StepError(`У приложения «${app.name}» не задан базовый URL — переимпортируйте спецификацию`)
  url = joinPath(app.baseUrl, path)   // base path приложения + путь эндпоинта
  await ssrfGuard.assertSafe(url)
  return url + buildQuery(query)
```

Кэш — `Map<string, AppDocument>` на время одного `executeRun`: сценарий из N шагов к одному приложению не делает N запросов в Mongo.

Склейка путей: `new URL(baseUrl)` даёт `pathname` (например `/v1`), путь шага (`/collections`) добавляется к нему — `/v1/collections`. Двойные слэши схлопываются. `WorkerService` требует инъекции модели `App` (сейчас в `ExecutionModule` зарегистрированы только `Run` и `Scenario`).

SSRF: базовый URL уже проверялся `SsrfGuard` при импорте, но между импортом и запуском DNS может измениться, а `path` приходит из сценария. Проверка резолвнутого URL перед `fetch` — дешёвая и закрывает окно.

### 4. Снапшот состояния отправляется сервером при подключении к комнате

Корень «тишины» в UI — события room-scoped и не реплеятся, а клиент входит в комнату уже после того, как worker начал (и, для быстрых запусков, закончил) исполнение. Чинить это на клиенте (`GET /api/runs/:id` после `connect`) можно, но остаётся окно между запросом и подпиской.

Решение на сервере: `RunGateway.handleConnection` после `client.join('run:<id>')` читает `Run` из Mongo и отправляет **этому** сокету событие `run:status` с полным состоянием (`status`, `currentStep`, `stepResults`). Клиент, подключившийся в любой момент — при старте, после реконнекта, после перезагрузки страницы — получает актуальную картину. Порядок (`join` → чтение → `emit`) гарантирует, что события, произошедшие после `join`, не потеряются, а произошедшие до — приедут в снапшоте.

Тип `run:status` уже несёт `stepResults` — контракт менять не нужно, `WebSocketModule` получает `MongooseModule.forFeature([Run])`.

**Альтернатива (отклонена):** переставить в `RunPanel` подключение сокета перед `POST /api/runs` — `runId` ещё неизвестен, комнаты нет.

### 5. `RunPanel` строит прогресс из снапшота, а не только из инкрементальных событий

Сейчас обработчик `run:status` читает лишь `payload.status`, а `stepResults` игнорирует. Добавляется восстановление `stepProgress` из снапшота (по `stepIndex`), с сохранением уже полученных инкрементальных обновлений. Фазы `done`/`error`/`cancelled` выставляются из снапшотного статуса — включая случай, когда запуск успел упасть до подключения (тогда `errorMessage` берётся из `Run.error`).

Чтобы `error` из снапшота был доступен, payload `run:status` дополняется полем `error?: string`.

### 6. Результат отображается как данные, а не как тайминги

`resultItems` перестраивается на полезную нагрузку последнего успешного шага:
- объект → пары «ключ → значение» в `KeyValueGrid`;
- массив объектов → количество записей и первые N записей;
- скаляр/строка → как есть.

Тайминги остаются, но перестают быть единственным содержимым. `totalDurationMs` считается worker'ом как сумма `durationMs` шагов (сейчас `0` захардкожен и UI прячет блок).

### 7. Идемпотентная запись шагов: массив фиксированной длины + адресный `$set`

`$push` + позиционный `$` ломаются при повторной доставке: дубликаты по `stepIndex`, обновление первого совпадения.

В начале `executeRun` (при `currentStep === 0` либо когда длина не совпадает с числом шагов) `stepResults` инициализируется массивом длины `steps.length` со статусом `pending` для каждого шага. Далее шаг обновляется адресно: `$set: { "stepResults.<i>": {...} }`. Повторное исполнение перезаписывает элемент, а не добавляет новый. Индекс известен из цикла — позиционный оператор не нужен.

### 8. Доменная ошибка шага не пробрасывается в consumer

`executeRun` ловит ошибку шага, вызывает `failRun` (терминальный статус + событие) и **возвращает управление** вместо `throw`. Сообщение удаляется из очереди — детерминированный сбой (битый URL, 404 у стороннего API) не имеет смысла ретраить: три доставки дают три одинаковых падения и портят состояние.

Пробрасываются только инфраструктурные сбои (ошибка записи в Mongo, недоступность БД) — там повторная доставка действительно спасает. Разделение: доменные ошибки шага оборачиваются в `StepExecutionError`; всё, что не `StepExecutionError`, летит наружу и уходит на retry/DLQ.

### 9. `getRun` проверяет владельца

Эндпоинт становится опорным для восстановления состояния (и уже используется снапшотом). Сейчас любой аутентифицированный пользователь может прочитать чужой `Run` по id. Добавляется сверка `run.userId` с `req.user.userId` → `ForbiddenException`. Гейт сокета (`handleConnection`) в этой итерации не ужесточается — отмечено как отдельный долг.

## Risks / Trade-offs

- **Изменение сигнатуры `OpenApiParser.parse`** затрагивает импорт и реимпорт. Митигация: оба вызова в `apps.service.ts`, покрываются юнит-тестом парсера на спеке без `servers`.
- **Приложения без `baseUrl` после бэкфилла** (например, `openapiUrl` невалиден) дадут доменную ошибку на запуске. Это лучше текущего `Failed to parse URL`: сообщение прямо указывает на переимпорт.
- **Снапшот при подключении добавляет чтение Mongo на каждый connect** — один `findById` на запуск, пренебрежимо.
- **Отказ от retry доменных ошибок** означает, что транзиентная ошибка стороннего API (503) больше не будет ретраиться очередью. Сегодня ретрай всё равно нерабочий (портит состояние). Retry шага с backoff — отдельная задача, вне этого изменения.
- **Стартовый снапшот может «догнать» инкрементальные события** и переписать более свежее состояние более старым. Митигация: `RunPanel` применяет снапшот только как базу (не понижает уже достигнутые статусы шагов).

## Migration Plan

1. Добавить `baseUrl` в схему и типы (обратно совместимо, поле опционально).
2. Обновить парсер и импорт/реимпорт — новые приложения получают `baseUrl` сразу.
3. Прогнать бэкфилл по существующим `apps`.
4. Выкатить worker с резолвом URL (без `baseUrl` шаг падает с понятной ошибкой — не хуже текущего состояния).
5. Выкатить снапшот в gateway и изменения `RunPanel`.

Откат: снять изменения worker'а/gateway; `baseUrl` в БД безвреден для старого кода.

## Open Questions

- Нужен ли оператору ручной override `baseUrl` в UI приложения (staging vs prod)? Схема это уже позволяет; форма редактирования — вне объёма.
- Ужесточение авторизации WebSocket-подключения по `runId` (сейчас комнату может слушать кто угодно, зная id) — отдельное изменение по безопасности.
