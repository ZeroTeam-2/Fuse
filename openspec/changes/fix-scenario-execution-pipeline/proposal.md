## Why

Основной механизм платформы не работает: запуск сценария стартует, но пользователь не получает ни результата, ни ошибки — UI навсегда остаётся в состоянии «Выполняем сценарий…».

Диагностика на живой БД (`test.runs`, 11 запусков: 7 `failed`, 4 `waiting_input`, 0 `completed`) и на реальном приложении «Fake Dadata» выявила четыре независимых дефекта, каждый из которых сам по себе ломает сквозной поток.

1. **HTTP-вызов шага не имеет базового URL (блокер).** `WorkerService.executeApiStep` вызывает `fetch(step.path)`, где `step.path` — относительный путь из OpenAPI (`/collections`). Node.js `fetch` требует абсолютный URL. Все три последних запуска сценария «Получить список организаций» упали с сохранённой в БД ошибкой `Failed to parse URL from /collections`. Worker вообще не загружает `App` по `step.appId` и не знает, к какому хосту обращаться.

2. **У импортированного приложения не заполнен хост.** `OpenApiParser.extractHost` читает только `spec.servers[0].url`. В реальной спецификации (`https://dadata-fake.cloud.astral-dev.ru/openapi.json`, FastAPI) поля `servers` нет — `host` сохраняется как `undefined`. Даже если бы worker читал `app.host`, базы для этого приложения не существовало бы. Дополнительно `host` хранит только имя хоста — без схемы и без base path из `servers[].url`.

3. **Гонка: события исполнения теряются до подключения сокета.** `RunPanel.startRun()` сначала делает `POST /api/runs` (сообщение уже в очереди — worker может начать исполнение немедленно) и только потом подключает WebSocket и входит в комнату `run:<id>`. socket.io не реплеит пропущенные события, а `RunPanel` никогда не читает `GET /api/runs/:id`. Любое событие, отправленное до входа в комнату, теряется безвозвратно — UI зависает без единого сообщения. Для быстрых и мгновенно падающих запусков это происходит практически всегда. Требование спеки «Reconnect после обрыва WebSocket» (получить текущий статус и накопленные результаты) в коде не реализовано.

4. **Результат шага не выводится в UI.** Даже при успешном запуске `resultItems` в `RunPanel` маппит шаги в пару «название → длительность»: фактические данные ответа (`step.result`) не отображаются вообще, а `run:done.totalDurationMs` захардкожен в `0`. То есть «на выходе» пусто по построению.

Дополнительно очередь загрязняется повторными доставками: `executeRun` пробрасывает ошибку шага наружу, `sqs-consumer` не удаляет сообщение, SQS доставляет его повторно (до 3 раз). Каждая доставка делает `pushRunStep` → в `stepResults` появляются дубликаты с одинаковым `stepIndex` (в БД: три записи «Загрузка файла» с `stepIndex: 0`), а `setRunStep` с позиционным оператором `$` обновляет первый попавшийся из них. Повторная доставка после `failRun` вернула один из запусков из `failed` обратно в `waiting_input` — статус в БД перестаёт быть достоверным.

Транспорт очереди (Yandex Cloud Message Queue, SQS-совместимый) при этом исправен: сообщения доставляются, worker их исполняет. Проблема целиком в движке исполнения и в контуре доставки состояния в UI.

## What Changes

- **Резолв базового URL для шагов.** Ввести на `App` поле `baseUrl` (абсолютный URL: схема + хост + base path). Выводить его при импорте и реимпорте: `servers[0].url`, разрешённый относительно `openapiUrl`, иначе — origin самого `openapiUrl`. Поле `host` остаётся производным (для отображения).
- **Бэкфилл существующих приложений**: одноразовый скрипт проставляет `baseUrl` уже импортированным приложениям (включая те, у которых `host` пуст).
- **Worker резолвит абсолютный URL** для `api`-, `periodic`- и `file`-шагов: загружает `App` по `appId` (с кэшем на время запуска), склеивает `baseUrl` + `path`, и только затем вызывает `fetch`. Отсутствие приложения или `baseUrl` даёт понятную доменную ошибку вместо `Failed to parse URL`.
- **Резолвнутый URL проходит через существующий `SsrfGuard`** перед исполнением.
- **Снапшот состояния при подключении к сокету.** `RunGateway.handleConnection` после входа в комнату отправляет подключившемуся клиенту текущее состояние `Run` (статус, `currentStep`, все `stepResults`). Это закрывает и стартовую гонку, и реконнект.
- **`RunPanel` восстанавливает прогресс из снапшота** `run:status` (сейчас обработчик читает только `status` и игнорирует `stepResults`).
- **`RunPanel` показывает данные результата** — полезную нагрузку последнего шага, а не только тайминги. `run:done.totalDurationMs` считается worker'ом по-настоящему.
- **Идемпотентная запись результатов шагов.** `stepResults` инициализируется массивом фиксированной длины (по шагу на элемент) в начале запуска; шаги обновляются адресно по индексу (`$set: stepResults.<i>`) вместо `$push` + позиционного `$`. Повторная доставка не плодит дубликаты и не портит статус.
- **Ошибка шага не приводит к повторной доставке.** `executeRun` не пробрасывает доменные ошибки наружу: терминальное состояние уже записано `failRun`. Пробрасываются только инфраструктурные сбои (недоступность БД), для которых retry осмыслен.
- **`GET /api/runs/:id` проверяет владельца** — эндпоинт становится опорным для восстановления состояния, читать чужие запуски нельзя.

## Capabilities

### New Capabilities

(нет новых capabilities — исправление существующего поведения)

### Modified Capabilities

- `scenario-execution`: требование «Движок исполнения шагов» дополняется резолвом базового URL приложения (шаг без резолвимого `baseUrl` — доменная ошибка, а не краш) и идемпотентной записью результатов при повторной доставке сообщения. Требование «Real-time прогресс через WebSocket» дополняется снапшотом состояния при подключении — клиент, подключившийся после старта исполнения, получает уже произошедшие события. Требование «Простой режим запуска» дополняется отображением данных результата, а не только таймингов.
- `api-app-management`: требование «Создание приложения с импортом OpenAPI» дополняется выводом и хранением `baseUrl` (включая спецификации без `servers`).

## Impact

- **Код (backend)**: `apps/backend/src/execution/worker.service.ts` (резолв URL, идемпотентные записи, обработка ошибок), `apps/backend/src/execution/execution.controller.ts` и `execution.service.ts` (проверка владельца в `getRun`), `apps/backend/src/websocket/run.gateway.ts` и `websocket.module.ts` (снапшот при подключении), `apps/backend/src/apps/openapi-parser.ts` и `apps/backend/src/apps/apps.service.ts` (вывод `baseUrl`), `apps/backend/src/apps/app.schema.ts` (поле `baseUrl`).
- **Код (frontend)**: `apps/frontend/components/RunPanel.vue` (восстановление прогресса из снапшота, вывод данных результата).
- **Типы**: `packages/shared` — `App.baseUrl`; payload `run:status` уже содержит `stepResults`.
- **Данные**: одноразовый бэкфилл `baseUrl` для существующих документов `apps`; существующие «грязные» `runs` (дубликаты `stepResults`, залипший `waiting_input`) остаются как есть — новые запуски пишутся корректно.
- **Не затрагивается**: транспорт очереди (SQS/YMQ), схема `Run`-статусов, авторизация, маркетплейс, playground, загрузка файлов в MinIO.
