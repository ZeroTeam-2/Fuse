## MODIFIED Requirements

### Requirement: Создание приложения с импортом OpenAPI
Система SHALL позволять создать «приложение» (подключённый API-провайдер): название, краткое описание и источник OpenAPI-спецификации. Источником MAY быть либо **ссылка (URL)** на спецификацию (JSON или YAML), либо **загруженный файл** (`.json`, `.yaml`, `.yml`). При импорте система MUST автоматически разобрать спецификацию и извлечь список endpoints (метод, путь, summary), хост API и версию.

Система MUST принимать YAML наравне с JSON: для URL-импорта формат определяется по `Content-Type` ответа (`application/json` → JSON; `application/yaml`, `application/x-yaml`, `text/yaml` → YAML), а при неинформативном `Content-Type` — по первому непробельному символу тела (`{` или `[` → JSON, иначе → YAML). Для файлового импорта формат MUST определяться по расширению/MIME файла, а при неоднозначности — по содержимому тела тем же правилом. Если разбор по определённому формату не удаётся, система MUST попробовать альтернативный формат в качестве запасного варианта.

Файловый импорт MUST принимать только один файл размером не более 10 МБ с расширением `.json`, `.yaml` или `.yml`. Файл MUST NOT сохраняться в файловом хранилище — система MUST разобрать его в памяти и сохранить только разобранный снимок спецификации (`specSnapshot`) и endpoints.

При импорте система MUST вывести и сохранить **базовый URL приложения** (`baseUrl`) — абсолютный адрес со схемой, хостом и, при наличии, базовым путём. Для URL-импорта правило вывода: значение `servers[0].url` из спецификации, разрешённое относительно URL самой спецификации; если поле `servers` отсутствует — origin URL спецификации. Для файлового импорта (где URL спецификации недоступен): система MUST использовать абсолютное значение `servers[0].url` из спецификации; если оно отсутствует или относительно, система MUST требовать от пользователя явный **«Базовый URL API»** и использовать его. Если базовый URL невозможно вывести ни из спецификации, ни от пользователя, импорт MUST завершиться ошибкой и приложение MUST NOT создаваться. Отображаемый `host` MUST быть производным от `baseUrl`. Базовый URL MUST быть достаточным для исполнения шагов сценария: путь endpoint'а, присоединённый к `baseUrl`, MUST давать вызываемый абсолютный URL.

Приложение, созданное из файла, MUST хранить `openapiUrl` пустым (источник — файл, не URL). Для таких приложений повторный импорт по URL MUST быть недоступен (пока не реализован повторный импорт файлом).

При разборе схемы ответа система MUST сохранять **признак массивного ответа** endpoint'а: если тело успешного ответа — коллекция, схема выходов MUST помечаться как массив, а извлечённые поля MUST описывать **элемент** коллекции. Признак MUST NOT теряться при разворачивании `items`. Для полей типа `array` внутри объекта система MUST сохранять схему элемента (поля элемента), если она задана в спецификации. Эти данные MUST быть доступны конструктору сценариев через схему шага.

То же правило MUST применяться при повторном импорте спецификации.

#### Scenario: Успешный импорт спецификации по URL
- **WHEN** пользователь указывает валидный URL OpenAPI JSON и нажимает «Импортировать»
- **THEN** система показывает «Спецификация разобрана — найдено N endpoints» и список endpoints с методами и summary

#### Scenario: Успешный импорт YAML-спецификации по URL
- **WHEN** пользователь указывает URL, отдающий валидный OpenAPI YAML с `Content-Type: application/yaml`
- **THEN** система разбирает YAML, извлекает endpoints и показывает «Спецификация разобрана — найдено N endpoints» с тем же результатом, что и для эквивалентного JSON

#### Scenario: Успешный импорт YAML-файла
- **WHEN** пользователь перетаскивает валидный OpenAPI файл `openapi.yaml` в зону загрузки и нажимает «Импортировать»
- **THEN** система разбирает YAML-файл, извлекает endpoints и показывает предпросмотр «Спецификация разобрана — найдено N endpoints» с тем же результатом, что и для эквивалентного JSON по URL

#### Scenario: Успешный импорт JSON-файла
- **WHEN** пользователь выбирает валидный OpenAPI файл `openapi.json` через клик по зоне загрузки
- **THEN** система разбирает JSON-файл и показывает предпросмотр endpoints, как при URL-импорте эквивалентной спецификации

#### Scenario: Файл с абсолютным servers URL
- **WHEN** загруженный файл содержит `servers[0].url: https://api.example.com/v1`
- **THEN** система сохраняет `baseUrl` равным `https://api.example.com/v1` без запроса базового URL у пользователя

#### Scenario: Файл без servers требует явный базовый URL
- **WHEN** загруженный файл не содержит поля `servers` и пользователь не указал «Базовый URL API»
- **THEN** система показывает ошибку «Не удалось определить базовый URL API. Укажите его в поле «Базовый URL API».» и не выполняет импорт

#### Scenario: Файл без servers с пользовательским базовым URL
- **WHEN** загруженный файл не содержит `servers`, но пользователь ввёл «Базовый URL API» равным `https://api.example.com`
- **THEN** система сохраняет `baseUrl` равным `https://api.example.com`, `host` — `api.example.com`, и импорт завершается успешно

#### Scenario: Перетаскивание нескольких файлов
- **WHEN** пользователь перетаскивает одновременно два файла в зону загрузки
- **THEN** система принимает только первый файл (или отклоняет с подсказкой «Можно загрузить только один файл»)

#### Scenario: Превышение размера файла
- **WHEN** пользователь загружает файл размером более 10 МБ
- **THEN** система отклоняет файл с ошибкой размера и не выполняет импорт

#### Scenario: Недопустимое расширение файла
- **WHEN** пользователь загружает файл с расширением, отличным от `.json`, `.yaml`, `.yml` (например, `.txt`)
- **THEN** система отклоняет файл с ошибкой «Допустимы только файлы .json, .yaml, .yml» и не выполняет импорт

#### Scenario: Невалидный файл спецификации
- **WHEN** загруженный файл не является валидным OpenAPI документом в формате JSON или YAML
- **THEN** система показывает ошибку импорта и не даёт создать приложение

#### Scenario: Спецификация без поля servers (URL-импорт)
- **WHEN** по URL импортируется спецификация, в которой отсутствует `servers`
- **THEN** система сохраняет `baseUrl` равным origin URL спецификации (`https://host`), и шаги сценария по этому приложению исполняются успешно

#### Scenario: Базовый путь из servers сохраняется
- **WHEN** спецификация содержит `servers[0].url` вида `https://api.example.com/v1`
- **THEN** система сохраняет `baseUrl` со всем базовым путём (`https://api.example.com/v1`), и вызовы шагов идут по `https://api.example.com/v1/<path>`

#### Scenario: Невалидная спецификация (URL)
- **WHEN** URL недоступен или документ не является валидным OpenAPI в формате JSON или YAML
- **THEN** система показывает ошибку импорта и не даёт создать приложение

#### Scenario: Создание после импорта
- **WHEN** спецификация разобрана и пользователь нажимает «Создать приложение»
- **THEN** система сохраняет приложение вместе с endpoints и их схемами входов/выходов (из параметров и примеров ответов)

#### Scenario: Созданное из файла приложение не имеет URL
- **WHEN** приложение создано из загруженного файла
- **THEN** система сохраняет приложение с пустым `openapiUrl`, а повторный импорт по URL для этого приложения недоступен

#### Scenario: Endpoint возвращает коллекцию
- **WHEN** успешный ответ endpoint'а описан как массив объектов (`type: array`, `items` с полями `id`, `name`)
- **THEN** система сохраняет выходы как поля элемента (`id`, `name`) вместе с признаком, что ответ — массив, и конструктор сценариев предлагает для такого источника условие фильтрации

#### Scenario: Поле-список внутри объекта
- **WHEN** ответ endpoint'а — объект с полем `items`, описанным как массив объектов
- **THEN** система сохраняет поле `items` с типом `array` и схемой его элемента, а не как тип без содержимого
