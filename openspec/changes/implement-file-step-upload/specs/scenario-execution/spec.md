# scenario-execution — delta (implement-file-step-upload)

## MODIFIED Requirements

### Requirement: Загрузка файлов (single и chunked)
Для файловых шагов система SHALL выбирать режим загрузки по размеру файла: файлы ≤ 10 МБ (настраиваемый порог `FILE_SINGLE_UPLOAD_MAX_MB`, публикуемый фронтенду через runtime-конфиг) загружаются одним запросом `multipart/form-data`; файлы > 10 МБ загружаются чанками (5–10 МБ) с отображением прогресса (проценты и байты), поддержкой паузы/продолжения и отмены. При обрыве соединения загруженные чанки MUST сохраняться, и пользователь MAY возобновить загрузку с места обрыва. После загрузки система SHALL показывать стадию обработки (опрос статуса) и результат (извлечённые поля/текст) с возможностью загрузить другой файл. Файлы MUST проверяться на допустимый формат и лимит размера, заданные страницей шага.

Загрузка MUST выполняться из страницы шага в хранилище платформы (существующее API `/api/uploads/*`) **до** сабмита страницы: пока обязательный dropzone-блок не завершил загрузку, продолжение шага MUST быть заблокировано. В данные сабмита страницы MUST попадать ссылка на загруженный объект (`objectName`, имя, размер, тип файла), а не содержимое файла и не голые метаданные.

#### Scenario: Single-загрузка до порога
- **WHEN** пользователь выбирает в dropzone файл размером не больше порога `FILE_SINGLE_UPLOAD_MAX_MB`
- **THEN** файл уходит одним `multipart/form-data`-запросом с отображением прогресса, и после завершения сабмит страницы передаёт шагу ссылку на объект хранилища

#### Scenario: Chunked-загрузка сверх порога
- **WHEN** пользователь выбирает файл больше порога
- **THEN** система инициирует чанковую сессию, грузит части по 5–10 МБ с прогрессом в процентах и байтах, позволяет паузу/продолжение и отмену, а по завершении передаёт шагу ссылку на объект

#### Scenario: Возобновление после обрыва
- **WHEN** соединение обрывается на середине загрузки
- **THEN** система показывает «Соединение прервалось на X% — загруженные чанки сохранены» и после «Возобновить загрузку» продолжает с того же места

#### Scenario: Файл не проходит валидацию страницы
- **WHEN** выбранный файл не подходит по формату или превышает лимит размера, заданные dropzone-блоком страницы шага
- **THEN** загрузка не начинается, пользователь видит понятную ошибку и может выбрать другой файл

## ADDED Requirements

### Requirement: Исполнение файлового шага
Worker SHALL исполнять файловый шаг, доставляя загруженный пользователем файл в API провайдера: по входам шага находится ссылка на объект хранилища (значение dropzone-блока по его привязке), объект читается из хранилища и отправляется запросом `uploadMethod` + `uploadPath` с телом `multipart/form-data`. Имя multipart-поля файла MUST определяться по body-входу бинарного формата из схемы endpoint'а, а при его отсутствии — по привязке dropzone-блока, если она совпадает с body-входом (спеки, импортированные до маппинга `format: binary`, описывают файловое поле строкой); крайний фолбэк — `file`; остальные разрешённые входы шага MUST раскладываться по своим местам (path/query/header — как обычно, прочие body-входы — текстовыми полями рядом с файлом). URL вызова MUST собираться существующим механизмом (базовый URL приложения + путь, проверка SSRF); заголовок `Content-Type` MUST выставляться средой исполнения (boundary), а не жёстко.

Если у шага задан `statusEndpoint`, после успешной загрузки worker MUST опрашивать его с интервалом `intervalSec` до признака завершения либо таймаута, публикуя `progress`-события WebSocket по полю `progressField` (по образцу периодического шага); во входы опроса MUST добавляться выходы ответа загрузки (для подстановки идентификатора задачи в путь статуса). Результат шага — последний ответ опроса, а без `statusEndpoint` — ответ загрузки; внутренние ссылки хранилища (presigned URL, `objectName`) MUST NOT попадать в результат шага.

Если во входах шага нет файловой ссылки (страница не сабмитила загрузку), шаг MUST завершаться доменной ошибкой с диагностикой «файл не был загружен», переводящей запуск в `failed` без повторной доставки.

#### Scenario: Файл доставлен провайдеру
- **WHEN** worker исполняет файловый шаг после сабмита страницы с загруженным файлом
- **THEN** worker читает объект из хранилища, вызывает `uploadPath` приложения multipart-запросом с файлом и остальными входами шага и сохраняет ответ провайдера как результат шага

#### Scenario: Обработка с опросом статуса
- **WHEN** у файлового шага задан `statusEndpoint` c интервалом и полем прогресса, и провайдер принял файл, вернув идентификатор задачи
- **THEN** worker опрашивает endpoint статуса, подставляя идентификатор из ответа загрузки, публикует прогресс на каждой итерации и завершает шаг финальным ответом опроса

#### Scenario: Файл не был загружен
- **WHEN** файловый шаг исполняется, а во входах шага нет ссылки на загруженный объект
- **THEN** шаг завершается доменной ошибкой «файл не был загружен» с названием шага, запуск переходит в `failed`
