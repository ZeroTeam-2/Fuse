## Why

Сценарии не умеют работать с массивами. Если шаг возвращает коллекцию (`[{...}, {...}]` или объект с полем-списком), у пользователя нет способа сказать, какой именно элемент передать дальше — а это самый частый случай в реальных API: «найди организацию по ИНН и возьми её `id`». Хуже того, система сейчас молча делает вид, что массива нет: парсер OpenAPI разворачивает массивный ответ в плоский список полей элемента, поэтому схема шага обещает объект с полем `id`, а рантайм получает массив и подставляет `undefined`. Сценарий не падает — он тихо уходит дальше со сломанным значением.

## What Changes

- **Схема шага знает про массивы.** При импорте OpenAPI сохраняется признак «ответ endpoint — массив» и схема элемента для полей-списков (сейчас и то, и другое теряется). `GET /api/scenarios/:id/step-schema/:i` отдаёт этот признак фронту.
- **Новый элемент маппинга — фильтр.** У входного параметра с источником «Из шага» появляется условие фильтрации: поле элемента, оператор (`=`, `≠`, `>`, `<`, `≥`, `≤`, «содержит») и значение сравнения. Значение задаётся тремя способами, как и сам маппинг: ручной ввод, константа (с поддержкой `{{s0:key}}`-шаблонов) или значение из другого шага. Тип значения сверяется с типом выбранного поля элемента.
- **Фильтр в UI настройки шага.** Блок фильтра появляется в `StepConfig` под селектами «Шаг-источник» / «Поле результата» — только когда шаг-источник действительно отдаёт массив.
- **Резолвер массивов в исполнении.** Worker находит массив в результате шага, отбирает элемент по условию и берёт из него нужное поле. Пустая выборка — доменная ошибка шага с внятной диагностикой; несколько совпадений — берётся первый элемент, в результат шага пишется предупреждение.
- **Предпосылка (побочный фикс).** Ветка «Ручной ввод» в резолвере сегодня мертва: worker вызывает `resolveMappings` без пользовательского ввода. Без этого режим «ручной ввод» у значения фильтра не заработает, поэтому ввод пользователя/данные страницы шага прокидываются в резолвер.
- Существующие сценарии продолжают работать: маппинг `s{idx}:{key}` без фильтра на массивном результате берёт первый элемент вместо сегодняшнего `undefined`.

## Capabilities

### New Capabilities

Новых capability нет — механизм расширяет уже существующие маппинги и движок исполнения.

### Modified Capabilities

- `scenario-builder`: требование «Маппинг данных между шагами» дополняется условием фильтрации для источника «Из шага», когда шаг-источник возвращает массив.
- `scenario-execution`: требование «Движок исполнения шагов» дополняется резолвом массивных выходов — отбор элемента по фильтру, поведение при 0 и >1 совпадениях, прокидывание пользовательского ввода в резолвер.
- `api-app-management`: требование «Создание приложения с импортом OpenAPI» дополняется сохранением признака массивного ответа и схемы элемента (иначе фильтр не на чем строить).

## Impact

**Модель данных (`packages/shared/src/types/index.ts`)**
- `SchemaField` — новое опциональное поле `items?: SchemaField[]` (поля элемента для `type: "array"`).
- `BaseStep` — новое опциональное поле `filters?: Record<string, StepFilter>` (ключ = ключ входного параметра).
- Новые типы `StepFilter`, `StepFilterValue`, `FilterOperator`.

**Бэкенд**
- `apps/backend/src/apps/openapi-parser.ts` — перестаёт молча схлопывать массивный ответ, извлекает схему элемента.
- `apps/backend/src/apps/app.schema.ts` — `SchemaFieldDoc.items`, признак массивного ответа у endpoint (сабсхема strict, без правки поле не сохранится).
- `apps/backend/src/scenarios/scenarios.service.ts` — `getStepSchema` отдаёт признак массива.
- `apps/backend/src/execution/mapping-resolver.ts` — резолв массивов и фильтров, сравнение по типам.
- `apps/backend/src/execution/worker.service.ts` — передача пользовательского ввода в резолвер, ошибка шага при пустой выборке.
- Mongoose-схема сценария (`steps: Mixed[]`) и `UpdateScenarioDto` (`steps?: unknown[]`) правок **не требуют** — новое поле шага доедет до БД как есть.

**Фронтенд**
- `apps/frontend/components/scenario/StepConfig.vue` — блок фильтра, сериализация `filters`.
- `apps/frontend/pages/my/scenarios/[id]/edit.vue` — сводка источника параметра в списке шагов показывает фильтр.

**Данные**
Существующие приложения импортированы без признака массива и схемы элемента. Чтобы настроить фильтр по такому API, спецификацию нужно переимпортировать (механизм «Обновление спецификации» уже есть). Рантайм при этом деградирует мягко — работает и без признака, по фактическому типу результата.

**Тесты**
`apps/backend/test/mapping-resolver.spec.ts` — сейчас ни одного кейса с массивным результатом шага; добавляются кейсы на фильтры, операторы, типы и краевые случаи (0 / >1 совпадений).
