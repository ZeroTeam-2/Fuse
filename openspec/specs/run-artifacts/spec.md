## Requirements
### Requirement: Сохранение файловых ответов API в S3
Worker SHALL распознавать файловый ответ внешнего API по заголовку `Content-Type` (не `application/json` и не `text/*`) либо по заголовку `Content-Disposition: attachment` и сохранять тело такого ответа в S3 бинарно, без прохождения через `response.text()`. Ключ объекта MUST строиться по схеме `runs/{userId}/{runId}/{stepIndex}-{uuid}{ext}`, где расширение выводится из `Content-Type` или имени файла в `Content-Disposition`. В `stepResults[i].result` MUST записываться файловая ссылка `RunFileRef` (`objectName`, `fileName`, `fileSize`, `fileType`), а не сырое тело. Размер сохраняемого файла MUST ограничиваться конфигурируемым лимитом; превышение — доменная ошибка шага.

#### Scenario: API возвращает PDF
- **WHEN** api-шаг получает ответ `200` с `Content-Type: application/pdf`
- **THEN** worker сохраняет тело в S3 под ключом `runs/{userId}/{runId}/…`, кладёт в результат шага `RunFileRef` с именем и размером файла, и запуск продолжается следующим шагом

#### Scenario: JSON-ответ сохраняется как раньше
- **WHEN** api-шаг получает ответ с `Content-Type: application/json`
- **THEN** результат шага — распарсенный JSON в Mongo, файл в S3 не создаётся

### Requirement: Реестр файлов запуска
Документ `Run` SHALL вести реестр всех связанных с запуском объектов S3: worker MUST регистрировать в нём каждый сохранённый файл-артефакт, а также файловые ссылки (`UploadedFileRef`), переданные пользователем во входах запуска и в данных страниц (`page-submit`). Реестр — источник истины для каскадного удаления файлов при удалении запуска.

#### Scenario: Файл пользователя учитывается за запуском
- **WHEN** пользователь на page-шаге загружает файл через dropzone и отправляет страницу
- **THEN** `objectName` загруженного файла попадает в реестр файлов запуска

#### Scenario: Артефакт учитывается за запуском
- **WHEN** worker сохраняет файловый ответ API в S3
- **THEN** `objectName` артефакта добавляется в реестр файлов того же документа `Run`

### Requirement: Скачивание файлов-результатов
Система SHALL предоставлять endpoint выдачи ссылки на скачивание файла запуска (presigned URL S3 с ограниченным сроком действия) по `runId` и `objectName`. Доступ MUST иметь только владелец запуска, а `objectName` MUST числиться в реестре файлов этого запуска (иначе 403/404). Интерфейс на экране результата MUST отображать файловый результат шага как карточку файла (имя, размер, тип) с кнопкой скачивания вместо сырого JSON.

#### Scenario: Владелец скачивает артефакт
- **WHEN** владелец завершённого запуска нажимает «Скачать» у файлового результата шага
- **THEN** фронтенд получает presigned URL и браузер скачивает файл из S3

#### Scenario: Чужой или посторонний objectName
- **WHEN** пользователь запрашивает ссылку для чужого запуска либо для `objectName`, не числящегося в реестре запуска
- **THEN** система отказывает (403/404) и presigned URL не выдаётся
