# scenario-builder

## Purpose

Конструктор сценариев: создание карточек сценариев, добавление шагов разных типов, маппинг данных между шагами, управление потоком и публикация.
## Requirements
### Requirement: Карточка сценария
Система SHALL позволять создать сценарий (use-case карточку) с названием, обложкой (загрузка/drag-and-drop изображения), rich-text описанием для маркетплейса (жирный, курсив, подчёркивание, заголовки, код) и категорией из справочника с поиском. Tagline карточки MUST выводиться из первых ~90 символов текстового описания. Сценарий — самостоятельная сущность и MAY собирать шаги из разных API.

#### Scenario: Создание сценария
- **WHEN** пользователь заполняет детали и нажимает «Создать сценарий»
- **THEN** система создаёт неопубликованный сценарий без шагов и открывает редактор шагов

#### Scenario: Редактирование метаданных
- **WHEN** пользователь меняет обложку, описание или категорию на вкладке «Основная»
- **THEN** изменения сохраняются и отражаются в карточке маркетплейса после публикации

### Requirement: Типы шагов
Редактор шагов SHALL поддерживать четыре типа шага:
1. **Endpoint API** — вызов метода выбранного приложения (выбор приложения с поиском → выбор endpoint); если у endpoint'а есть файловый body-вход (`type: "file"`, из `format: binary` спецификации), файл поставляется пользователем через dropzone-блок страницы шага, привязанный к этому входу, — отдельный тип шага для этого не требуется;
2. **Другой сценарий** — вложение готового сценария (его входы → выходы); система MUST предотвращать циклические вложения;
3. **Задержка** — пауза на заданное число секунд перед следующим шагом (пресеты 1/3/5/10 сек и произвольное значение);
4. **Периодический запрос** — опрос выбранного endpoint с интервалом (1–600 сек) и полем прогресса из схемы ответа.

Тип шага «Файл» MUST быть удалён из палитры: загрузка файла — свойство api-шага (dropzone-привязка к файловому входу), режим single/chunked выбирается автоматически по размеру файла при загрузке со страницы, а стадия обработки строится композицией со следующим шагом «Периодический запрос». Существующие сценарии со старыми шагами типа `file` MUST открываться в редакторе без ошибок.

Периодический запрос — тот же вызов endpoint, что и «Endpoint API», повторяемый до признака завершения. Шаг MUST сохранять ссылку на выбранный endpoint (`endpointId`), и его настройка MUST быть идентична настройке api-шага: панель шага показывает те же секции входных параметров по местам (Path/Query/Header/Body из схемы endpoint) и те же выходные поля, доступные следующим шагам; отличаются только настройки опроса (интервал, поле прогресса). Для periodic-шагов, созданных до появления `endpointId`, система MUST восстанавливать схему endpoint по совпадению метода и пути опроса среди endpoints приложения; если endpoint не найден, шаг показывается с пустой схемой без ошибки редактора.

При выборе endpoint (типы «Endpoint API» и «Периодический запрос») список endpoints выбранного приложения MUST выводиться **сворачиваемыми смысловыми блоками** (имя блока — тег операции, иначе первый сегмент пути; одноуровневые пути без тега — «Прочее») и MUST сопровождаться **сквозным поиском** по методу, пути, summary и имени блока. Стиль поиска и группировки MUST быть единым с карточкой приложения. При активном поиске отображаются только совпавшие endpoints; блоки с совпадениями раскрываются.

#### Scenario: Добавление API-шага
- **WHEN** пользователь выбирает тип «Endpoint API», приложение и endpoint, задаёт название шага и нажимает «Добавить шаг»
- **THEN** шаг появляется в потоке с методом, путём и названием приложения

#### Scenario: Выбор endpoint сгруппирован и с поиском
- **WHEN** пользователь выбрал приложение и открыл выбор endpoint
- **THEN** endpoints показаны сворачиваемыми смысловыми блоками (тег или первый сегмент пути) с полем сквозного поиска в едином стиле с карточкой приложения; ввод запроса оставляет только совпавшие endpoints

#### Scenario: Вложение сценария
- **WHEN** пользователь выбирает тип «Другой сценарий» и указывает сценарий
- **THEN** шаг принимает входы вложенного сценария и отдаёт его выходы; сценарии, приводящие к циклу, не предлагаются

#### Scenario: Настройка файлового шага
- **WHEN** пользователь добавляет шаг «Endpoint API» по endpoint'у с файловым body-входом и другими параметрами (например, `doc` + `data`) и привязывает dropzone-блок страницы к файловому входу
- **THEN** остальные параметры настраиваются в тех же секциях Path/Query/Header/Body (константа, ручной ввод, «Из шага»), отдельного типа «Файл» в палитре шагов нет, а режим single/chunked определяется автоматически по размеру файла при загрузке

#### Scenario: Периодический шаг настраивается как api-шаг
- **WHEN** пользователь добавляет шаг «Периодический запрос» по endpoint с path-, query-, header- и body-параметрами и открывает панель настройки шага
- **THEN** панель показывает секции Path/Query/Header/Body с параметрами endpoint и его выходные поля — тот же набор, что у шага «Endpoint API» по этому endpoint, плюс настройку интервала опроса

#### Scenario: Выходы периодического шага доступны следующим шагам
- **WHEN** после periodic-шага добавлен шаг, чей вход настраивается «Из шага»
- **THEN** periodic-шаг предлагается как шаг-источник, и его выходные поля (из схемы ответа endpoint) доступны для выбора

#### Scenario: Легаси periodic-шаг без ссылки на endpoint
- **WHEN** пользователь открывает панель настройки periodic-шага, созданного до появления `endpointId`, а у приложения есть endpoint с тем же методом и путём
- **THEN** панель показывает секции параметров и выходы этого endpoint, восстановленные по совпадению метода и пути

### Requirement: Маппинг данных между шагами
Для каждого входного поля шага система SHALL позволять выбрать источник: выход одного из предыдущих шагов, ввод пользователя или константа. Система MUST автоматически подставлять источник по совпадению ключа или метки с выходами предыдущих шагов. Заголовки авторизации MUST по умолчанию получать константу `Bearer {{access_token}}`. Панель шага SHALL показывать выходные поля endpoint с примерами значений. Шаг без входных данных MAY стоять первым в потоке.

#### Scenario: Автоподстановка источника
- **WHEN** пользователь добавляет шаг, у которого поле `id` совпадает с выходом предыдущего шага
- **THEN** система автоматически маппит поле на выход этого шага, оставляя возможность сменить источник вручную

### Requirement: Управление потоком шагов
Редактор SHALL позволять вставлять шаг в любую позицию, редактировать и удалять шаги, менять порядок drag-and-drop. Для мультисервисных сценариев система SHALL показывать индикатор числа задействованных API.

#### Scenario: Перестановка шагов
- **WHEN** пользователь перетаскивает шаг на новую позицию
- **THEN** порядок исполнения обновляется и сохраняется

### Requirement: Публикация сценария
Система SHALL позволять публиковать и скрывать сценарий. Публикация MUST требовать хотя бы один шаг. Опубликованный сценарий появляется в маркетплейсе, скрытый — виден только владельцу.

#### Scenario: Публикация без шагов
- **WHEN** пользователь пытается опубликовать сценарий без шагов
- **THEN** система блокирует публикацию с подсказкой «Добавьте хотя бы один шаг»

### Requirement: Фильтрация массивного выхода шага

Когда входное поле шага маппится на выход предыдущего шага («Из шага»), а этот шаг-источник возвращает **массив** (сам ответ — коллекция либо выбранное поле является списком объектов), система SHALL требовать условие фильтрации, отбирающее из массива один элемент. Условие состоит из трёх частей: **поле элемента** (из схемы элемента массива), **оператор** и **значение сравнения**.

Набор операторов: `=` (равно), `≠` (не равно), `>` (больше), `<` (меньше), `≥` (больше либо равно), `≤` (меньше либо равно), «содержит» (подстрока). Система MUST предлагать только операторы, осмысленные для типа выбранного поля элемента: для `boolean` — только `=` и `≠`; «содержит» — только для `string`.

Значение сравнения MUST задаваться теми же тремя способами, что и сам маппинг: **ручной ввод** (пользователь введёт значение при запуске сценария), **константа** (в том числе с подстановкой `{{s0:key}}`-шаблона) или **значение из шага** (выход одного из предыдущих шагов). Тип значения MUST соответствовать типу выбранного поля элемента.

Блок фильтра SHALL отображаться только тогда, когда шаг-источник действительно отдаёт массив; для скалярного выхода настройка маппинга не меняется. Сводка параметра в списке шагов SHALL отражать наличие фильтра вместе с источником.

#### Scenario: Настройка фильтра для массивного источника

- **WHEN** пользователь выбирает для входного параметра источник «Из шага», а выбранный шаг-источник возвращает массив организаций
- **THEN** под селектами «Шаг-источник» и «Поле результата» появляется блок условия — выбор поля элемента, оператора и значения — и параметр не считается настроенным, пока условие не задано

#### Scenario: Скалярный источник фильтра не требует

- **WHEN** шаг-источник возвращает объект (не массив)
- **THEN** блок фильтра не отображается, а маппинг настраивается как раньше — только источником и полем результата

#### Scenario: Значение сравнения берётся из другого шага

- **WHEN** пользователь задаёт условие «`inn` = значение из шага 1 · `inn`»
- **THEN** система сохраняет условие со ссылкой на выход шага 1, а не с зафиксированной константой

#### Scenario: Операторы ограничены типом поля

- **WHEN** пользователь выбирает для условия поле элемента типа `boolean`
- **THEN** система предлагает только операторы `=` и `≠`, без сравнений порядка и «содержит»

### Requirement: Удаление сценария
Редактор сценария SHALL позволять владельцу удалить сценарий. Удаление MUST быть необратимым и требовать подтверждения через модальное окно. Визуально кнопка удаления SHALL повторять паттерн удаления приложения: `Button` со стилем `danger` и иконкой корзины. После успешного удаления система MUST перенаправить пользователя на список его сценариев. Удаление использует существующий эндпоинт `DELETE /api/scenarios/:id` без изменений бэкенда.

#### Scenario: Удаление сценария владельцем
- **WHEN** пользователь нажимает кнопку «Удалить» в шапке редактора сценария
- **THEN** система показывает модальное окно подтверждения с предупреждением о необратимости удаления (шаги и публикация в маркетплейсе будут утрачены)

#### Scenario: Подтверждение удаления
- **WHEN** пользователь подтверждает удаление в модальном окне
- **THEN** система вызывает `DELETE /api/scenarios/{id}` и перенаправляет на `/my/scenarios`

#### Scenario: Отмена удаления
- **WHEN** пользователь закрывает модальное окно подтверждения или нажимает «Отмена»
- **THEN** сценарий не удаляется, модальное окно закрывается

### Requirement: Вкладка «Окружения» сценария
Редактор сценария SHALL иметь отдельную вкладку **«Окружения»**. Вкладка MUST перечислять всех поставщиков (apps), задействованных в шагах сценария, и для каждого позволять выбрать используемое окружение из окружений этого поставщика. По умолчанию для каждого поставщика MUST быть выбран Prod. Выбор MUST сохраняться в составе сценария как соответствие «поставщик → окружение».

Список поставщиков на вкладке MUST быть производным от шагов: поставщик появляется, когда в сценарий добавлен шаг по его endpoint, и исчезает, когда таких шагов не остаётся. Исполнение шага MUST использовать окружение, выбранное для поставщика этого шага (см. capability `api-app-environments`).

#### Scenario: Выбор окружения для поставщика
- **WHEN** в сценарии есть шаги по поставщику, и пользователь на вкладке «Окружения» выбирает для него «Staging»
- **THEN** выбор сохраняется, и последующее исполнение шагов этого поставщика идёт по `baseUrl` окружения Staging

#### Scenario: Список поставщиков выводится из шагов
- **WHEN** пользователь добавляет первый шаг по новому поставщику
- **THEN** на вкладке «Окружения» появляется этот поставщик с выбором окружения по умолчанию Prod

#### Scenario: По умолчанию Prod
- **WHEN** пользователь не менял окружение поставщика
- **THEN** для него используется окружение Prod
