## Requirements
### Requirement: Список запусков пользователя (API)
Система SHALL предоставлять endpoint `GET /api/runs`, возвращающий запуски **только текущего пользователя** (фильтр по `userId` из JWT) в формате `PaginatedResponse` (`{ data, total, page, limit, totalPages }`), отсортированные по `createdAt` по убыванию. Endpoint MUST поддерживать параметры `page`, `limit` и опциональный фильтр `status` (по одному или нескольким значениям `RunStatus`). Элемент списка MUST содержать данные для отображения без догрузки полного `Run`: `id`, `scenarioId`, название сценария, `status`, `createdAt`, `updatedAt`, номер текущего шага и общее число шагов, текст ошибки при наличии. Коллекция `runs` MUST иметь индекс, покрывающий выборку по `userId` с сортировкой по `createdAt`.

#### Scenario: Пагинированный список своих запусков
- **WHEN** авторизованный пользователь запрашивает `GET /api/runs?page=1&limit=10`
- **THEN** система возвращает первые 10 его запусков (новые сверху) и метаданные пагинации; чужие запуски в выборку не попадают

#### Scenario: Фильтр по статусу
- **WHEN** пользователь запрашивает `GET /api/runs?status=running,waiting_input`
- **THEN** в ответе только запуски в указанных статусах

### Requirement: Раздел «Запуски» в интерфейсе
Интерфейс SHALL содержать приватный пункт навигации «Запуски», ведущий на страницу списка запусков текущего пользователя, свёрстанную по эталону дизайн-системы (`assets/Fuse Design System/ui_kits/fuse-app/Runs.jsx`): вкладки «Активные · N» (`pending`, `running`, `waiting_input`) и «Завершённые · N» (`completed`, `failed`, `cancelled`), под ними список строк-аккордеонов с пагинацией по образцу существующих списков.

Свёрнутая строка MUST показывать плитку статуса, название сценария, id запуска и время; для выполняющегося запуска — чип «шаг X из N» с живым таймером прошедшего времени, для остальных — бейдж статуса. Развёрнутая строка MUST показывать переключаемые панели (SegmentedControl): прогресс шагов (для активных), результат/блок ошибки или отмены (для завершённых), входные файлы и файлы-результаты с кнопками скачивания. Панель действий MUST предлагать: для активного — отмену; для терминального — «Открыть результат» (переход на экран запуска `/cards/{scenarioId}/run?run={runId}`, который восстанавливает состояние по снапшоту `run:status`, включая продолжение запуска в `waiting_input`), «Повторить запуск» и «Удалить из истории». Пустые состояния вкладок MUST соответствовать эталону. Статусы активных запусков MUST актуализироваться без ручной перезагрузки страницы (realtime-событие или обновление при возврате фокуса).

#### Scenario: Возврат к активному запуску
- **WHEN** пользователь ушёл со страницы запуска, а позже открыл раздел «Запуски» и кликнул по запуску в статусе `waiting_input`
- **THEN** открывается экран запуска с восстановленной страницей ввода, и пользователь может продолжить выполнение

#### Scenario: Просмотр результата завершённого запуска
- **WHEN** пользователь открывает из списка запуск в статусе `completed`
- **THEN** экран запуска показывает итоговый результат, тайминги шагов и ссылки на скачивание файлов-результатов

#### Scenario: Пустая история
- **WHEN** у пользователя нет ни одного запуска
- **THEN** страница показывает пустое состояние с подсказкой запустить сценарий из маркетплейса или своих сценариев

### Requirement: Финальная страница как результат запуска
Если сценарий заканчивается шагом «Страница» без блоков ввода (display-only), эта страница — как правило, отформатированный итоговый экран (текст/markdown, значения блоков) — MUST сохраняться при исполнении вместе с разрешёнными данными блоков отображения (в `Run.finalPage`), поскольку живого события `page:required`, несущего эти данные, при просмотре из истории или повторном открытии уже нет. При наличии сохранённой финальной страницы результат завершённого запуска MUST показывать её (а не сырые данные последнего шага) — и в развёрнутой строке истории, и на экране запуска, открытом по `?run={runId}`. Последняя display-only страница запуска перезаписывает предыдущую и считается финальным экраном.

#### Scenario: Отформатированный итог в истории
- **WHEN** пользователь разворачивает в истории запуск, сценарий которого завершается display-only страницей с текстом/markdown и значениями блоков
- **THEN** панель «Результат» показывает эту страницу с разрешёнными данными, а не сырой JSON последнего шага

#### Scenario: Финальная страница при открытии из истории
- **WHEN** пользователь открывает завершённый запуск по «Открыть результат» (`/cards/{scenarioId}/run?run={runId}`), когда живого `page:required` уже не было
- **THEN** экран запуска восстанавливает финальную страницу из сохранённых данных и показывает отформатированный итог

### Requirement: Повторный запуск из истории
Для запуска в терминальном статусе система SHALL позволять повторный запуск: создаётся **новый** `Run` по тому же `scenarioId` с входными данными (`inputs`) исходного запуска — через существующий `POST /api/runs`. Исходный запуск MUST остаться в истории без изменений; данные страниц (`page-submit`) исходного запуска MUST NOT переиспользоваться — новый запуск снова остановится на page-шаге.

#### Scenario: Повтор упавшего запуска
- **WHEN** пользователь нажимает «Повторить запуск» у запуска в статусе `failed`
- **THEN** создаётся новый запуск с теми же входами, он появляется во вкладке «Активные», а исходный остаётся в «Завершённых» без изменений

### Requirement: Удаление запуска с каскадным удалением файлов
Система SHALL предоставлять endpoint `DELETE /api/runs/:id`. Удалять запуск MUST только его владелец (иначе 403); удаление запуска в нетерминальном статусе (`pending`, `running`, `waiting_input`) MUST отклоняться с ошибкой, предлагающей сначала отменить запуск. При удалении система MUST удалить из S3 все файлы, связанные с запуском (файлы-артефакты, полученные от внешних API, и загруженные пользователем в рамках запуска файлы, учтённые в реестре файлов запуска), удалить связанные с запуском уведомления и затем удалить документ `Run`. Ошибка удаления отдельного объекта S3 (например, объект уже отсутствует) MUST NOT блокировать удаление запуска. В интерфейсе удаление MUST требовать подтверждения.

#### Scenario: Удаление завершённого запуска с файлами
- **WHEN** пользователь удаляет свой запуск в статусе `completed`, у которого есть файлы-артефакты в S3
- **THEN** система удаляет объекты из S3, уведомления запуска и документ `Run`; запуск исчезает из списка, а его файлы недоступны для скачивания

#### Scenario: Попытка удалить активный запуск
- **WHEN** пользователь пытается удалить запуск в статусе `running`
- **THEN** система возвращает ошибку с указанием сначала отменить запуск; документ и файлы не затронуты

#### Scenario: Попытка удалить чужой запуск
- **WHEN** авторизованный пользователь вызывает `DELETE /api/runs/:id` для запуска другого пользователя
- **THEN** система возвращает 403, запуск не удаляется
