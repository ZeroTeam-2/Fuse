## Requirements
### Requirement: Уведомление о завершении запуска
Система SHALL создавать пользователю уведомление при переходе его запуска в терминальный статус (`completed`, `failed`, `cancelled`), а также при переходе в `waiting_input` (запуск ждёт действий пользователя). Уведомление MUST содержать `userId`, `runId`, `scenarioId`, название сценария, тип события, момент создания и флаг прочитанности. Повторная обработка того же перехода статуса (повторная доставка сообщения очереди) MUST NOT создавать дубликат уведомления. При удалении запуска связанные уведомления MUST удаляться.

#### Scenario: Успешное завершение
- **WHEN** запуск пользователя переходит в `completed`
- **THEN** создаётся непрочитанное уведомление «Сценарий … завершён» с ссылкой на запуск

#### Scenario: Запуск ждёт ввода
- **WHEN** запуск переходит в `waiting_input`, а пользователь ушёл со страницы запуска
- **THEN** пользователь получает уведомление о том, что сценарий ожидает его ввода

#### Scenario: Повторная доставка не дублирует
- **WHEN** воркер повторно обрабатывает уже терминальный запуск
- **THEN** новое уведомление не создаётся

### Requirement: Колокольчик и лента уведомлений
Шапка интерфейса SHALL содержать колокольчик со счётчиком непрочитанных уведомлений текущего пользователя (виден только авторизованным), выполненный по эталону дизайн-системы (`NotificationBell` в `assets/Fuse Design System/ui_kits/fuse-app/Runs.jsx`): бейдж с числом непрочитанных, выпадающая панель с последними уведомлениями (новые сверху: плитка статуса, «название сценария — событие», относительное время, точка непрочитанности) и футером «Все запуски →», ведущим в раздел «Запуски». Открытие панели MUST помечать показанные уведомления прочитанными (сброс счётчика); клик по уведомлению MUST вести на экран соответствующего запуска. Система SHALL предоставлять API: список уведомлений с пагинацией, счётчик непрочитанных, пометка всех прочитанными. Все операции — только над уведомлениями текущего пользователя.

#### Scenario: Непрочитанные в счётчике
- **WHEN** у пользователя 3 непрочитанных уведомления
- **THEN** на колокольчике бейдж «3»; после открытия панели уведомления помечаются прочитанными и бейдж исчезает

#### Scenario: Переход из уведомления
- **WHEN** пользователь кликает по уведомлению о завершённом запуске
- **THEN** открывается экран этого запуска с результатами

#### Scenario: Переход ко всем запускам
- **WHEN** пользователь нажимает «Все запуски →» в панели колокольчика
- **THEN** открывается раздел «Запуски»

### Requirement: Realtime-доставка уведомлений
Система SHALL доставлять уведомления в реальном времени через WebSocket: авторизованный клиент подключается к пространству уведомлений и попадает в комнату `user:{userId}`; при создании уведомления сервер MUST эмитить событие в эту комнату, а клиент MUST обновить счётчик и список без перезагрузки страницы. При подключении сервер MUST отдавать снапшот (актуальный счётчик непрочитанных), чтобы события, отправленные до подключения, не терялись. При недоступности WebSocket колокольчик MUST показывать актуальные данные при загрузке страницы через REST.

#### Scenario: Уведомление приходит на другой странице
- **WHEN** пользователь находится в маркетплейсе, а его фоновый запуск завершился
- **THEN** счётчик на колокольчике увеличивается в реальном времени без перезагрузки

#### Scenario: Подключение после события
- **WHEN** запуск завершился, пока у пользователя не было открытой вкладки
- **THEN** при следующем входе колокольчик показывает корректный счётчик непрочитанных из снапшота/REST
