## Requirements
### Requirement: Контейнеризованная локальная инфраструктура

Проект SHALL предоставлять compose-файл, который поднимает только инфраструктурные зависимости — MongoDB, SQS-совместимую очередь и S3-совместимое хранилище — без сервисов приложения (`backend`, `frontend`, `caddy`). Каждый сервис MUST публиковать порт на хосте, чтобы процессы, запущенные вне докера, могли к нему подключиться.

#### Scenario: Поднятие инфраструктуры одной командой

- **WHEN** разработчик выполняет `pnpm infra` в чистом рабочем каталоге
- **THEN** запускаются контейнеры MongoDB, LocalStack (SQS) и MinIO, и ни один сервис приложения не стартует
- **AND** команда завершается успешно только после того, как все сервисы прошли healthcheck

#### Scenario: Инфраструктура доступна процессам на хосте

- **WHEN** инфраструктура запущена
- **THEN** MongoDB доступна на `localhost` по опубликованному порту, SQS-эндпоинт — на `http://localhost:4566`, S3 — на `http://localhost:9000`, а консоль MinIO — на `http://localhost:9001`
- **AND** бэкенд, запущенный на хосте через `pnpm dev`, подключается к ним без дополнительной настройки сети

### Requirement: Готовность очереди и бакета при старте

Инфраструктура SHALL самостоятельно создавать ресурсы, которые приложение ожидает найти готовыми: очередь исполнения сценариев с DLQ и бакет S3, имя которого задано в `S3_BUCKET`. Бутстрап MUST быть идемпотентным — повторный запуск на существующих томах не MUST завершаться ошибкой и не MUST терять данные.

#### Scenario: Очередь создаётся при первом старте

- **WHEN** LocalStack стартует впервые
- **THEN** создаются очереди `scenario-execution` (visibility timeout 7200 секунд) и `scenario-execution-dlq`, связанные redrive-политикой с `maxReceiveCount: 3`
- **AND** бэкенд, запущенный на хосте, успешно отправляет и получает сообщения по URL очереди из окружения

#### Scenario: Бакет создаётся при первом старте

- **WHEN** MinIO стартует впервые
- **THEN** бакет с именем из `S3_BUCKET` существует до того, как инфраструктура будет объявлена готовой
- **AND** загрузка файла из локально запущенного бэкенда проходит без ошибки «bucket does not exist»

#### Scenario: Повторный запуск на существующих данных

- **WHEN** разработчик выполняет `pnpm infra` повторно, при уже существующих томах и созданных ресурсах
- **THEN** бутстрап завершается успешно, очередь и бакет сохраняются, а ранее записанные данные остаются на месте

### Requirement: Изоляция от боевых сервисов

Локальный профиль окружения SHALL направлять бэкенд исключительно на контейнеризованную инфраструктуру. Профиль по умолчанию MUST NOT содержать адресов или учётных данных боевых сервисов; секретные значения MUST NOT попадать в файлы, отслеживаемые git.

#### Scenario: Активирован локальный профиль

- **WHEN** разработчик применяет локальный профиль окружения и запускает бэкенд на хосте
- **THEN** бэкенд подключается к MongoDB, очереди и S3 в контейнерах на `localhost`
- **AND** ни один запрос не уходит к прод-MongoDB, общей Yandex Message Queue или внешнему S3

#### Scenario: Запуск сценария не перехватывается задеплоенным бэкендом

- **WHEN** пользователь запускает сценарий из локального UI при активном локальном профиле
- **THEN** сообщение попадает в локальную очередь и обрабатывается локальным воркером
- **AND** статус запуска доходит до терминального состояния, а не остаётся в «Выполняем сценарий…» бесконечно

#### Scenario: Секреты не коммитятся

- **WHEN** локальный профиль записан в файл окружения
- **THEN** этот файл игнорируется git, а в репозитории присутствует только его пример без реальных секретов

### Requirement: Переключение профилей окружения

Разработчик SHALL иметь возможность переключаться между локальной инфраструктурой и удалёнными сервисами, не редактируя вручную существующий файл `.env` с боевыми учётными данными. Локальный профиль MUST иметь приоритет над базовым при разрешении переменных.

#### Scenario: Включение локального профиля

- **WHEN** разработчик создаёт файл локального профиля из примера и перезапускает бэкенд
- **THEN** значения из локального профиля перекрывают одноимённые переменные базового `.env`
- **AND** переменные, не заданные в локальном профиле, продолжают браться из базового `.env`

#### Scenario: Возврат к удалённым сервисам

- **WHEN** разработчик удаляет (или переименовывает) файл локального профиля и перезапускает бэкенд
- **THEN** бэкенд снова использует значения из базового `.env`
- **AND** содержимое базового `.env` не было изменено при переключении в любую из сторон

### Requirement: Управление жизненным циклом инфраструктуры

Проект SHALL предоставлять команды остановки, просмотра логов и полного сброса состояния инфраструктуры. Сброс MUST удалять тома данных, чтобы среду можно было пересоздать детерминированно.

#### Scenario: Остановка инфраструктуры

- **WHEN** разработчик выполняет команду остановки
- **THEN** инфраструктурные контейнеры останавливаются, а тома с данными сохраняются
- **AND** последующий `pnpm infra` поднимает среду с прежними данными

#### Scenario: Полный сброс состояния

- **WHEN** разработчик выполняет команду сброса
- **THEN** контейнеры и тома данных MongoDB и MinIO удаляются
- **AND** следующий запуск инфраструктуры создаёт пустую БД, пустой бакет и очередь заново, без ручных шагов
