# api-app-management

## Purpose

Управление подключёнными API-приложениями: импорт OpenAPI-спецификаций, просмотр и обновление endpoints, управление публикацией.
## Requirements
### Requirement: Создание приложения с импортом OpenAPI
Система SHALL позволять создать «приложение» (подключённый API-провайдер): название, краткое описание и ссылку на OpenAPI-спецификацию в формате **JSON или YAML** либо загружаемый файл спецификации. При импорте система MUST автоматически разобрать спецификацию и извлечь список endpoints (метод, путь, summary), хост API и версию.

Система MUST принимать YAML-тело ответа наравне с JSON: формат определяется по `Content-Type` ответа (`application/json` → JSON; `application/yaml`, `application/x-yaml`, `text/yaml` → YAML), а при неинформативном `Content-Type` — по первому непробельному символу тела (`{` или `[` → JSON, иначе → YAML). Если разбор по определённому формату не удаётся, система MUST попробовать альтернативный формат в качестве запасного варианта.

При извлечении endpoints система MUST сохранять **тег** операции (первый элемент массива `tags`, если задан) и делать его доступным фронтенду. Тег — приоритетный источник имени смыслового блока; при его отсутствии блок выводится из пути endpoint'а (первый сегмент при наличии вложенности, иначе «Прочее») — правило группировки на стороне UI.

При импорте система MUST вывести и сохранить **базовый URL приложения** (`baseUrl`) — абсолютный адрес со схемой, хостом и, при наличии, базовым путём. Правило вывода: значение `servers[0].url` из спецификации, разрешённое относительно URL самой спецификации (это корректно обрабатывает как абсолютные, так и относительные значения `servers`); если поле `servers` отсутствует — origin URL спецификации. Отображаемый `host` MUST быть производным от `baseUrl`. Базовый URL MUST быть достаточным для исполнения шагов сценария: путь endpoint'а, присоединённый к `baseUrl`, MUST давать вызываемый абсолютный URL. Выведенный `baseUrl` MUST становиться Base URL окружения Prod приложения (см. capability `api-app-environments`).

При разборе схемы ответа система MUST сохранять **признак массивного ответа** endpoint'а: если тело успешного ответа — коллекция, схема выходов MUST помечаться как массив, а извлечённые поля MUST описывать **элемент** коллекции. Признак MUST NOT теряться при разворачивании `items` — иначе схема шага обещает объект там, где API возвращает список. Для полей типа `array` внутри объекта система MUST сохранять схему элемента (поля элемента), если она задана в спецификации. Эти данные MUST быть доступны конструктору сценариев через схему шага — на них строится условие фильтрации массива.

Система MUST NOT сохранять полный развёрнутый снапшот спецификации инлайном в документе приложения. Сохраняются только производные данные, необходимые для работы (endpoints, схемы входов/выходов, теги, `baseUrl`, `host`, версия) — чтобы документ приложения гарантированно помещался в потолок документа хранилища независимо от размера исходной спеки.

При импорте из загружаемого файла максимальный размер файла MUST браться из конфигурации и MUST быть ограничен так, чтобы результат импорта помещался в хранилище; загрузка файла сверх лимита MUST отклоняться понятной ошибкой о превышении размера.

То же правило MUST применяться при повторном импорте спецификации.

#### Scenario: Успешный импорт спецификации
- **WHEN** пользователь указывает валидный URL OpenAPI JSON и нажимает «Импортировать»
- **THEN** система показывает «Спецификация разобрана — найдено N endpoints» и список endpoints с методами и summary

#### Scenario: Успешный импорт YAML-спецификации
- **WHEN** пользователь указывает URL, отдающий валидный OpenAPI YAML с `Content-Type: application/yaml`
- **THEN** система разбирает YAML, извлекает endpoints и показывает «Спецификация разобрана — найдено N endpoints» с тем же результатом, что и для эквивалентного JSON

#### Scenario: YAML-спецификация без явного Content-Type
- **WHEN** сервер отдаёт YAML-документ с `Content-Type: text/plain` (тело не начинается с `{` или `[`)
- **THEN** система определяет формат как YAML по содержимому тела и успешно разбирает спецификацию

#### Scenario: Endpoints группируются в превью
- **WHEN** спецификация содержит операции с тегами (например, `Users`, `Orders`) и/или пути с вложенностью (`/repos/{owner}/{repo}`, `/users/{name}`)
- **THEN** превью импорта показывает endpoints сворачиваемыми блоками: по тегу, если он есть, иначе по первому сегменту пути (`repos`, `users`), а одноуровневые пути без тега — в блоке «Прочее»

#### Scenario: Спецификация без поля servers
- **WHEN** импортируется спецификация, в которой отсутствует `servers` (типично для FastAPI, где документ отдаётся по `https://host/openapi.json`)
- **THEN** система сохраняет `baseUrl` равным origin URL спецификации (`https://host`), и шаги сценария по этому приложению исполняются успешно

#### Scenario: Базовый путь из servers сохраняется
- **WHEN** спецификация содержит `servers[0].url` вида `https://api.example.com/v1`
- **THEN** система сохраняет `baseUrl` со всем базовым путём (`https://api.example.com/v1`), и вызовы шагов идут по `https://api.example.com/v1/<path>`

#### Scenario: Невалидная спецификация
- **WHEN** URL недоступен или документ не является валидным OpenAPI в формате JSON или YAML
- **THEN** система показывает ошибку импорта и не даёт создать приложение

#### Scenario: Создание после импорта
- **WHEN** спецификация разобрана и пользователь нажимает «Создать приложение»
- **THEN** система сохраняет приложение вместе с endpoints, их тегами и схемами входов/выходов (из параметров и примеров ответов), не сохраняя при этом полный развёрнутый снапшот спецификации, и создаёт окружение Prod с Base URL, равным `baseUrl`

#### Scenario: Импорт из файла в пределах лимита
- **WHEN** пользователь загружает файл спецификации в пределах настроенного лимита размера
- **THEN** система разбирает спеку и сохраняет приложение без ошибки сериализации хранилища

#### Scenario: Импорт из файла сверх лимита
- **WHEN** пользователь загружает файл спецификации больше настроенного лимита
- **THEN** система отклоняет загрузку с понятной ошибкой о превышении размера, а не падает с внутренней ошибкой

#### Scenario: Endpoint возвращает коллекцию
- **WHEN** успешный ответ endpoint'а описан как массив объектов (`type: array`, `items` с полями `id`, `name`)
- **THEN** система сохраняет выходы как поля элемента (`id`, `name`) вместе с признаком, что ответ — массив, и конструктор сценариев предлагает для такого источника условие фильтрации

#### Scenario: Поле-список внутри объекта
- **WHEN** ответ endpoint'а — объект с полем `items`, описанным как массив объектов
- **THEN** система сохраняет поле `items` с типом `array` и схемой его элемента, а не как тип без содержимого

### Requirement: Выходы endpoint'а из примера ответа при пустой схеме
Если разбор схемы успешного ответа не дал ни одного поля (схема отсутствует, пуста или без `properties`), система SHALL выводить схему выходов endpoint'а из **примера ответа**: `content.example`, затем `example` внутри схемы, затем значение первого элемента карты `content.examples`. Пример-объект даёт поля по ключам с типом по типу значения и значением в качестве образца; пример-массив объектов помечает ответ коллекцией, а поля описывают элемент; поле-массив объектов внутри примера-объекта получает схему элемента (один уровень вложенности — как при разборе схемы). Схема ответа с хотя бы одним извлечённым полем MUST выигрывать у примера — фолбэк в этом случае не применяется.

#### Scenario: FastAPI-ручка без response_model
- **WHEN** импортируется спецификация, где успешный ответ endpoint'а описан `schema: {}` с `example: {status: "...", result: "..."}` рядом (типичный вывод FastAPI без `response_model`)
- **THEN** система сохраняет выходы `status` и `result` типа строка, и в конструкторе сценариев к ним можно привязать маппинг «Из шага» и блок отображения страницы

#### Scenario: Пример-массив помечает коллекцию
- **WHEN** схема успешного ответа пуста, а пример — массив объектов с полями `id`, `name`
- **THEN** система сохраняет выходы `id`, `name` как поля элемента вместе с признаком массивного ответа

#### Scenario: Настоящая схема выигрывает у примера
- **WHEN** успешный ответ описан схемой с `properties` и рядом лежит пример с другим набором ключей
- **THEN** выходы строятся только из схемы, поля примера не подмешиваются

### Requirement: Список и карточка приложения
Раздел «Мои API» SHALL показывать список приложений пользователя с пагинацией (имя, описание, хост, версия, число endpoints, число использующих сценариев). Карточка приложения SHALL показывать импортированные endpoints, дату последней синхронизации и число сценариев, использующих этот API.

Endpoints на карточке приложения MUST быть организованы по **смысловым категориям** в виде «сайдбар + содержимое»: слева — список категорий (имя категории и счётчик endpoints), справа — **прокручиваемый список** endpoints выбранной категории, под ним — **пагинация** (общий компонент, см. `pagination-control`) с не более чем **25 endpoints на странице**. Имя категории endpoint'а определяется так: тег операции, если задан; иначе — **первый сегмент пути**, когда у пути есть вложенность (2-й и далее уровни); одноуровневые (без 2-го уровня) и параметр-первые пути без тега MUST попадать в категорию «Прочее». MUST быть **сквозной поиск** по методу, пути, summary и имени категории: он сужает и список категорий (со счётчиками), и содержимое; ввод поиска сбрасывает на первую страницу. Клик по endpoint MUST **раскрывать под строкой** его схему: какие параметры он принимает (с указанием расположения — path/query/header/body, типа и обязательности) и что отдаёт (поля ответа, признак массива, поля элемента для полей-списков). Вёрстка строки endpoint MUST устойчиво вести себя при длинном summary — описание усекается/переносится и MUST NOT выходить за пределы контейнера.

#### Scenario: Просмотр карточки приложения
- **WHEN** пользователь открывает приложение из списка
- **THEN** система показывает его метаданные, сайдбар категорий endpoints и счётчик «сценариев используют этот API»; выбор категории показывает её endpoints списком с прокруткой и пагинацией по 25 на странице

#### Scenario: Сквозной поиск по endpoints провайдера
- **WHEN** пользователь вводит запрос в поиск endpoints на карточке приложения
- **THEN** список категорий и содержимое сужаются до endpoints, совпавших по методу, пути, summary или имени категории, а пагинация сбрасывается на первую страницу

#### Scenario: Раскрытие схемы endpoint
- **WHEN** пользователь кликает по endpoint в списке карточки приложения
- **THEN** под строкой раскрывается панель с параметрами (расположение, тип, обязательность) и полями ответа; повторный клик сворачивает её

#### Scenario: Длинное описание endpoint не ломает вёрстку
- **WHEN** у endpoint'а очень длинный summary
- **THEN** строка endpoint остаётся в пределах контейнера (описание усекается или переносится), а метод и путь остаются читаемыми

### Requirement: Обновление спецификации
Система SHALL позволять повторно импортировать OpenAPI-спецификацию приложения. Система MUST показать diff — новые endpoints помечаются меткой «НОВЫЙ» — до сохранения. Сохранение обновления MUST NOT ломать существующие сценарии: уже используемые endpoints сохраняются.

#### Scenario: Обновление с новыми endpoints
- **WHEN** пользователь указывает URL новой версии спецификации и запускает проверку
- **THEN** система показывает «Найдены изменения — новых endpoints: K из N» со списком, и после «Сохранить обновление» новые endpoints доступны в конструкторе сценариев

### Requirement: Повторный импорт спецификации из файла
Страница обновления приложения SHALL предлагать выбор источника спецификации — **URL либо файл** — независимо от того, как приложение было создано. Файловая ветка SHALL принимать файл `.json`/`.yaml`/`.yml` с тем же лимитом размера, что и при создании, и необязательное переопределение базового URL. Система MUST построить тот же диф (новые / удалённые / без изменений), что и при реимпорте по URL, и по подтверждению применить то же слияние endpoints (сохранение id совпавших, пометка исчезнувших как deprecated). Если из файла базовый URL не выводится (нет абсолютного `servers`, переопределение не задано), приложение MUST сохранить свой текущий `baseUrl`; сохранённый `openapiUrl` приложения при файловом реимпорте MUST NOT изменяться и MUST NOT требоваться.

#### Scenario: Обновление приложения, созданного из файла
- **WHEN** пользователь открывает «Проверить обновления» приложения, импортированного из файла, выбирает источник «Файл», загружает новую версию спеки и подтверждает сохранение
- **THEN** система показывает диф endpoints, а после подтверждения сливает их по правилам реимпорта — без заглушки и без требования URL

#### Scenario: Файл без servers не затирает базовый URL
- **WHEN** загруженный при реимпорте файл не содержит абсолютного `servers`, и поле переопределения пусто
- **THEN** после применения `baseUrl` приложения остаётся прежним, и шаги сценариев продолжают исполняться по нему

#### Scenario: Реимпорт файлом у приложения с URL
- **WHEN** у приложения задан `openapiUrl`, но пользователь выбирает источник «Файл» и применяет обновление
- **THEN** endpoints обновляются из файла, а сохранённый `openapiUrl` не меняется — следующая проверка по URL продолжает работать

### Requirement: Публикация приложения
Система SHALL позволять включать и выключать публикацию приложения. Endpoints неопубликованного приложения MUST NOT быть доступны другим пользователям при сборке сценариев.

#### Scenario: Скрытие приложения
- **WHEN** владелец выключает публикацию приложения
- **THEN** приложение помечается как скрытое и исчезает из выбора API у других пользователей; собственные сценарии владельца продолжают работать
